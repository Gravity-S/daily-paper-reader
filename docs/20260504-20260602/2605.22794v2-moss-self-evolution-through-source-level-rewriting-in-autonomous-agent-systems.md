## 摘要
部署后的自主智能体系统大多静态不变：它们无法从用户交互中学习，反复出现的故障会持续到下一次人工驱动的更新带来修复。为此，自我进化智能体应运而生，但所有进化都局限于文本可变的工件——技能文件、提示配置、记忆模式、工作流图——而智能体核心保持不变。由于路由、钩子排序、状态不变量和调度存在于代码而非任何文本工件中，整个结构故障类别从文本层实际上不可达。我们论证源码级适应是一种更通用的媒介：它是图灵完备的，是每个文本可变范围的严格超集，能确定性地生效而非通过基础模型遵从，并且不会在长上下文漂移下受损。我们提出MOSS，一个在生产级智能体基座上执行源码级自我重写的系统。每次进化基于自动整理的故障证据批次，通过确定性多阶段流水线进行；代码修改委托给可插拔的外部编码智能体CLI，而MOSS保留阶段排序和裁决。候选者通过在临时试验工作器中回放批次验证，然后通过用户同意门控、原地容器替换和健康检查门控回滚进行升级。在OpenClaw上，MOSS在一个周期内无需人工干预将四个任务的平均评分从0.25提升到0.61。

## Abstract
Autonomous agentic systems are largely static after deployment: they do not learn from user interactions, and recurring failures persist until the next human-driven update ships a fix. Self-evolving agents have emerged in response, but all confine evolution to text-mutable artifacts -- skill files, prompt configurations, memory schemas, workflow graphs -- and leave the agent harness untouched. Since routing, hook ordering, state invariants, and dispatch live in code rather than in any text artifact, an entire class of structural failure is physically unreachable from the text layer. We argue that source-level adaptation is a fundamentally more general medium: it is Turing-complete, a strict superset of every text-mutable scope, takes effect deterministically rather than through base-model compliance, and does not erode under long-context drift. We present MOSS, a system that performs self-rewriting at the source level on production agentic substrates. Each evolution is anchored to an automatically curated batch of production-failure evidence and proceeds through a deterministic multi-stage pipeline; code modification is delegated to a pluggable external coding-agent CLI while MOSS retains stage ordering and verdicts. Candidates are verified by replaying the batch against the candidate image in ephemeral trial workers, then promoted via user-consent-gated, in-place container swap with health-probe-gated rollback. On OpenClaw, MOSS lifts a four-task mean grader score from 0.25 to 0.61 in a single cycle without human intervention.

- **现有局限**：部署后的自主智能体系统（如 OpenClaw）是静态的，无法从用户交互中学习；重复出现的故障只能等待人工更新修复。已有的自进化智能体（如 Hermes Agent、SkillClaw、EvoAgentX 等）将进化范围限制在 **文本可变工件**（技能文件、提示配置、记忆模式、工作流图），而 **智能体核心**（路由、钩子排序、状态管理、调度）作为代码层始终未被触及。
- **根本矛盾**：文本层修改只能改变智能体的“思考”内容（如说什么、调用哪个技能），无法改变核心的决定逻辑。当故障源自代码层（例如消息路由错误、钩子触发顺序错乱、会话状态损坏、并发技能原子性缺陷），任何文本工件的更新都无法修复——因为问题不在提示文本中。
- **论文主张**：**源码级适应** 是更通用的媒介：它是 **图灵完备** 的，是文本可变范围的 **严格超集**（任何提示编辑能做到的事，代码编辑也能做到，反之不成立）；其效果是 **确定性** 的（代码逻辑不依赖基础模型是否理解新文本），且不会因 **长上下文漂移** 而退化（代码行为编码在机器指令中，而非每次重新读取的文本）。
- **研究意义**：将源码级自我重写能力从学术探索（如 SICA、Darwin Gödel Machine、HyperAgents）迁移到 **生产级、应用级** 智能体系统，面临代码量大、无干净基准分数、需要保留用户状态和在线流量等工程挑战。MOSS 填补了这一空白。

## 2. 论文提出的方法论

### 核心思想
MOSS 是一个 **源码级自我重写系统**，运行在生产级智能体基座上（如 OpenClaw）。它通过以下机制实现从用户故障到部署修复的闭环：
- 自动收集 **生产故障证据批次**。
- 执行 **确定性多阶段流水线**，由 **可插拔外部编码智能体 CLI** 完成代码修改，MOSS 自身保留阶段排序和裁决。
- 在 **临时试验工作器** 中回放批次验证候选者。
- 通过 **用户同意门控 + 原地容器替换 + 健康检查门控回滚** 升级候选者。

### 关键技术细节

#### 系统架构（5个组件）
1. **主容器**：运行用户交互智能体。
2. **控制面 CLI**：通过 `moss evo` 子命令暴露进化生命周期（status, batches, start, stop, apply, flag 等）。注入到智能体的系统提示中，用户可通过对话接口驱动进化。
3. **外部编码智能体 CLI**：可插拔，支持 Claude Code、OpenAI Codex、DeepSeek-TUI、OpenCode。MOSS 通过四方法运行接口抽象，负责执行代码修改。
4. **宿主守护进程**（host-daemon）：管理容器生命周期、编码智能体调用、试验工作器、自动扫描引擎、交换监督任务。
5. **临时试验工作器**：每次迭代启动 N 个短暂容器，在候选镜像中自动处理批次任务，结束后销毁。

#### 进化过程

1. **定向进化触发**：
   - 自动扫描：定期扫描会话 JSONL，切分块后经 Task-Evaluate 阶段评分，仅将评分为 weak/missing 的块追加到对应对话的开放批次。
   - 用户标记：用户不满时，通过对话中的 `moss evo flag` 命令触发相同扫描。
   - 批次达到阈值（默认 8 块）后密封，并被选中作为进化目标。

2. **进化循环**：
   - 先执行 **基准评估**（pre-loop baseline）：对已捕获的原始转录评分，锁定关键点集合。
   - 循环迭代，每次迭代执行七阶段流水线。终止条件：批次被修复、模型达到上限、架构限制，或连续无改进达到阈值。可通过进化深度（light/standard/deep）控制预算。

3. **七阶段流水线**（每阶段调用一次外部编码智能体）：
   - ① **Locate**：读取基准踪迹和故障，诊断但不提修复方案。
   - ② **Plan**：识别根因并指定修复（哪些文件修改、添加什么逻辑）。
   - ③ **Plan-Review**：质量门控，批准或拒绝计划（可多轮）。
   - ④ **Implement**：编写代码并提交为单个 git commit。
   - ⑤ **Code-Review**：质量门控，审查 diff 与计划一致性（可多轮，失败则重置到循环起始提交）。
   - ⑥ **Task-Evaluate**：在 Build 和 Trial 后，对每个任务评分（strong/adequate/weak/missing）。
   - ⑦ **Verdict**：综合评估给出四种裁决之一：CONVERGED、NEED_MORE_WORK、模型极限、架构极限。

4. **运行时验证**：
   - Build 后，宿主守护进程启动 N 个短暂容器（候选镜像），让智能体自主处理批次任务，每个任务执行多次以暴露不稳定情况。验证环境与生产环境一致（网络隔离、无用户状态挂载）。

5. **原地容器替换**：
   - 收敛后不自动替换：触发 `evolution-converged` 网络钩子通知智能体，智能体通知用户；用户需调用 `moss evo apply` 授权。
   - 宿主守护进程检测到交换请求文件后，用候选镜像重启主容器。
   - 90 秒健康探测窗口（每 5 秒采样一次，四个检查），连续三次通过则提交交换，否则回滚到最后已知良好镜像。
   - 用户状态卷在交换中保持挂载，确保状态幸存。

## 6. 论文的主要结论与发现

- **核心结论**：源码级适应是比文本层更通用的进化介质，能够修复文本层无法触及的结构故障。
- **实证结果**：在 OpenClaw 上，单次进化循环（无需人工干预）将四个任务的平均 claweval 评分从 0.2526 提高到 0.6100，增幅 0.3574。最大提升来自 T138（0.2090 → 0.9049，跨越 pass 阈值 0.75）。SLA 任务提升至约 0.53–0.55，一次迭代未完全解决细节计算。
- **修复本质**：修改了智能体核心本身（工具结果中介器的新注释分支、前置调用拒绝门），而非任何文本工件，证明了代码级修改的必要性和有效性。
- **过程可审计**：所有计划、diff、构建日志、评分矩阵均可由用户按需读取，增强了透明度和信任。

## 7. 优点

1. **首次将源码级自我进化应用于生产级智能体**：突破了先前仅限文本工件的限制，触及路由、状态管理、并发同步等关键层次。
2. **定向进化驱动而非随机探索**：使用真实用户故障批次作为锚点，避免盲目突变，更符合实际部署需求。
3. **严格的运行时验证**：通过瞬时试验工作器在接近生产环境的条件下回放批次，比单元测试或合成基准更可靠。
4. **容错和回滚机制**：健康检查门控 + 原地容器替换 + 自动回落到已知良好镜像，确保生产稳定性。
5. **用户参与门控**：交换需要用户显式授权，避免自动部署可能造成的意外。
6. **可插拔外部编码智能体**：解耦代码编辑能力与进化逻辑，便于切换不同模型和提供商。
7. **架构透明**：七阶段流水线分离关注点，每个阶段人工可审阅。
8. **在四任务案例中展示了显著提升**，且无需人工干预即实现闭环。

## 8. 不足与局限

1. **实验规模小**：仅 4 个任务、单次循环，缺乏多批次、多基座、多次迭代的系统评估。不能有力证明方法的通用性。
2. **缺少与基线方法的对比**：未与 Hermes Agent、SkillClaw 等文本层自进化方法在相同任务上直接比较，不能直接证明代码修改带来的增益优于文本修改。
