## 摘要
大型语言模型（LLM）代理的兴起，结合工具使用、技能和外部知识，引入了新的安全风险。其中，提示注入攻击——对手将恶意指令嵌入到代理工作流程中——已成为主要威胁。然而，现有的基准测试和防御措施从根本上受到限制，因为它们假设了上下文不敏感的设置，即代理在完全指定的用户指令下工作，且攻击是直接且与上下文无关的。因此，它们未能捕捉到现实世界部署中代理行为通常依赖于动态上下文（而不仅仅是用户提示）的情况，并且对手可以根据不同上下文调整其攻击。同样，基于这种狭隘威胁模型构建的现有防御措施忽视了现实世界代理委托的本质。在本文中，我们提出了AgentLure，一个捕获上下文相关任务和上下文感知提示注入攻击的基准测试。AgentLure涵盖四个代理领域和跨不同攻击面的八个攻击向量。我们的评估表明，现有防御措施在这种设置下往往难以应对，在代理系统中对此类攻击的表现不佳。为解决这一限制，我们提出了ARGUS，一种为LLM代理强制执行来源感知决策审计的防御机制。ARGUS构建了一个影响来源图，以追踪不可信上下文如何传播到代理决策中，并在执行前验证决策是否由可信证据证明合理。我们的评估显示，ARGUS将攻击成功率降低至3.8%，同时保留了87.5%的任务效用，显著优于现有防御措施，并且对自适应白盒对手保持稳健。

## Abstract
The rise of Large Language Model (LLM) agents, augmented with tool use, skills, and external knowledge, has introduced new security risks. Among them, prompt injection attacks, where adversaries embed malicious instructions into the agent workflow, have emerged as the primary threat. However, existing benchmarks and defenses are fundamentally limited as they assume context-insensitive settings in which the agent works under a fully specified user instruction, and the attacks are straightforward and context-independent. As a result, they fail to capture real-world deployments where agent behavior usually depends on dynamic context, not just the user prompt, and adversaries can adapt their attacks to different context. Similarly, existing defenses built on this narrow threat model overlook the nature of real-world agent delegation.   In this paper, we present AgentLure, a benchmark that captures context-dependent tasks and context-aware prompt injection attacks. AgentLure spans four agentic domains and eight attack vectors across diverse attack surfaces. Our evaluation shows that existing defenses often struggle in this setting, yielding poor performance against such attacks in agentic systems. To address this limitation, we propose ARGUS, a defense mechanism that enforces provenance-aware decision auditing for LLM agents. ARGUS constructs an influence provenance graph to track how untrusted context propagates into agent decisions and verify whether a decision is justified by trustworthy evidence before execution. Our evaluation shows ARGUS reduces attack success rate to 3.8% while preserving 87.5% task utility, significantly outperforming existing defenses and remaining robust against adaptive white-box adversaries.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题背景**：LLM Agent（工具增强型大语言模型代理）在自动执行多步骤任务时，面临提示注入攻击（Prompt Injection）的严重威胁，被OWASP列为LLM应用十大风险之首。现有防御和基准测试均基于**上下文不敏感**假设——即用户提示已完全确定正确动作，攻击负载独立于上下文且简单粗暴。
- **核心缺陷**：真实部署中，Agent常执行**上下文依赖任务**（如“支付最近的电费账单”，具体收款人、金额需从运行时读取的账单中获取）。同时，攻击者采用**上下文感知攻击**，将恶意指令嵌入载体数据中，伪造成普通数据，难以检测。
- **整体含义**：现有防御在上下文依赖场景下全面失效：要么错误放行攻击，要么过度拦截合法操作。论文旨在填补这一空白，提出贴近真实世界的基准（AgentLure）和防御机制（ARGUS）。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想
- 建立**影响溯源图**（Influence-Provenance Graph, IPG），将运行时上下文分解为句子级（span）粒度，追踪每个工具调用参数源自哪个span，并区分良性（benign）与异常（anomalous）span。
- 通过**局部溯源审计**和**全局任务级审计**双重验证，确保代理的每个状态更改动作（如转账、发邮件）仅当参数源于可信span、不违反用户任务约束、且由良性证据支撑时才允许执行。

### 2.2 关键技术细节

#### （1）影响溯源图（IPG）
- 有向无环图，节点表示内容单元（系统提示、用户查询、工具返回、检索文档、内存、技能等）。
- 每个节点具有：源类型、基础信任分数（τ₀）、内容span集合（标记为BENIGN或ANOMALOUS）、动态信任分数（τ = τ₀ × max(η, 良性字符比例)）。
- 基础信任分数按源类型预设：system/user=1.0, skill=0.6, tool_return=0.5, memory/RAG=0.4, agent_msg=0.3。

#### （2）四个LLM驱动的安全工具
- **ContentSegmenter（内容分割器）**：对未分割的IPG节点，将文本划分为span并标记为BENIGN或ANOMALOUS。标记依据：异常span指超出正常证据角色的指令，如添加副作用、重定向、伪造异常等；模糊内容保守标记为BENIGN。
- **ArgumentGrounder（参数溯源器）**：将每个工具调用参数映射到其来源span，并记录溯源类型（copy, normalize, derive, resolve, ungrounded）。确定参数值是否来自异常span。
- **InvariantChecker（不变性检查器）**：在会话初始化时从用户查询提取2~3个任务级不变约束（如“转账接收方必须是账单中指定的收款人”），后续每次审计时检查动作是否违反这些约束。
- **EntailmentVerifier（蕴含验证器）**：检查动作是否仅由良性证据支持，且是否可能被异常内容影响。仅当“不被良性证据蕴含”且“可被异常内容影响”时阻止。

#### （3）审计算法（Algorithm 1）
- 读操作（只读工具）直接放行，返回仍加入IPG。
- 状态更改操作触发审计：
  1. 对未分割节点运行ContentSegmenter。
  2. 运行ArgumentGrounder，获取四个风险信号：tainted参数、未知来源、异常上下文、高风险来源。
  3. 根据风险信号选择性调用全局检查：
     - 若参数tainted，先检查InvariantChecker，若违反则阻止；再检查EntailmentVerifier，若被异常影响且不被良性支持则阻止。
     - 若存在异常上下文或高风险来源，运行EntailmentVerifier。
     - 若参数未知或高风险，运行InvariantChecker。
  4. 阻止时返回**良性数据提示（Benign Data Hint）**，从良性span中提取候选值，供代理重试，重试后的调用需重新审计。

#### （4）恢复机制
- 阻止时返回HINTS，提取与阻塞参数相关的良性span值（如合法收款人IBAN），代理可据此修正调用并重新提交审计。

## 3. 实验设计：数据集/场景、基准、对比方法

### 3.1 基准：AgentLure
- 自建基准，包含**320个样本**，覆盖4个领域（Banking, Travel, Workspace, Slack），8个攻击向量（CR, AT, CF, RH, PC, IA, SI, WF），6个攻击面（工具文档、工具返回、检索文档、内存记录、安装技能、代理间消息）。
- 每个样本均为上下文依赖任务（用户提示不唯一确定动作）+ 上下文感知攻击（负载嵌入载体，伪造成正常内容）。
- 构建流程：人工指定抽象任务/攻击轨迹 → LLM具体化 → 双人验证（inter-annotator agreement κ=0.87）。

### 3.2 对比方法（8种）
- **文本级防御**：Delimiters（分隔符包裹）、Sandwich Defense（每次观察后重述用户指令）、Instructional Prevention（硬化指令）。
- **执行级防御**：Tool Filter（基于提示预测允许的工具集）、InjecGuard（分类器检测）、MELON（双轨迹对比）、ACE（组件隔离+参考监控）、DRIFT（动态规则合成+隔离）。
- 基线：无防御。

### 3.3 评估指标
- 安全：攻击成功率（ASR）、最差向量ASR（Worst-vec. ASR）
- 效用：干净任务完成率（Utility w/o atk.）、受攻击任务完成率（Utility w/ atk.）、拒绝率（Refusal Rate）
- 成本：Token开销（相对基线）
- 综合：有效防御得分（EDS = (1-ASR) × Utility w/o atk.）

## 6. 论文的主要结论与发现

1. **上下文依赖场景下现有防御失效**：文本级防御ASR高达18.8%~34.7%，执行级防御如MELON虽ASR低（1.6%）但效用严重下降（干净任务完成率仅65%），Tool Filter、InjecGuard等也各有严重弱点。
2. **ARGUS实现最佳安全-效用平衡**：ASR仅3.8%，干净任务完成率87.5%，EDS达84.2%，领先第二名（InjecGuard 78.5%）5.7个百分点。唯一在安全、效用、成本三个维度均无明显短板的防御。
3. **各向量均保持低ASR**：最差向量ASR为7.5%（SI），其余向量≤5%，无单向量盲点。
4. **四个安全工具缺一不可**：移除任一个工具，EDS下降4~13个百分点；ContentSegmenter移除后ASR升至25%，证明细粒度溯源至关重要。
5. **白盒自适应攻击下仍鲁棒**：ASR从3.8%微升至5.9%，仍比无防御基线低79%。IA向量最脆弱（2.5%→15.0%），但整体仍安全。
6. **成本可接受**：1.24倍Token开销，在中等可接受范围内。

## 7. 优点（方法/实验设计亮点）

- **问题界定的新颖性**：首次系统揭示现有防御在上下文依赖场景下的三大失效模式（仅凭提示验证、无法接纳良性运行时证据、缺乏参数级溯源），具有理论指导意义。

- **基准的先进性**：AgentLure是首个同时支持上下文依赖任务和上下文感知攻击的基准，覆盖8个攻击向量和6个攻击面，更贴近真实威胁。

- **实用性**：恢复机制（Benign Data Hint）减少任务失败，避免过度拒绝；Token开销适中，适合部署。
- **开放性与可复现性**：开源全部代码、基准和实验脚本。

## 8. 不足与局限

### 8.1 实验覆盖方面
- **基准规模有限**：仅320个样本，4个领域。现实世界中有更多领域（医疗、法律、电商等）和更复杂的任务，泛化性需更大规模评估。
- **LLM骨干单一**：仅使用GPT-4o-mini，未验证在其他模型（如GPT-4o、Claude、Llama）上的表现。防御依赖LLM的偏置能力，可能对其他模型效果不同。
- **Agent框架单一**：基于AgentDojo实现，未在LangChain、AutoGPT等流行框架上测试。

### 8.2 防御局限性
- **载体完整性假设**：ARGUS假设任务载体本身是合法的；攻击者若伪造整个载体（如完全伪造的发票），则所有span被标记为良性，防御失效。论文承认这一点（Case F1），属于供应链攻击，需数据层保护。
- **仅针对提示注入**：未覆盖更广泛的Agent安全威胁（如不安全工具组合、越狱导致策略违例、长期任务意图漂移等）。
- **依赖LLM判断**：四个安全工具（尤其是ContentSegmenter和ArgumentGrounder）自身可能遭受次级注入攻击？论文未充分讨论对抗性对防御LLM的扰动。
- **性能开销**：虽Token开销1.24倍，但增加了延迟（多次LLM调用），对实时性要求高的场景可能不适用。

### 8.3 其他潜在偏差
- **自建基准**：基准任务和攻击可能无意中偏向ARGUS的设计逻辑，尽管自适应攻击缓解了部分偏差。
- **白盒攻击强度**：自适应攻击由单一人为设计，可能未穷尽绕过策略；更系统的自动红队方法（如优化算法）可能发现更多漏洞。

### 8.4 适用性限制
- **需可定义只读工具**：ARGUS需预定义只读工具集（read-only set），若所有工具均可改状态则无法区分。
- **复杂多步骤攻击**：审计单步调用，但攻击可能跨步利用良性行为的累积效应；论文未深入讨论时序关联性。
