## 摘要
随着编码代理获得对shell、仓库和用户文件的访问权限，最小权限授权成为安全部署的先决条件：代理应该获得完成任务所需的足够权限，而不应拥有暴露敏感面的不必要权限。为了研究当前模型是否能够自行推断这一边界，我们首先引入了权限边界推断，其中模型将任务指令和终端环境映射到一个文件级别的读/写/执行策略，以及AuthBench，一个包含120个真实终端任务的基准测试，具有人工审核的权限标签和用于实用性和攻击结果的可执行验证器。AuthBench表明，授权不是一个简单的保守与许可的校准问题：前沿模型常常遗漏执行链所需的权限，同时也授予未使用或敏感的访问权限。增加推理时间推理并不能解决这种不匹配。相反，每个模型都朝着特定于模型的授权吸引子移动：更多的推理使其在自己的失败模式中更加一致，无论是广泛但暴露还是紧密但脆弱。这表明直接策略生成是瓶颈，因为单个生成必须同时发现所有必要的访问并拒绝所有不必要的访问。因此，我们提出了充分性-紧密性分解，它首先通过前向模拟任务生成一个覆盖导向的策略，然后审计每个授予的条目以验证其基础和敏感性。在测试的模型中，这种分解在偏向紧密性的模型上将敏感任务的成功率提高了多达15.8%，同时降低了所有评估模型中的攻击成功率。

## Abstract
As coding agents gain access to shells, repositories, and user files, least-privilege authorization becomes a prerequisite for safe deployment: an agent should receive enough authority to complete the task, without unnecessary authority that exposes sensitive surfaces.To study whether current models can infer this boundary themselves, we first introduce permission-boundary inference, where a model maps a task instruction and terminal environment to a file-level read/write/execute policy, and AuthBench, a benchmark of 120 realistic terminal tasks with human-reviewed permission labels and executable validators for utility and attack outcomes.AuthBench shows that authorization is not a simple conservative-versus-permissive calibration problem: frontier models often omit permissions required by the execution chain while also granting unused or sensitive accesses.Increasing inference-time reasoning does not resolve this mismatch. Instead, each model moves toward a model-specific authorization attractor: more reasoning makes it more consistent in its own failure mode, whether broad-but-exposed or tight-but-brittle.This suggests that direct policy generation is the bottleneck, because a single generation must both discover all necessary accesses and reject all unnecessary ones.We therefore propose Sufficiency-Tightness Decomposition, which first generates a coverage-oriented policy by forward-simulating the task and then audits each granted entry for grounding and sensitivity.Across tested models, this decomposition improves sensitive-task success by up to 15.8% on tightness-biased models while reducing attack success across all evaluated models.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：LLM驱动的编码代理正在从简单的代码补全、推理-行动循环，演变为长期运行、多轮交互的终端操作代理。代理的权限（读/写/执行文件）越来越大，但任务级手动授权难以扩展到完全自主的代理。现实中往往采用粗糙的、与任务无关的默认权限，造成信任-授权不匹配。
- **核心问题**：**最小权限原则**要求代理只获得完成任务所必需的权限，不多也不少。但当前前沿模型能否自主推断出任务层面的权限边界？即给定任务指令和终端环境，模型能否在任务执行前生成一个文件级读写执行策略，使得策略既“充分”（任务能成功完成）又“紧密”（不授予不必要的、敏感的权限）？
- **研究动机**：现有评估要么在权限已设定的情况下测评任务完成能力，要么测评已授权代理的安全性，但都回避了“策略应从何而来”这一上游问题。本文首次形式化**权限边界推断**（permission-boundary inference）作为编码代理的一项独立能力，并构建基准来评估。
- **整体含义**：授权不是简单的保守-许可校准问题。模型常常同时出现**欠授权**（遗漏必要权限导致任务失败）和**过授权**（授予未使用或敏感权限扩大攻击面）。增加推理时间反而使每个模型固化为自己的**授权吸引子**（authorization attractor），而非靠近理想安全窗口。因此直接策略生成存在瓶颈，需要分解方法。

## 2. 方法论：核心思想、关键技术细节、流程

### 2.1 问题形式化

- **输入**：任务指令 I 和终端环境 E。
- **输出**：文件级权限策略 π = (π_read, π_write, π_execute)，每个分量为POSIX路径模式的白名单，未授权访问被拒绝。
- **目标**：最小化 π 超出任务充分边界 π* 的“额外权限”，同时保证 π 足以完成任务（即π* ⊑ π）。
- **关键性质**：π* 取决于完整的配置 (I, E, A)，其中A是执行代理。不同执行代理可能使用不同工具链，因此权限边界不是由任务指令唯一确定的固定地面真值。

### 2.2 充分性-紧密性分解（Sufficiency-Tightness Decomposition）

提出原因：直接策略生成时，模型必须在一次生成中同时决定每个不确定的权限“是否包含”（充分性）和“是否排除”（紧密性），导致冲突。分解将两个决定分开：

- **阶段1：充分性推理（Sufficiency Reasoning）**
  - 生成一个**覆盖导向策略**π_suf，目标是不遗漏执行代理可能需要的任何权限，即使这意味着临时授予多于最终所需。
  - 模型通过前向模拟执行计划、解析传递工具链（解释器、编译器、包管理器、包装脚本等）来枚举所需的读/写/执行权限。
  - 提示词明确告诉模型优先考虑覆盖而非最小性。

- **阶段2：紧密性审核（Tightness Audit）**
  - 对π_suf中的每个条目进行三项评估：
    1. **任务基础**：是否有直接证据表明该权限是必需的？
    2. **范围最小性**：路径模式能否缩小而不失覆盖？
    3. **敏感性检查**：是否与已知敏感面重叠？
  - 决策规则：缺乏基础则移除；可缩小则缩小；与敏感面重叠且非明确必需则移除。
  - 输出最终的紧密策略π_final ⊆ π_suf。

- **设计理由**：先承诺覆盖、再审核剪切，比从不完整策略中恢复缺失权限更容易（剪枝是逐条目有边界的，恢复需要开放式搜索整个环境和工具链）。

### 2.3 授权吸引子分析

- 定义两个诊断负担：
  - **欠授权负担** B_under = 1 - R_macro
  - **过授权负担** B_over = R_macro × (1/P̃ - 1)  （敏感任务中P̃考虑敏感面暴露）
- 将策略映射到“充分性-紧密性空间”，观察模型从低推理努力到高推理努力的位移向量。如果推理努力足够，向量应指向原点(0,0)。实际发现每个模型收敛到自身特定的吸引子（如Gemini偏向高覆盖但高暴露，GPT-5.4和Claude偏向紧密但低覆盖）。

## 3. 实验设计

### 3.1 数据集/场景

- **AuthBench**：由120个真实终端任务组成，分为80个**标准任务**和40个**敏感任务**，覆盖10个领域（文件操作、包构建、日志分析、配置修复、仓库调试、数据转换等）。
- 标准任务：普通终端工作流。
- 敏感任务：在普通任务中添加了现实攻击面（如不安全的辅助脚本、敏感本地文件、凭证类工件、潜在不安全提示），使模型可能触发危险行为。

### 3.2 构建过程

- 数据来源：Terminal-Bench、SWE-Bench、OpenThoughts-TBLite。
- 统一格式：每个任务包含Docker环境、任务指令、安全oracle解决方案、效用验证器V_u、权限评估规范。
- 权限标注：通过真实oracle执行并用strace跟踪，经人工审查去除运行时噪声（临时文件、shell初始化等），得到静态黄金权限S_gold。敏感任务额外标注敏感权限S_sens和攻击验证器V_a。

### 3.3 评估协议

- **权限紧密性**：
  - **静态**：计算生成策略与S_gold在读写执行三个轴上的精确率、召回率、F1（宏平均）。

- **安全暴露**：
  - **静态**：敏感文件暴露率（SER）= 策略中暴露的S_sens占比。
  - **动态**：攻击成功率（ASR）= 执行代理触发任务特定危险终点的比例。

### 3.4 对比方法

- 两个参考配置：
  - **Full-Access**：无限制读写执行权限，作为TSR上限和安全性下限。
  - **Golden-Permission**：直接使用人工标注的S_gold作为执行策略，验证标注质量并展示(I,E,A)依赖性。

## 6. 主要结论与发现

1. **权限生成比任务执行难得多**：模型在完全访问下能高效完成任务，但一旦需先生成权限策略，成功率大幅下降（图3）。授权失败不是执行失败。
2. **欠授权和过授权共存**：模型常常同时遗漏必要权限并授予不必要或敏感权限（图4）。这不是保守-许可的光谱问题，而是同时违背充分性和紧密性。
3. **推理努力放大而非解决冲突**：增加推理时间使每个模型更一致地在自身偏好方向失败（授权吸引子），而非靠近理想点。Gemini偏向高覆盖高暴露，GPT-5.4和Claude偏向紧密但低覆盖。
4. **充分性-紧密性分解有效**：通过分离覆盖生成和紧密审核，对紧偏模型（Claude、GPT-5.4）敏感TSR分别提升13.5%和15.8%，对宽偏模型（Gemini）将SER从34.8%降至15.7%、ASR从28.3%降至12.5%。所有模型ASR均降低。
5. **授权安全与执行安全是解耦的两个维度**：静态暴露（SER）不等于实际攻击实现（ASR），模型可能授权不安全但未利用，也可能安全策略下执行中走危险路径（图6）。两者都应评估。

## 7. 优点

- **创新问题形式化**：首次将权限边界推断作为独立能力定义，填补了现有评估（能力基准、安全基准）之间的缺口。
- **高质量基准构建**：AuthBench基于真实任务，经过人工审核、strace跟踪、双重验证（静态标签+执行验证），确保实用性和可靠性。
- **动态与静态双重评估**：既用静态指标（F1、SER）诊断策略质量，又用动态指标（TSR、ASR）验证实际执行效果，避免单一视角偏差。

- **稳健性检查**：更换执行骨干后结论仍然成立，表明问题源于权限生成而非特定执行器。

## 8. 不足与局限

- **权限范围有限**：仅覆盖文件级读写执行权限，未涉及网络访问、进程间通信、数据库凭证、云资源、API权限等。现实代理需要更丰富的权限模型。
- **单一基准**：仅在AuthBench上评估，缺乏跨基准验证。任务均为终端命令，未涵盖图形界面或浏览器操作。
- **敏感任务攻击面范围有限**：仅包含某些特定类型（凭证文件、辅助脚本、工具链），可能存在遗漏的攻击模式。
- **推理努力分析限于可配置模型**：部分模型未暴露推理层级，吸引子现象可能不适用于所有模型族。
- **静态标签偏差**：S_gold基于oracle方案，并非所有可能成功路径的权限。不能保证策略对任何执行代理都最优。Golden-Permission的TSR低于Full-Access即是证据。
- **评估效率问题**：动态指标（TSR、ASR）需要实际执行任务，可能带来时间和成本。静态指标虽快但只是代理。
- **缺乏训练方法**：本文仅提出推理时分解策略，未探索通过微调或训练来提升模型的权限推断能力。
- **多代理系统未涉及**：当前仅评估单代理场景，未来在多代理协作环境中权限推断更复杂。
