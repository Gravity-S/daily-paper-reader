## 摘要
随着组织转向部署执行非确定性工作流、维护有状态会话、通常具有访问内部服务特权的AI代理，工程挑战从构建单个代理转向在适当的隔离、治理和安全条件下大规模运行它们。在本文中，我们介绍了Agyn，一个围绕适合代理工作负载的三个关键原则设计的开源平台：基于Kubernetes的信号驱动、有状态无服务器运行时；用于代理和控制器定义的Terraform提供程序；以及基于零信任和最小权限原则的安全模型。Agyn是代理无关、模型无关和云无关的。

## Abstract
As organizations move toward production deployments of AI agents, which execute non-deterministic workflows, maintain stateful sessions, and often operate with privileged access to internal services, the engineering challenge shifts from building individual agents to operating them at scale with proper isolation, governance, and security. In this paper we present Agyn, an open-source platform designed around three key principles tailored for agent workloads: a signal-driven, stateful serverless runtime on Kubernetes; a Terraform provider for agent and harness definition; and a security model grounded in zero-trust and least-privilege principles. Agyn is agent-agnostic, model-agnostic, and cloud-agnostic.

# Agyn 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：随着 AI 代理从实验室走向生产环境，组织面临三个无法同时满足的关键需求：
  1. **高效按需执行**：无法为每个代理定义预先预留专用计算资源，需实现空闲资源回收和按需启动。
  2. **代理定义即代码**：代理的提示词、工具、密钥等应像基础设施一样接受版本控制、代码审查和可回滚。
  3. **零信任访问内部服务**：每个代理实例必须拥有独立身份、默认拒绝访问、凭据不暴露给模型，实现最小权限。
- **背景**：现有解决方案（如 Knative、AWS Lambda、IaC 框架、零信任网络）仅部分满足上述需求，缺乏集成平台。因此，Agyn 应运而生，目标是提供一个开源、代理无关、模型无关、云无关的一体化平台。

## 2. 方法论：核心思想与关键技术细节
- **核心思想**：将 AI 代理视为有状态的短生命周期计算单元，通过信号驱动、基础设施即代码、零信任安全三大原则实现生产级运营。
- **关键技术细节**：
  - **运行时：信号驱动的有状态无服务器模型**
    - 每个代理实例绑定到单个用户和单个会话线程（thread）。
    - 消息到达时，Notifications 服务发布事件，Agents Orchestrator 订阅事件，解析代理定义、获取秘密、向 OpenZiti 请求身份，并通过 k8s-runner 创建 Kubernetes Pod。
    - Pod 挂载持久卷（保留工作区状态），接收线程上下文，实现冷启动后恢复状态。
    - 空闲超时（默认 5 分钟）后，Orchestrator 停止 Pod 并删除网络身份；下次消息到来时重新生成实例，状态保持不变。
  - **配置：Terraform 提供程序实现代理即代码**
    - 代理定义（容器、工作区配置、系统提示、模型、密钥、MCP 服务器、持久卷等）通过 Terraform 声明。
    - 内置三个预构建代理镜像（Claude Code、OpenAI Codex、自家 agent-loop），也支持自定义容器。
    - Terraform 模块可打包通用 harness 模式为共享模块，避免重复。
    - `terraform apply` 后，定义通过 Gateway 解析到 Agents Service，后续消息即按新定义启动 Pod，无需重启。
  - **安全：零信任与最小权限**
    - **容器级隔离**：主容器运行代理进程，MCP 服务器作为 sidecar 容器，各自独立文件系统和进程树，仅共享 loopback 接口；秘密仅注入需要它的 sidecar，绝不进入代理容器。
    - **零信任覆盖网络（OpenZiti）**：每个代理实例在启动时获得独立 x509 身份，mTLS 握手后 Gateway 提取身份并传播到下游服务，实现调用链身份感知；ABAC 策略限制每个代理仅能访问明确许可的服务。
    - **基于关系的授权（ReBAC）**：使用 OpenFGA（受 Zanzibar 启发）通过图遍历实施细粒度权限，如所有者、维护者、参与者角色。

## 7. 优点
- **思想系统性**：将生产级运营需求（弹性、治理、安全）整合到一个平台，而非碎片化方案。
- **技术实现亮点**：
  - 信号驱动 + 持久卷实现了类似 FaaS 的弹性，同时保持有状态对话能力。
  - 通过 Terraform 提供程序，让代理定义像基础设施一样受版本控制，降低运维门槛。
  - 零信任覆盖网络与容器级 sidecar 隔离的组合，有效应对间接提示注入等安全威胁。
