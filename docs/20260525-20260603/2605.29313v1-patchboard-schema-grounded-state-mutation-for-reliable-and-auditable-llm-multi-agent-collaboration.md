## 摘要
LLM多智能体系统通常通过自然语言对话或松散结构的共享内存进行协调，使得中间状态难以验证、归因和审计。我们引入PatchBoard，一种基于模式的协作架构，用对共享结构化状态的经过验证的JSON Patch变异取代了智能体间的对话。一个架构师智能体构建任务特定的模式和workflow规则，而一个确定性内核在事务性地提交每个提议的状态变异之前，根据模式约束、角色特定的写契约和运行时不变性对其进行验证。在630个匹配的ALFWorld片段上，PatchBoard实现了84.6%的成功率，而LangGraph为30.8%，Flock为61.6%，同时每个成功任务的令牌数减少到45.5k，而它们分别为368.3k和64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

## 1. 核心问题与整体含义（研究动机和背景）

- **当前LLM多智能体系统的痛点**：现有系统（如AutoGen、ChatDev、MetaGPT）主要依靠自然语言对话或松散结构的共享内存进行智能体间协调。这种机制导致中间状态（如计划、证据、动作意图）难以验证、归因和审计。一个下游智能体可能将未经验证的观察、过时的计划或错误的中间声明当作可靠的任务状态，从而导致错误传播和任务失败。
- **关键问题**：缺少系统级的边界来区分“模型建议”和“已提交的共享状态”，一旦被污染的信息进入共享上下文，后续所有智能体都可能放大错误。
- **研究意义**：提出一种新的协作范式，将多智能体交互从自由形式的对话转变为**基于模式（schema）的、经过验证的状态变异**，从而提升协作的可靠性、可审计性和成本效率。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：用**JSON Patch**操作（RFC 6902）替代自然语言消息作为智能体间的通信接口。智能体只能通过提议对共享JSON状态树进行受限制的补丁式修改，这些修改必须通过确定性内核的验证才能成为已提交状态。
- **关键技术细节**：
  - **架构师（Architect）智能体**：在任务初始化时，根据用户请求生成一个**任务蓝图（blueprint）**，包含：
    - 全局状态模式（JSON Schema）
    - 工人（worker）契约（定义每个角色的读写路径、上下文预算、补丁格式约束）
    - 工作流规则（基于状态事件触发后续工人调用）
    - 上下文预算（限制每个工人可观察的状态大小）
  - **确定性内核（Deterministic Kernel）**：运行时的核心控制器，负责：
    - 初始化全局状态树
    - 为每个工人构造受限的上下文视图（budgeted context views）
    - 验证工人提出的JSON Patch（检查语法、路径授权、模式约束、运行时不变性）
    - 事务性提交通过验证的补丁，拒绝的补丁仅记录日志而不修改状态
    - 管理工作流调度（根据已提交的状态变化触发后续工人）
  - **状态变异验证公式**：  
    接受条件 = 语法合法 ∧ 路径授权 ∧ 应用后状态非空 ∧ 后状态满足模式 ∧ 满足运行时不变性
  - **上下文视图**：内核根据角色读契约和预算，从全局状态中切片出工人所需的字段，大集合用摘要和句柄表示，避免暴露无关信息。
  - **事务日志**：记录每个工人的输入视图哈希、提议的补丁、验证结果、结果状态哈希，支持可重放审计。

## 6. 主要结论与发现

- **任务成功**：PatchBoard在ALFWorld上达到**84.6%成功率**，远高于LangGraph（30.8%）和Flock（61.6%）。
- **成本效率**：每成功任务仅消耗**45.5k tokens**，而LangGraph为368.3k，Flock为64.2k。
- **共享记忆并非充分条件**：Structured JSON Blackboard虽降低token消耗，但成功率（77%）仍低于PatchBoard，表明事务性验证和写契约贡献额外增益。
- **消融表明核心组件**：去除补丁/模式接口和上下文切片导致成功率下降最大（-15%），且token增加显著。
- **上下文预算**：较小的预算（1k字符）表现最佳，增加预算未提升成功率。
- **架构师生成schema优于固定schema**：生成式schema带来更高成功率和更低token消耗。
- **故障隔离**：PatchBoard对Invalid JSON、Bad Path/Type、Unauthorized Write的污染率为0%；假声明仍可能（43%被接受）；循环停止率达96%。
- **语义限制**：HotpotQA诊断显示，所有系统答案准确率相近，PatchBoard不能消除无法入证据的声明，说明模式验证不能保证事实正确。

## 7. 优点

- **方法创新**：将多智能体通信从自由文本约束为模式化的、可验证的状态变异，提供了系统级的可靠性边界。
- **可审计性**：事务日志记录每个已接受/拒绝的补丁，可完整回放状态演化过程。
- **成本控制**：通过受限上下文视图和角色写契约，显著减少无关信息暴露和token浪费。

- **实验严谨**：大量控制实验和消融实验，验证了每个组件的贡献；故障注入证明系统对格式错误和越权写入的免疫性。
