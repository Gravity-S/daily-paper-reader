## 摘要
我们提出了 HORIZON，一个将硬件设计视为仓库级代码演化的自演化智能体框架。一个 Markdown 工具链被编译成一个项目包，包含领域知识、可执行评估器、验收谓词以及 git/运行时策略；一个免操作智能体循环随后演化一个隔离的 git 工作树，利用仓库操作进行状态管理、追踪和重放。这将先前从 EDA 软件系统到硬件设计工件本身的仓库级自演化工作进行了扩展。我们在 ChipBench、RTLLM、Verilog-Eval 以及九个 CVDP 类别上评估了我们的方法，通过完全免操作的智能体循环在所有测试套件上实现了 100% 的基准完成率。然而，我们并不声称硬件设计的代理型 AI 问题已解决：这些基准只是芯片设计中更广泛工程问题的受控代理。第~\ref{sec:discuss} 节审视了当前研究的局限性，并指出了开放的研究挑战。

## Abstract
We present HORIZON, a self-evolving agent framework that treats hardware design as repository-level code evolution. A Markdown harness is compiled into a project pack containing domain knowledge, an executable evaluator, an acceptance predicate, and a git/runtime policy; a hands-free agent loop then evolves an isolated git worktree, using repository operations for state management, tracing, and replay. This extends prior works of repository-scale self-evolution from EDA software systems, to hardware-design artifacts themselves. We evaluate our approach on ChipBench, RTLLM, Verilog-Eval, and nine CVDP categories, achieving 100\% benchmark completion across all suites with a fully hands-free agentic loop. However, we do not claim that agentic AI for hardware design is solved: these benchmarks are controlled proxies for a much broader engineering problem in chip design. Section~\ref{sec:discuss} examines the limitations of the current study and highlights open research challenges.

## 2. 方法论：核心思想、关键技术细节、公式/算法流程

- **核心思想**：HORIZON 将硬件设计问题封装为一个独立的git工作树，包含：
  - 一个**结构化Markdown harness**（用户输入），包含目标、领域知识、评估器规范、验收谓词。
  - 一个**引导智能体**将该harness编译成**项目包**，包括：智能体策略提示和工具约定（π_agent）、可执行评估器（E_p）、验收谓词（A_p）、版本控制与工件策略（Γ_p）、领域技能和仓库指令（Ω_p）。
  - 一个**免操作智能体循环**：在隔离的git工作树中迭代地编辑、评估、提交或拒绝候选版本，直到验收条件满足。

- **关键技术细节**：
  - **git作为核心基板**：git不仅用于版本控制，还提供隔离的演化环境、状态跟踪（diff）、提交（commit）作为验收检查点、日志存储评估器证据、历史记录可回放。

  - **跟踪缓冲区**：基于git原生构建，每个提交附带评估结果和奖励；历史即经验缓冲区。
  - **会话复用与缓存**：持久模型会话跨迭代复用，使大部分token为缓存输入（约91%），降低边际成本。
  - **算法流程**：用户提供Markdown harness → 引导智能体生成项目包 → 免操作循环：生成/编辑候选工件 → 运行评估器 → 评分 → 满足验收则提交，否则拒绝并记录 → 重复直到收敛或预算耗尽。

## 6. 主要结论与发现

- **基准完成率**：HORIZON在**所有测试套件上达到了100%的pass rate**，且完全免操作（hands-free）。唯一未通过的ChipBench任务是由于原始基准的规约-工具链缺陷。
- **收敛路径差异大**：简单生成套件（RTLLM-2.0、Verilog-Eval）在2次迭代内饱和；而复杂类别（如CID 002代码补全、CID 004修改）需要最多82次迭代。
- **token消耗集中**：总消耗约210M token，其中97.1%用于CVDP类别；CID 002（代码补全）消耗最多（56M token），CID 013（检查器生成）虽起始极低（3.8% pass）但仅消耗14.2M token即收敛。
- **缓存效果显著**：约91%的token是缓存输入，大大降低了API成本。
- **覆盖率诊断**：在激励生成（CID 012）中，平均覆盖率从86.5%提升至97.9%，但未达100%是因为验收门限是“通过测试集”，而非覆盖率。断言生成（CID 014）起始覆盖率已接近98%，迅速饱和。
- **主要瓶颈**：正确性收敛后，主要研究瓶颈变为收敛效率和验证质量。

## 7. 优点

- **框架设计新颖**：将git版本控制与可执行反馈循环深度整合，使硬件设计问题像软件工程一样可演化、可追踪、可回放。
- **免操作与通用性**：从用户只需提供Markdown harness到智能体全自动完成，且框架不依赖特定生成器（GPT-5.3仅为实例），可扩展至RTL、EDA软件、流程探索等。
- **透明度高**：提供每次迭代的pass rate、token消耗、覆盖率等详细信息，便于分析收敛行为。
- **诚实承认局限**：明确讨论reward hacking风险、反馈延迟问题，并建议未来基准设计分离调试反馈与最终评分，符合学术严谨性。
- **系统性评估**：覆盖了从简单生成到验证生成的多种任务类型，并区分了开放源码和商用EDA后端。

## 8. 不足与局限

- **缺乏横向对比**：未与AutoChip、RTLFixer、VerilogCoder、MAGE、ACE-RTL等现有RTL agent系统进行定量比较，无法证明框架优越性。
