## 摘要
大语言模型已从独立的文本生成系统逐渐演变为能够调用外部工具、访问环境并执行多步骤任务的智能体。然而，传统的函数调用基准主要评估任务完成度和API正确性，而隐私评估基准通常关注最终响应或隐私判断。这两种视角均未能捕捉跨执行的多工具轨迹中受目的约束的信息流。鉴于当前智能体评估中存在的这一局限，ToolPrivacyBench旨在审计任务私有原子是否仅被路由至授权工具及下游接收端，从而评估工具使用过程中的任务完成度与隐私过度披露情况。该基准包含2150个案例，其中1150个为完全合成的隐私敏感型业务工作流，1000个源自现有多工具与函数调用基准。每个案例均配备策略知识库。当智能体针对模拟业务后端执行后，评估器将记录的参数和后端审计日志与策略知识库进行比对。评估覆盖9款广泛使用的智能体，以刻画目的约束下的隐私过度披露特征。结果表明：成功的工具执行并不等同于恰当的隐私披露——智能体可能在完成任务的同时，通过中间工具调用传输不必要的隐私信息。因此，ToolPrivacyBench定义了一种“按需知晓”的披露边界，即每个工具仅应接收其既定目的所必需的信息，并通过轨迹级审计识别多工具工作流中的隐私过度披露。

## Abstract
Large language models (LLMs) have increasingly moved from standalone text generation systems to agents that invoke external tools, access environments, and execute multi-step tasks. However, conventional function-calling benchmarks mainly evaluate task completion and API correctness, while privacy evaluation benchmarks typically focus on final responses or privacy judgments. Neither perspective captures purpose-bound information flow across an executed multi-tool trajectory. Motivated by this limitation in current agent evaluation, ToolPrivacyBench audits whether task-private atoms are routed only to authorized tools and downstream sinks, thereby evaluating both task completion and privacy over-disclosure during tool use. The benchmark contains 2,150 cases, including 1,150 fully synthetic privacy-sensitive business workflows and 1,000 cases adapted from existing multi-tool and function-calling benchmarks. Each case is represented by a policy knowledge base. After an agent executes against mock business backends, the evaluator compares recorded tool arguments and backend audit logs with this policy knowledge base. The evaluation covers nine widely used agents to characterize purpose-bound privacy over-disclosure. The results show that successful tool execution does not imply appropriate privacy disclosure: an agent may complete a task while transmitting unnecessary private information through intermediate tool calls. ToolPrivacyBench therefore formalizes a need-to-know disclosure boundary, under which each tool should receive only the information necessary for its stated purpose, and uses trajectory-level auditing to identify privacy over-disclosure in multi-tool workflows.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：当前大语言模型（LLM）已从单纯文本生成演变为调用外部工具、执行多步骤任务的智能体（Agent）。现有基准主要评估任务完成度和 API 正确性（如 API-Bank、BFCL、τ-bench），而隐私评估主要关注模型记忆、训练数据提取或最终输出中的隐私泄露（如 Secret Sharer、PrivLM-Bench、PrivacyLens）。这两种视角都**无法捕捉在多工具执行轨迹中，私有信息是否被路由到不应接收的工具或接收端**，即违反了“目的绑定（purpose-bound）”的披露边界。  
- **整体含义**：高性能的工具调用能力并不等于隐私合规。智能体可能正确完成任务，但在中间步骤将不必要的私有信息（如症状、身份证号、API 密钥）写入工单、通知、交接消息等不应使用这些信息的渠道，造成过度披露。论文旨在形式化这一问题，并提出首个针对多工具工作流的目的绑定隐私审计基准 ToolPrivacyBench。

## 2. 方法论

### 2.1 核心思想

- **按需知道（need-to-know）原则**：每个工具只应接收其声明的目的所必需的信息，而非将所有上下文传递给所有工具。私有原子的授权状态随工具目的和接收端类型变化，而非全局敏感或非敏感。
- **知识驱动的审计**：使用显式策略知识库（Policy Knowledge Base）定义私有原子、工具目的、接收端类型、授权/禁止关系。策略库**不提供给智能体作为运行时约束**，仅用于执行后的审计。

### 2.2 关键技术细节

1. **案例表示**：每个案例由四部分组成：  
   - 用户任务 x  
   - 工具集合 T（每个工具包含名称、目的、输入模式、接收端类型）  
   - 当前任务私有原子集合 P（包含类型、值、别称、严重性）  
   - 字段-工具授权矩阵 A（0/1 矩阵，表示某私有原子对某工具是否授权）  

2. **策略知识库**：包含私有原子节点、工具目的节点、信息接收端节点，以及它们之间的授权关系。通过目标函数将关系投影为授权矩阵。

3. **执行与审计流水线**：  
   - 智能体在 OpenClaw 包装器下调用工具，工具通过 Python 桥接调用模拟业务后端。  
   - 后端记录所有接收的参数到审计日志。  
   - 评估器中的披露检测器（Disclosure Detector）识别参数中的私有原子（精确匹配、别名、语义变体）。  
   - 授权推理器（Authorization Reasoner）查询策略库判断每个被发现原子-工具对是否授权。  
   - 度量聚合器计算效用和隐私指标。

4. **关键公式（用文字说明）**：  
   - **FOR (Field Opportunity-Normalized Over-Disclosure Rate)**：在所有被执行的工具调用中，计算“禁止的字段-工具机会”中实际发生披露的比例。  
   - **SWLR (Severity-Weighted Leakage Rate)**：在 FOR 基础上加入私有原子的严重性权重，反映高敏感字段的泄露风险。  
   - **ToolFOR**：按工具聚合的 FOR，定位高风险的接收端。  
   - **FreeTextFOR**：仅针对自由文本字段（如 message、notes）的 FOR。  
   - **MidFOR**：仅针对中间工作流步骤（非最终步骤）的 FOR。  
   - **MT-POI (Multi-Tool Privacy Over-Disclosure Index)**：加权组合 FOR、SWLR、ToolFOR、FreeTextFOR、MidFOR，提供 0-100 的总体隐私风险分（越低越好）。  
   - **SMTC (Safety-Maintained Task Completion)**：TaskSuccess × (1 - MT-POI/100)，联合评估效用与隐私。  
   - 效用度量 TaskSuccess 由任务结果、工作流覆盖率和必要事实传递的几何平均组成。

## 3. 实验设计

### 3.1 数据集

- **公开派生拆分**：1000 个案例改编自 τ-bench、API-Bank、BFCL、AppWorld 等现有基准，保留了原始工具调用结构，新增私有原子、授权注释和自由文本接收端。平均 8.89 个工具、12.77 个私有原子。  
- 总计 2150 个案例，包含 20,817 个私有原子实例和 219,986 个字段-工具授权对。

## 6. 主要结论与发现

1. **任务完成≠隐私合规**：所有模型在合成私有拆分上的 TaskSuccess 高达 92-98%，但 MT-POI 仍在 19-28 之间，表明存在普遍过度披露。  
2. **高完成度可能伴随更高披露**：例如 Qwen3.6-plus、Kimi K2.5 的 TaskSuccess 最高（97.70），但 MT-POI 也最高（>27），而 Gemini 3.5 Flash 的 TaskSuccess 较低（92.45）但隐私风险最低（MT-POI 19.19）。  
3. **过度披露集中在协作接收端**：工单（Ticket）FOR 高达 51.43%，交接（Handoff）34.79%，而后端记录（Backend record）仅 5.79%。  
4. **自由文本字段是高危通道**：message、description、work_notes 的 FTSlotRate 超过 80%，description 的 FreeTextFOR 达 45.82%。  
5. **路径级泄漏**：记录→工单过渡是首次泄漏最主要的发生点（37.27%），且具有高传播风险（RiskyTransitionRate 96.89%）。  
6. **域和私有原子类型影响**：安全敏感、医疗-金融等类型的私有原子泄露更严重；低权限团队渠道是最常见的未授权接收端。  
7. **代表性失败模式**：目的特定上下文折叠、渐进式工作流扩散、自由文本放大。

## 7. 优点

1. **问题定义清晰**：首次形式化多工具工作流中的目的绑定隐私过度披露问题，区分于传统记忆泄露或最终输出检测。  
2. **基准规模大且领域多样**：2150 个案例覆盖 10+ 业务领域，既有合成控制案例，也有公开改编案例，提高了泛化性。  
3. **可执行审计**：使用真实工具调用和模拟后端记录，而非仅离线 JSON 检查，更贴近真实运行时隐私风险。  
4. **细粒度诊断指标**：提出 FOR、SWLR、ToolFOR、FreeTextFOR、MidFOR、MT-POI、SMTC 等多层次指标，支持从整体到局部的分析。  
5. **模块化审计系统**：策略知识库、披露检测器、授权推理器、度量聚合器独立设计，便于扩展和复现。  
6. **丰富诊断分析**：不仅报告总分，还深入分析接收端、自由文本、工作流路径、域、私有原子类型等维度，为模型改进提供具体方向。  
7. **非对抗性场景**：评估的是良性任务中由模型自身导致的非故意泄露，贴近实际部署场景。

## 8. 不足与局限

1. **合成数据与现实差距**：大部分案例使用合成或测试值，模拟后端，未使用真实业务系统、真实用户数据或真实权限控制，可能无法完全反映生产环境中的隐私风险。  
2. **策略知识库标注偏差**：授权标注基于人工判断（按最小必要原则），不同标注者间可能存在不一致（尽管报告了 93.1% 同意率和 κ=0.86），仍可能引入主观性。  
3. **仅评估非对抗性场景**：未覆盖提示注入、越狱、恶意请求等对抗性攻击，攻击下隐私风险可能更大。  
4. **模型黑盒性**：仅比较了不同模型的表现，未分析导致泄露的内部机制（如注意力分布、上下文压缩策略等）。  
5. **资源消耗未公开**：未提供训练或推理所需的算力信息，不利于其他研究者复现或比较成本。  
6. **单次执行**：未进行多轮重复（如不同随机种子）来评估指标稳定性，可能受偶然因素影响。  
7. **工具集固定**：每个案例的工具集和流程固定，未评估模型在更动态（如未知工具出现）场景下的表现。  
8. **未考虑长期记忆或跨任务泄露**：仅聚焦当前任务内的目的绑定，未评估智能体共享记忆或跨任务上下文污染的风险。
