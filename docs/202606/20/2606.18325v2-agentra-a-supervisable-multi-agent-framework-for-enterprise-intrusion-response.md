## 摘要
企业入侵响应仍然依赖于静态剧本和分析人员驱动的分类，导致告警生成与事件遏制之间存在延迟。我们提出了Agentra，一个可监督的多智能体入侵响应系统（IRS）框架，它将来自IDS、EDR和XDR平台的告警转换为基于MITRE ATT&CK、MITRE D3FEND和NIST CSF 2.0的结构化应急响应计划。Agentra将响应推理分解到具有角色范围的智能体中，通过有界的规划器-验证器审查循环验证提出的计划，通过审核人安全网关筛选检索到的威胁情报，通过行动目录和风险评分对行动进行门控，并将决策记录在仅追加的审计日志中。我们在一个从ThreatHunter-Playbook、Splunk BOTSv3和DARPA OpTC提取的120个事件语料库上，将Agentra与静态的OASIS CACAO v2.0网络剧本基线进行了评估。最强配置将考虑假阳性的IRS F1分数从0.61提升到0.84，并在仅使用规划器配置引入不安全过度反应后，将预计的有害行动率恢复至静态基线水平0.0%。这些结果表明，多智能体响应规划可以在保留分析人员审批和可审计性的同时，提高基于本体的IRS覆盖率。

## Abstract
Enterprise intrusion response still depends on static playbooks and analyst-driven triage, creating delay between alert generation and containment. We present Agentra, a supervisable multi-agent Intrusion Response System (IRS) framework that converts alerts from IDS, EDR, and XDR platforms into structured incident response plans grounded in MITRE ATT&CK, MITRE D3FEND, and NIST CSF 2.0. Agentra decomposes response reasoning across role-scoped agents, validates proposed plans through a bounded Planner--Validator review loop, screens retrieved threat intelligence through a Moderator security gateway, gates actions through an Action Catalog and risk score, and records decisions in an append-only audit log. We evaluate Agentra against a static OASIS CACAO v2.0 cyber-playbook baseline on a 120-event corpus drawn from ThreatHunter-Playbook, Splunk BOTSv3, and DARPA OpTC. The strongest configuration improves FP-aware IRS F1 from 0.61 to 0.84 and restores the projected harmful-action rate to the static baseline level of 0.0% after Planner-only configurations introduce unsafe overreaction. These results indicate that multi-agent response planning can improve ontology-grounded IRS coverage while preserving analyst approval and auditability.

## 1. 核心问题与整体含义（研究动机与背景）

- **现状矛盾**：现代安全检测已大幅加速（如平均 eCrime 突破时间仅 29 分钟），但响应仍依赖静态剧本和人工分析，导致告警生成到处置之间存在严重延迟。
- **现有局限**：SOAR 剧本虽可重复执行，但无法适应多变战术、跨域攻击链及工具间缝隙；分析师认知能力成为 SOC 瓶颈。
- **目标**：构建一个**可监督的、基于多智能体**的入侵响应系统（IRS），将 IDS/EDR/XDR 告警自动转化为结构化的应急响应计划，同时保留分析师审批、审计追踪和安全控制，避免完全自主执行带来的风险。

## 2. 方法论：核心思想与关键技术细节

### 2.1 核心思想

- **职责分离**：将响应推理分解为多个角色受限的智能体（Planner、Validator、Moderator、Executor），各司其职，无单一智能体拥有完整控制权。
- **分层监督**：通过「Planner 提案 → Validator 审查 → Moderator 安全网关 → Analyst 批准 → Action Catalog + 风险评分 → 可审计日志」的流水线，确保每步可追溯、可干预。
- **本体接地**：所有响应计划基于 MITRE ATT&CK（攻击技术）、MITRE D3FEND（防御措施）、NIST CSF 2.0（安全框架）和 OASIS CACAO v2.0（标准化剧本格式）构建。

### 2.2 关键技术细节

- **Incident Context 构建**：将告警证据标准化为包含 IoC、根因假设、时间线、ATT&CK 技术、D3FEND 候选措施、NIST CSF 功能的类型化记录。
- **Planner - Validator 循环**：
  - Planner 在 Moderator 清洗后的上下文中起草响应计划；
  - Validator 独立评估计划并返回批准/拒绝及理由；
  - 若未达 k-of-n 阈值（实验中 n=1, k=1但有界多轮），则 Planner 根据反馈修订，最多 R 轮；超限则升级给分析师。
- **Moderator 安全网关**：
  - 拦截所有内外部检索内容，筛查指示注入、允许列表违规、幻觉标识、跨站脚本、数据投毒等；
  - 可疑内容被重写或拒绝；Planner 不直接接触外部威胁情报。
- **Gated Execution**：
  - Action Catalog 定义网络、身份、终端、ITSM 四类允许操作；
  - Sanity Checker 检查动作一致性（如依赖已隔离资产）；
  - Risk Score 综合资产关键性、爆炸半径、可逆性，映射为 `advise/approve/senior-approve` 三级授权；
  - `StateChanged()` 检测规划时的系统快照是否改变，若变则失效并重新分析。
- **Append-Only Audit Log**：所有 Prompt、检索、审核投票、风险评分、分析师决策均以 Merkle 树哈希链式存储，实现防篡改。

### 2.3 形式化算法流程（文字说明）

1. 输入：告警证据 E，剧本库 L，动作目录 C，验证器池 V，批准阈值 k，修订上限 R，风险阈值 Θ，审计日志 A。
2. 构建 Incident Context I；
3. 通过 Graph-RAG 和外部检索获取知识 Xint/Xext → Moderator 合成清洗后的 X；
4. Planner 生成候选计划 P（基于 I, X, L）；
5. 循环 t=1..R：每个 Validator vi 对 P 投票 -> 若批准数 ≥k 则通过，否则反馈给 Planner 修订；
6. 若 R 轮未通过则 escalate，返回 escalate；
7. 分析师审查 d，若拒绝则返回 rejected；
8. 检查 StateChanged，若变则 invalidate；
9. 逐动作检查是否属于 Catalog、通过 Sanity Check、计算风险评分、按授权层级决定执行或 hold；
10. 所有步骤记录到 A；返回最终计划 P*。

## 3. 实验设计

### 3.1 数据集 / 场景

- **总语料**：120 个事件，来自三个公开数据集：
  - **ThreatHunter-Playbook (THP)**：恶意入侵活动；
  - **Splunk BOTSv3 (BOTS)**：恶意 + 误报压力事件；
  - **DARPA OpTC (OPTC)**：良性管理活动。
- **标签分布**：58 个恶意事件，62 个良性/误报压力事件；覆盖 43 种 MITRE ATT&CK 技术/子技术。
- **预处理**：所有事件被重新舞台为 Kubernetes 审计事件，统一格式；标识符进行确定性匿名化。
- **基准（baseline）**：静态 OASIS CACAO v2.0 剧本查找（A0），直接根据 ATT&CK 技术匹配预定义剧本，无 LLM。

### 3.2 对比方法

- **五步消融实验（A0 ~ A4）**：
  - A0：CACAO 查找（无 LLM）；
  - A1：+ Planner（LLM 生成或增强 CACAO）；
  - A2：+ Validator 辩论；
  - A3：+ Web 爬虫 + Moderator（外部威胁情报 + 清洗 + D3FEND 广度检查）；
  - A4：+ Graph-RAG（企业配置知识，完整堆栈）。
- **五种 Planner-Validator 组合（M1~M5）**：
  - M1：Qwen3-32B-Instruct + Mistral-Small-3.2-24B（replace-CACAO）；
  - M2：同 M1 但 enhance-CACAO；
  - M3：Qwen3-32B + Foundation-Sec-8B-Reason（enhance）；
  - M4：Claude Sonnet 4.6 + Foundation-Sec-8B-Reason（enhance）；
  - M5：GPT-5 + Claude Sonnet 4.6（enhance）。
- 评估维度：FP-aware IRS F1、假阳性率、D3FEND 覆盖率/精度、预计有害动作率、延迟（约 40s/事件，批量分级）。

## 4. 资源与算力

- **硬件**：单张 NVIDIA H200 NVL（143 GB HBM3e）。
- **模型部署**：本地模型通过 vLLM + FP8 量化；包括 Qwen3-32B、Mistral-Small-3.2-24B、Foundation-Sec-8B-Reason；API 模型（Claude Sonnet 4.6, GPT-5）远程调用。
- **限制说明**：单 GPU 只能同时 host 一个 Planner 和一个 Validator（来自所选模型类）；无法并行运行多个大型异构 Validator。实验使用单 Validator 实例，k-of-n 通用设计尚未在多加速器上评估。

## 5. 实验数量与充分性

- **实验总数**：5 种消融 × 5 种模型组合 = 25 个配置点，每个配置点报告一个种子（seed=42）的指标。
- **消融设计**：逐层添加组件，清晰展示每个阶段贡献。
- **度量全面**：包含 FP-aware F1、召回、精度、误报率、D3FEND 覆盖/精度、有害动作率等。还分析了收敛率（计划在时限内被 Validator 批准的比例）和每数据集 F1。
- **局限**：
  - 仅单种子，未报告多种子分散性或 McNemar 显著性检验（作者声明将延迟至扩展版）；
  - D3FEND 参考标签由人工映射，非穷举真实 ground truth；
  - 有害动作率基于静态依赖覆盖的投影，非实际执行测量；
  - 无对抗性检索实验，Moderator 对提示注入的鲁棒性未验证；
  - 语料总量 120 事件较小，且所有数据集在模型训练前公开，存在记忆化风险（虽已匿名化）。

整体而言，实验设计 **充分且客观**（在同一协议下对比消融），但受限于规模、种子数和对抗测试缺失。

## 6. 主要结论与发现

1. **A3 (Moderator + CTI) 是质量转折点**：引入外部威胁情报清洗后，所有组合的 D3FEND 精度从 0.15~0.30 跳升至 0.68~0.95，有害动作率降至 ≤0.9%。
2. **全堆栈 A4 达到最佳平衡**：
   - M1（全本地）取得最高 FP-aware F1=0.84，召回 0.90，FP率 0.47；
   - M4（API Planner）达到完美召回 1.00，但 FP率 0.90，F1 仅 0.81；
   - M5 获得最高 D3FEND 精度 0.95 但召回较低 0.57。
3. **Planner 单独使用会导致不安全过度反应**：A1 配置下有害动作率攀升至 14~16%，而静态基线 A0 为 0.0%。加入 A3/A4 后恢复至 0.0%。
4. **Replace-CACAO vs Enhance-CACAO**：在完整堆栈下，Replace 模式（M1）优于 Enhance（M2），说明在强本地 Planner 下，A3/A4 接地可补偿缺失的查找起点。
5. **Validator 的作用依赖于组合**：对某些组合（如 M5）Validator 过于严格导致召回崩溃（从 0.91 降到 0.14），需结合封闭词典和广度检查（A3）才能稳定。

## 7. 优点

- **架构清晰、可监督**：职责严格分离，无单一智能体能独立完成执行，符合企业安全要求。
- **审计可追溯**：所有决策均通过 Merkle 树记录，防篡改。
- **模块化设计**：Incident Context、Planner、Validator、Moderator、Executor 均解耦，便于替换升级。
- **安全控制多层堆叠**：检索清洗、动作目录、风险评分、状态变化检测等多项措施协同降低风险。
- **开源本体支撑**：直接利用 MITRE ATT&CK/D3FEND、NIST CSF、CACAO 等成熟标准，结果可解释、可复用。
- **消融实验设计严谨**：能够量化每个组件的边际贡献（尤其是 A3 的关键作用）。

## 8. 不足与局限

- **实验规模有限**：仅 120 个事件、单一种子；未做多轮随机种子测试，缺乏统计置信区间。
- **静态语料局限性**：基于公开数据集并重新舞台，记忆化风险仅被降低但未消除；恶意事件类型覆盖率虽广但深度不够。
- **Validator 多样性不足**：实际只用了单 Validator 实例，未测试 k-of-n 多验证容错。作者解释受限于单 GPU 显存，但仍是架构评价的空白。
- **无对抗性鲁棒性评估**：Moderator 安全网关对提示注入、检索投毒的防御未做定量检验；仅作为设计控制提出。
- **有害动作率仅为投影**：基于静态依赖覆盖的模拟，并非实际企业环境执行结果，可能低估真实风险。
- **分析师审批模拟**：实验中 AnalystReview 由测试工具模拟，未做真实分析师操作时间与负担的测量。
- **延迟信息不足**：仅报告 M4 堆栈平均 40s/事件（批量分级），未给出其他组合 latency 对比，也未涉及实时性要求高的场景。
- **未涉及多加速器扩展**：更大规模的 Validator 池需额外 GPU，文中未给出硬件成本或扩展实验。
