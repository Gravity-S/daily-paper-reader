## 摘要
代理浏览器将自主AI代理集成到Web浏览器中，使用户能够通过自然语言指令完成Web任务。同源策略（SOP）是一种基本的浏览器安全机制，可防止由脚本引发的未经授权的跨源数据自动流动。然而，SOP在代理浏览器中是否仍然有效，是一个尚未被系统研究的开放问题。在这项工作中，我们填补了这一空白。我们首先观察到代理浏览器本身可以作为跨源数据流动的自动化渠道，可能导致SOP违规。为了研究这一现象，我们构建了SOPBench，一个用于评估代理浏览器中SOP违规的基准测试。我们的评估表明，现有的代理浏览器在良性设置和攻击下都频繁违反SOP。为了解决这个问题，我们提出了SOPGuard，一种针对代理浏览器定制的SOP强制机制。我们在开源代理浏览器BrowserOS中实现了SOPGuard。大量评估表明，SOPGuard在保持实用性和仅产生较小运行时开销的同时，有效地强制执行了SOP。我们的代码和数据可在https://github.com/wxl-lxw/BrowserOS-SOPGuard获取。

## Abstract
Agentic browsers integrate autonomous AI agents into web browsers, enabling users to accomplish web tasks through natural-language instructions. The same-origin policy (SOP) is a fundamental browser security mechanism that prevents unauthorized automated cross-origin data flows induced by scripts. However, whether SOP remains effective in agentic browsers is an open question that has not been systematically studied. In this work, we bridge this gap. We first observe that an agentic browser can itself serve as an automated channel for cross-origin data flows, potentially leading to SOP violations. To investigate this phenomenon, we construct SOPBench, a benchmark for evaluating SOP violations in agentic browsers. Our evaluation shows that existing agentic browsers frequently violate SOP, both in benign settings and under attacks. To address this problem, we propose SOPGuard, an SOP enforcement mechanism tailored to agentic browsers. We implement SOPGuard in BrowserOS, an open-source agentic browser. Extensive evaluations demonstrate that SOPGuard effectively enforces SOP while preserving utility and incurring only a small runtime overhead. Our code and data are available at https://github.com/wxl-lxw/BrowserOS-SOPGuard.

- **问题**：代理浏览器（如BrowserOS、Perplexity Comet、ChatGPT Atlas）集成AI代理后，可以自动执行跨源数据流动——代理从源网页读取敏感信息，保留在交互历史中，随后写入不同源的接收网页。传统浏览器中的同源策略（SOP）仅针对脚本引发的自动数据流，但代理本身构成了新的自动通道，可能导致SOP被绕过。
- **背景**：先前工作主要关注同源内的提示注入攻击，缺乏对跨源数据流的系统研究。部分博客和研讨会论文提到代理浏览器可能绕过SOP，但未提供基准或防御方案。本文首次系统研究并量化该问题。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将同源策略从“脚本级别”扩展到“代理行为级别”，通过标签传播和用户确认机制，将代理自动触发的跨源数据流转化为用户手动授权的数据流。
- **关键技术细节**：
    - **SOPBench基准**：构建包含10个源网页类别（电商、银行、航空等）和5个接收网页类别（X、Discord、GitHub等）的合成+少量真实网页。每个组合生成100个测试用例，评估被动和主动SOP违规。
    - **被动SOP违规**：在接收网页中嵌入源网页iframe，用户要求总结页面，代理自动将iframe中的源数据写入接收网页。
    - **主动SOP违规**：在接收网页中注入恶意内容（启发式或优化式提示注入），诱导代理将历史中的源数据写入接收网页。
    - **SOPGuard机制**：
        - **标签数据库**：记录每个数据对象的唯一ID、内容、来源标签集合和依赖集合。
        - **标签分配**：代理调用读取工具时，将当前网页的源标签分配给返回数据。
        - **标签传播**：代理生成中间结果时，通过询问代理自身识别数据依赖关系，继承标签。
        - **检测**：代理调用写入工具时，比较待写入数据的标签与接收网页的源标签，若不一致则标记为潜在SOP违规。
        - **用户确认**：弹出窗口通知用户，由用户决定是否批准写入。若批准则转化为手动数据流，不视为SOP违规。
- **实现**：在开源代理浏览器BrowserOS中实现，得到BrowserOS-SOPGuard。

## 6. 主要结论与发现

- **现有代理浏览器频繁违反SOP**：在主动违规下，所有测试浏览器均表现出非零违规率（最高可达100%）；被动违规（iframe场景）下所有浏览器均100%违反SOP。
- **优化攻击比启发式攻击更有效**：优化式（基于TAP）的违规率显著高于启发式（如从0.42提升至0.84）。
- **SOPGuard彻底消除违规**：在SOPBench上，BrowserOS-SOPGuard将违规率降至0.00。
- **实用性与运行时开销可接受**：与原生BrowserOS相比，实用分数无统计学显著差异（p>0.05），运行时开销仅2.07%–5.79%。
- **标签传播准确性高**：对于GPT-5.4和OpenAI o3，Jaccard系数接近1.0（≥0.955）；GPT-5.4-mini稍低但仍>0.9。
- **现有检测器不可靠**：5种提示注入检测器均无法同时实现低FNR和低FPR，无法防御SOP违规。
- **自适应攻击困难**：仅略微增加SOPGuard的违规率（从0.00到0.04），且需大量查询。

## 7. 优点

- **首次系统研究**：填补了代理浏览器中SOP违反研究的空白。
- **基准全面**：SOPBench覆盖多种源/接收网页类别、攻击方法，包含合成和真实网页，便于复现。
- **解决方案有效且低开销**：SOPGuard通过标签传播+用户确认，在不显著影响用户体验的前提下实现强安全。
- **开源实现**：代码和基准已公开，促进社区验证和后续改进。
- **实验严谨**：采用严格匹配、重复实验、统计假设检验，对比多种基线方法。

## 8. 不足与局限

- **闭源浏览器评估不充分**：仅进行3次手动实验，统计置信度低，可能低估或高估其违规率。
- **被动违规场景单一**：仅测试了iframe嵌入场景，未涵盖其他被动违规模式（如代理无意中结合历史信息写入）。
- **标签传播依赖LLM自身**：较弱的LLM（如GPT-5.4-mini）准确性下降，导致SOPGuard可能产生误报或漏报。
- **自适应攻击潜力未穷尽**：尽管当前自适应攻击效果有限，但更复杂的攻击可能绕过SOPGuard（如操纵中间结果标签传播）。
- **API成本增加**：标签传播调用额外LLM，导致经济开销上升。
- **未涉及真实世界部署挑战**：如用户确认弹窗的可用性、用户误操作风险、分布式或大规模环境下的性能。
