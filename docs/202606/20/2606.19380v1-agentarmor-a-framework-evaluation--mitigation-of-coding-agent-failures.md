## 摘要
软件工程和部署正越来越多地交由AI编码代理完成。其大规模应用正暴露出一些罕见但极具破坏性的故障模式。在本文中，我们将这些故障模式归因于三种不同机制：欠规范（默认模型行为不安全）、能力错误（安全动作可用但模型因偏差或能力限制未遵循）以及代理框架错误（模型未能通过框架执行安全动作）。我们通过8种不同的评估（每种均受真实部署失败启发，总计20个编码环境和59个合成转录模板）对这些机制进行了评估。基于此评估，我们提出了AgentArmor（一种代理框架修改方案）以缓解这些错误。通过增加扩展系统提示、独立的命令分类器、“三振出局”策略、确定性护栏以及让代理编辑自身上下文的工具，我们证明AgentArmor在统计显著数量的样本上更为安全。因此，我们为当前编码代理提出了具体的缓解措施，并为未来代理框架特性提出了一种设计哲学。

## Abstract
Software engineering and deployment are increasingly being delegated to AI coding agents. The scale of their adoption is surfacing rare, but highly destructive, failure modes. In this paper, we study these failure modes as stemming from three distinct mechanisms: underspecification, where default model behavior is unsafe; capability errors, where the safe action is available but the model does not adhere to it due to bias or capability limitations; and agent harness errors, where the model fails to execute the safe action through the harness. We evaluate these across 8 different evaluations, each inspired by real-life deployment failures, totaling 20 coding environments and 59 synthetic transcript templates. Based on this evaluation, we propose AgentArmor, an agent harness modification, to mitigate these errors. By adding an extended system prompt, a separate command classifier, a ``3 strikes'' policy, deterministic guardrails, and tools for the agent to edit its own context, we show that AgentArmor is safer across a statistically significant number of samples. Thus, we suggest concrete mitigations for current coding agents and a design philosophy for future agent harness features.

## 2. 方法论：核心思想与技术细节

- **三类失败机制**：
  - **M1**（欠规范）：任务不明确时，模型默认采取不安全行为。
  - **M2**（能力错误）：规范明确时，模型因偏差、拒绝或能力限制仍无法执行安全动作。
  - **M3**（代理框架错误）：由于随机采样、上下文衰减等原因，模型无法安全执行动作。
  - 总失败概率公式：\( P(\text{unsafe}) = 1 - (1 - f_1)(1 - f_2)(1 - f_3) \)，其中 \( f_i \) 为给定前阶段成功时第 i 阶段失败的概率。
- **AgentArmor框架**（基于OpenCode构建）：
  - **扩展系统提示**：增加默认安全行为指引。
  - **两阶段命令分类器**：
    - 第一级：本地Qwen3.5 0.8B微调模型，快速过滤低风险命令（目标：100%召回高风险命令）。
    - 第二级：云端Haiku 4.5模型，高精度判断命令是否与用户最近10个查询意图一致（阻止高风险且意图不匹配的命令）。
    - 创新点：该分类器仅查看用户输入，忽略助手输出，实现**说服盲（persuasion blindness）**，防止目标漂移。
    - **“三振出局”政策**：若模型连续三次尝试执行被分类器阻止的命令，则强制结束本轮。
  - **确定性护栏**：
    - 执行 `rm -rf` 前强制运行 `ls -la`。
    - 执行脚本前强制读取脚本内容。
  - **代理自编辑工具**：
    - **文件不可变**：通过系统守护进程使模型可标记文件为不可修改/删除，仅用户可撤销。
    - **上下文修剪**：模型可选择性截断大型工具输出（中间部分首选保留），减少上下文膨胀。每50k tokens自动提示。
- **评估机制**：通过设计隔离各机制的场景（欠规范任务留空、能力任务给定明确规范并预填充正确动作）来独立测量 \( f_1, f_2, f_3 \)。

## 3. 实验设计：数据集、基准与对比方法

- **评估场景**（共8个场景，20个编码环境，59个合成转录模板）：

  2. **意外上下文学习**（Accidental In-Context Learning）：10个合成转录，通过3个上下文示例使“清理”等词与危险命令关联，然后询问类似但不同的任务。

  6. **提示校准**（Prompting Calibration）：基于升级任务，模型为每个bash命令设置风险标志（1-5级），评估过度提示和不足提示率。
