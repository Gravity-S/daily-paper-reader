## 摘要
AI代理现在通过工具使用、函数调用和多轮对话处理个人数据，这可能产生《通用数据保护条例》（GDPR）下的义务。当前的测试实践主要依赖离线红队测试或静态提示审查，但无法在运行时保证代理行为遵循监管规则。我们提出C-Trace（基于合规性追踪的运行时代理一致性执行），一个验证框架，该框架：（i）将GDPR要求的子集，包括同意、目的限制、数据最小化和删除权，表示为代理执行轨迹上的形式化策略谓词；（ii）使用运行时监视器拦截每次工具调用和模型输出，并拒绝不合规行为；（iii）使用攻击对话测试代理，包括DSPy生成的提示和来自红队测试语料的逐字提示，试图引发违规行为。我们在四个重新调整为GDPR案例的研究中评估该框架。在每类别10%的提取器噪声（包括丢失和过度键入）下，监视器将攻击成功率保持在12%或以下，低于我们比较的基线，并将误报率保持在16%或以下，在完美提取下达到0%的攻击成功率。

## Abstract
AI agents now handle personal data through tool use, function calls, and multi turn dialogue, which can create obligations under the General Data Protection Regulation (GDPR). Current testing practices mainly rely on offline red teaming or static prompt review, but they do not guarantee at runtime that agent behavior follows regulatory rules. We propose C-Trace (Compliance Trace based Runtime Agent Conformance Enforcement), a verification framework that: (i) expresses a subset of GDPR requirements, including consent, purpose limitation, data minimization, and the right to erasure, as formal policy predicates over agent execution traces; (ii) uses a runtime monitor that intercepts every tool invocation and model output and rejects non-compliant actions; and (iii) tests the agent with attack dialogues, including DSPy generated prompts and verbatim prompts from red teaming corpora, that try to induce violations. We evaluate the framework on four case studies reframed to GDPR. Under 10 percent per-category extractor noise, including drop-out and over-typing, the monitor keeps the attack success rate at less than or equal to 12 percent, below the baselines we compare against, and false positives at less than or equal to 16 percent, and reaches 0 percent ASR under perfect extraction.

- **研究动机**：AI代理（如客户服务助理、医疗分诊前端、财务顾问）通过工具调用、函数调用和多轮对话处理个人数据，这会产生GDPR义务。例如，调用`lookup_customer`时传递邮箱属于个人数据处理（Art. 4(2)）；自由文本回复中重复存储地址需要合法依据（Art. 6）。由于工具选择、参数生成和自然语言披露由模型在运行时控制，同一部署在不同对话中可能合规也可能违规，但代码、提示或模型权重不变。
- **现有方法的不足**：
  - 离线红队测试：只报告分数，不强制运行时执行，不观察部署代理的会话状态。
  - 防护栏库（如Llama Guard、Nemo Guardrails）：仅匹配单个输入/输出模式，不追踪会话历史（如是否在前面的对话中已授予同意）。
- **核心问题**：AI代理的GDPR合规是一个运行时验证问题，需要在事件轨迹上实时检查，并搭配攻击驱动程序测试监视器。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
C-Trace框架将GDPR原则形式化为事件轨迹上的谓词，在代理循环中放置一个拦截器，每步评估谓词，阻止不合规事件，并生成审计日志。框架包含三个部分：
1. **类型化事件模型**：将每次运行记录为带标签的事件序列（UserMsg, AsstMsg, ToolCall, ToolRet, Consent, Erasure），每个事件标注携带的数据类别和目的。
2. **四个谓词（P1–P4）**：编码GDPR的四个原则，作为事件序列上的一阶谓词，一旦前缀中出现违规立即报告。
3. **攻击驱动器和可观察预言机**：用对抗性对话测试受监控的代理，独立于谓词库判断攻击是否成功。

### 关键技术细节

- **事件模型**：轨迹τ = e₁ e₂ ... eₙ，每个事件属于六种类型之一。监控器添加两个注解：C(e)（数据类别）和Π(e)（目的集）。一个目的集允许服务于多个目的。工具调用可服务于多个目的，P3检查数据类别对至少一个目的是否必要。

- **四个谓词**：
  - **P1（处理前需同意）**：∀i, c ∈ C(eᵢ), π ∈ Π(eᵢ): 若eᵢ携带c且π不是合法利益（LegInt），则在索引j<i处必须有Consent(·, π, ⊤)事件。Consent事件仅由应用的同意捕获机制记录，不能从用户自然语言声明推断。
  - **P2（目的限制）**：∀i, π ∈ Π(eᵢ): π ∈ Π_declared。即每个事件的目的必须在策略中已声明。
  - **P3（数据最小化）**：∀i: eᵢ = ToolCall(·) ⇒ C(eᵢ) ⊆ ∪_{π∈Π(eᵢ)} necessary(π)。工具调用携带的数据类别必须对至少一个目的是必要的。
  - **P4（删除权）**：∀i, j: eᵢ = Erasure(·, s) ∧ j > i ⇒ ¬ discloses(eⱼ, s)。擦除事件后，任何后续事件不得披露同一主体。

- **监视器与拦截器**：拦截器位于代理循环和LLM/tool端点之间。对每个事件e，执行：
  1. 运行类别提取器填充C(e)；
  2. 从策略解析Π(e)；
  3. 将e追加到工作轨迹τ；
  4. 评估已启用的谓词子集；
  5. 决策：forward（无违规）、redact（应用重写规则后重检）、block（移除事件）。
  维护两个轨迹：接受轨迹τ（下游谓词和代理循环所见）和审计轨迹τ_a（所有事件加判决记录）。

- **攻击驱动**：每个谓词对应一个攻击族：
  - A1: 声称先前同意但实际未给；
  - A2: 请求重用数据用于未声明目的；
  - A3: 请求非必要数据字段；
  - A4: 擦除事件后请求回忆主体。
  两个提示集：Generated（由DSPy程序生成，包括改写、角色扮演、紧迫/权威框架、越狱前缀等变体）和Real-corpus（从HarmBench、AdvBench、DAN等语料中拿到的逐字提示，第一轮是原始对抗提示，第二轮指定工具名）。

- **可观察预言机**：每个家族有一个固定判决过程，只读取结果事件轨迹和静态策略，判断家族的违禁行为是否出现在防御允许通过的轨迹中。预言机从不调用运行时谓词，确保判决独立。

- **交叉验证**：同一逻辑策略在三个运行时实现（Python、Rego/OPA、MFOTL/MonPoly）并验证一致性。对所有测试轨迹，三者的每时点违规集完全一致。

## 6. 论文的主要结论与发现

1. **无监控时代理极易违规**：同意绕过（A1）和过度收集（A3）最容易触发（最高达100%和70–100%）；目的洗白（A2）和擦除后回忆（A4）主要在真实语料寄存器中出现较多。
2. **无状态防护栏的局限**：Regex和Presidio可以捕获明显的敏感字段违规（降低A3的ASR），但对依赖会话历史的同意、目的限制、擦除几乎无效。
3. **C-Trace在完美提取下ASR和FPR均为0%**：所有已实现违规都被阻止，良性对话无误报。
4. **在10%提取器噪声下，ASR ≤ 12%，FPR ≤ 16%**，远低于无状态基线。噪声增大后性能下降，但P2不受影响（因为它依赖目的字段，不被扰动）。
5. **消融实验证明每个谓词都必要**：去掉P1后A1恢复至68–100%；去掉P3后A3恢复至41–90%；去掉P4后A4恢复至28–47%；去掉P2影响较小（因为P1通常先报告），但P2仍然是P1漏报时的边际防护。
6. **谓词评估计算开销极低**：中位1.8微秒/事件，可忽略不计。
7. **三类运行时实现完全一致**（Python、Rego、MFOTL），确保策略声明的可信任。

## 7. 优点

### 方法上的亮点
- **首个将GDPR运行时验证应用于AI代理自然语言执行轨迹的工作**：不是单纯检查单条消息，而是建立会话状态感知的轨迹属性。
- **事件模型设计**：将自然语言和自由参数映射到类型化事件，配合类别和目的注释，使一阶谓词可以直接使用。
- **拦截器+审计日志**：不仅阻止违规，还提供可追溯的审计记录，满足合规审查需求。
- **策略可配置**：合法利益例外、数据最小化白名单、工具-目的绑定均可由策略文件定义，无需修改监视器代码。
- **攻击驱动的验证**：使用生成和真实语料两种对抗性输入，测试监视器在对抗场景下的表现，而非仅靠陈述性论证。
- **独立预言机**：避免监视器自我验证的问题；还引入独立GDPR法官交叉验证，增强结论可靠性。

### 实验上的亮点
- **覆盖多种合法依据**（同意、明确同意、法律义务）和多个领域（零售、航空、医疗、金融）。
- **消融实验**验证每个谓词的个体贡献，而非仅报告整体ASR。
- **灵敏度分析**量化了提取器噪声影响，指出关键失败模式，对部署有指导意义。
- **三类运行时交叉验证**确保了策略实现的一致性。

## 8. 不足与局限

### 实验覆盖
- **通用性有限**：四个案例覆盖三种合法依据，但仅使用一个模型（gpt-4o-mini），一个后端。绝对ASR、实现率、FPR数字不能泛化到其他模型或工作流。更强的或更弱的对齐模型会改变这些数字，但0% ASR/FPR稳定。
- **攻击工作量规模有限**：每个（家族，寄存器）仅50个样本，虽然涵盖多种变体，但统计效力有限。
- **良性边缘场景**：只测试了用户在合规方式下提供敏感信息的场景，但未覆盖其他边缘情况如提取器对多语言/混合语言/混淆输入的鲁棒性。

### 方法论局限
- **谓词仅捕获GDPR的一种合理解读**：作者承认这是他们合理的解读而非唯一解读，其他部署可能改变谓词。策略是可配置的，但默认解释的合理性需要共识。
- **提取器是薄弱环节**：实验使用正则表达式的替代品模拟Presidio，但只测试了独立逐类别丢失或过度键入，未测试真实抽取器的所有失败模式（如语言漂移、跨度错误、上下文依赖）。作者保持四个预期失败测试来保持可见性。
- **无法处理组织要求**：如数据处理影响评估、违规通知、处理记录等不在会话内监控范围内。
- **无状态的基线选择**：Random基线q=0.5是人为设定，并非最优比较；但意图是展示随机拒绝无法替代有状态监控。
- **攻击驱动只测试了固定攻击池**：未模拟自适应对手（根据监视器响应调整提示），这被标注为未来工作。

### 偏差风险
- **独立预言机一致性率仅88%**：主要是同意上的歧义（71%一致），表明GDPR条款在具体操作化上存在空间，可能影响结果解释。
- **监控器依赖精确的类别提取**：提取器在真实部署中可能比模拟的丢失更多模式，导致ASR高于报告的12%。

### 应用限制
- 仅适用于可以收集工具调用和模型输出的架构；对于纯文本式对话（无工具调用）可能不适用。
- 需要应用层提供明确的同意捕获机制和擦除后端；如果应用本身不记录Consent/Erasure事件，监控器无法工作。
- 增加少量延迟（微秒级），但在极高并发会话下可能累积；论文未测试大规模并发的压力。
