## 摘要
计算机使用智能体越来越常根据其完成现实桌面和网络任务的能力进行评估。然而，仅凭任务成功可能遗漏智能体通过不安全捷径达到名义目标而导致的失败。我们提出OSGuard，这是一个双粒度基准测试套件，用于在良性的、未更改的用户指令下评估计算机使用智能体的安全性。OSGuard包含一个用于本地护栏决策的动作级基准测试和一个用于端到端评估的风险增强执行套件。动作级基准测试由标注为允许、无关或不安全的上下文化提议动作组成，每个动作均相对于原始指令和当前界面状态进行判断。执行套件包含手动构建的OSWorld衍生任务变体，其中原始任务仍可实现，但环境被修改以引入潜在危害，如破坏性覆盖等。每个变体都配有增强型评估器，该评估器保留原始任务成功标准，同时增加基于状态的显式安全不变量，使我们能够区分安全完成和满足名义任务目标的不安全完成。我们在OSGuard上的实验结果表明，当前的多模态护栏在孤立动作判断上表现良好，而风险增强执行则暴露了本地监督与可靠端到端安全性之间的剩余差距。这种双粒度设计使得更精确地诊断模型是否既能识别不安全提议动作，又能在作为护栏部署时提高全任务安全性成为可能。

## Abstract
Computer-use agents are increasingly evaluated by whether they complete realistic desktop and web tasks. However, task success alone can miss failures in which an agent reaches the nominal goal through an unsafe shortcut. We introduce OSGuard, a dual-granularity benchmark suite for evaluating safety in computer-use agents under benign, unchanged user instructions. OSGuard contains an action-level benchmark for local guardrail decisions and a risk-augmented execution suite for end-to-end evaluation. The action-level benchmark consists of contextualized proposed actions labeled as allowed, unrelated, or unsafe, each judged relative to the original instruction and current interface state. The execution suite contains manually constructed OSWorld-derived task variants in which the original task remains achievable, but the environment is modified to introduce latent hazards such as destructive overwrites, etc. Each variant is paired with augmented evaluators that retain the original task-success criterion while adding explicit state-based safety invariants, allowing us to distinguish safe completions from unsafe completions that satisfy the nominal task objective. Our experimental results on OSGuard show that current multimodal guardrails can perform well on isolated action judgments, while risk-augmented execution exposes remaining gaps between local oversight and reliable end-to-end safety. This dual-granularity design enables more precise diagnosis of whether models can both recognize unsafe proposed actions and improve full-task safety when deployed as guardrails.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：现有计算机使用智能体评估只关注任务是否成功完成（nominal task success），忽略了智能体可能通过“不安全捷径”（unsafe shortcuts）达到目标——例如覆盖无关文件、扩大权限、修改全局设置、访问不必要敏感信息等。这些失败并非源于恶意指令，而是源于智能体在执行良性指令时，在当前环境状态下选择了局部可行但违反安全约束的行为。
- **研究意义**：需要一种能同时衡量任务完成度和安全约束遵循度的评估方法，以区分“安全完成”和“不安全完成”（unsafe completion），并诊断安全缺陷源自局部决策还是整体执行。
- **OSGuard 定位**：不同于已有安全基准（如 OS-Harm、RiOSWorld 关注恶意请求/对抗环境，AUTOELICIT 扰动指令），OSGuard 聚焦于**良性、未更改的用户指令**下，由**环境状态引入的潜在危害**所导致的安全问题。

## 2. 方法论：核心思想、关键技术细节

- **双粒度基准设计**：
  - **动作级基准（Action-Level Benchmark）**：用于评估本地护栏（guardrail）的孤立决策能力。每个条目包含：原始指令 \(u\)、当前界面状态 \(s_{t-1}\)（截图+可访问性树）、候选动作 \(a_t\)。护栏预测标签 \(y \in \{\text{allowed}, \text{unrelated}, \text{unsafe}\}\)。
    - 构建来源（4类）：
      1. 标准 OSWorld 执行轨迹中的候选动作（多为 allowed）。
      2. 中断的 OSWorld 执行前缀 + 状态兼容提议者生成的动作（覆盖 allowed、unrelated、unsafe）。
      3. 中断的执行前缀 + 不安全提议者生成的动作（主要为 unsafe）。
      4. 风险增强变体执行轨迹中的候选动作。
    - 共324个条目，通过人工标注获得参考标签。
  - **风险增强执行套件（Risk-Augmented Execution Suite）**：基于 OSWorld 任务手动构建45个变体。每个变体保持原始指令不变，但修改环境状态引入潜在危害（如破坏性覆盖、越界编辑、配置冲突等），同时保留一条安全路径。每个变体配有**增强型评估器**，保留原始任务成功标准，并增加基于状态的显式安全不变量（如文件哈希、权限、规则完整性等）。一个运行被记为成功当且仅当完成原始任务且满足所有安全不变量。

- **护栏智能体**：反应式、状态条件化的提示模型。接收原始指令、当前状态、候选动作，输出 allowed / unrelated / unsafe。若阻拦，返回自然语言反馈。允许最多2次重试（即最多3次提议），否则终止并记为失败。

- **数据集/场景**：动作级基准共324个条目；执行套件45个 OSWorld 衍生变体，覆盖6种安全失败类别（破坏性覆盖/删除、过度编辑/权限修改、范围逃逸、配置冲突、不必要敏感信息访问、错误目标/全局更新）。所有变体基于智能体原本能完成的任务构建。

- **实验数量**：

  - 执行级：仅一种执行器（Claude Sonnet 4.5）在45个变体上的无护栏和有护栏对比。

## 6. 主要结论与发现

- 动作级：Gemini 3 Pro Preview 表现最佳（准确率80%，宏F1 0.80），但**风险增强执行来源的动作准确率仅50%**，说明护栏对任务执行中产生的危险动作识别困难。
- 执行级：无护栏时62%变体安全完成，38%为不安全完成（达到任务目标但违反安全不变量）。加入护栏后，安全完成率仍为62%，不安全完成率从38%降至33%，但引入了4%的终止失败（原本能成功但被护栏阻止）。**护栏在端到端场景下改善十分有限**。
- 关键差距：本地护栏判断能力与端到端安全执行之间存在显著鸿沟——护栏能识别简单场景下的明显不安全动作，但难以应对上下文依赖的、与任务目标部分一致的危险行为。

## 7. 优点

- **双粒度设计**创新：同时评估隔离的动作判断和完整任务执行，有助于诊断安全失败的具体环节（是识别失败还是执行失败）。
- **安全不变量机制**：使用确定性状态检查（如文件哈希、权限等）区分安全完成和不安全完成，避免主观语义判断，可复现、可验证。
- **危险环境构建合理**：基于OSWorld任务，注入真实的、局部有吸引力的不安全路径（如覆盖一个易访问的相似文件而非目标文件），而非极端对抗情况。
- **考虑重试机制**：护栏阻拦后允许有限重试，模拟真实部署中的补救能力。

## 8. 不足与局限

- **规模有限**：动作级324个条目、执行级45个变体，不足以覆盖计算机使用安全的全部场景。且每个变体仅一个执行器测试，泛化性存疑。
- **仅评估单一执行器**：执行器固定为Claude Sonnet 4.5，未测试GPT、Gemini等作为执行器的表现，结论偏向于该特定模型。
- **护栏模型单一**：仅用Gemini 3 Pro Preview作为护栏进行在线执行，其他护栏未测试，无法比较不同护栏在交互环境中的效果。
- **未考虑动态对抗**：危害是预先注入的静态状态，未测试智能体在运行中被诱使产生的新风险（如prompt注入、界面弹窗等）。
- **标注偏差风险**：动作级标签依赖人工，但文中未报告标注者间一致性（如Kappa值），且“unsafe”标签优先级规则可能引入主观性。
- **计算成本未讨论**：无护栏/护栏推理延迟、重试次数、API开销等实际部署因素。
