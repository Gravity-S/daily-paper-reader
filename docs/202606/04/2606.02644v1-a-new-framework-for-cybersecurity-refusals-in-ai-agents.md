## 摘要
代理脚手架显著提升了大型语言模型在复杂、长期任务上的表现，在网络安全等领域既带来了广泛益处也放大了风险。现有的AI代理网络安全基准主要关注能力测量——即代理如何有效完成进攻性安全任务——但忽略了一个关键问题：代理何时以及如何拒绝有害请求？我们提出了首个在进攻性安全场景中建立拒绝边界的框架。该框架定义了（1）任务应被拒绝的原则性标准，（2）应被拒绝的任务类别，以及（3）在良性及对抗性条件下衡量代理鲁棒性的评估方法。我们将该框架应用于评估当前基于LLM的代理在一系列基于网络的进攻性安全场景中遵循适当拒绝边界的情况，发现测试的8个前沿模型中有6个的拒绝率接近零，只有2个模型（GPT-5.2和GPT-5.1 Codex）表现出任何有意义的拒绝行为。

## Abstract
Agentic scaffolds have dramatically improved LLM performance on complex, long-horizon tasks, yielding both broad benefits and amplified risks in domains like cybersecurity. Existing benchmarks for AI agents in cybersecurity focus mainly on measuring proficiency--how effectively agents can complete offensive security tasks--but neglect a critical question: when and how should agents refuse harmful requests? We present the first framework for establishing refusal boundaries in offensive security contexts. Our framework defines (1) principled criteria for when tasks should be refused, (2) categories of tasks that warrant refusal, and (3) evaluation methodology for measuring agent robustness under both benign and adversarial conditions. We apply this framework to assess how current LLM-powered agents adhere to appropriate refusal boundaries across a range of web-based offensive security scenarios, finding that 6 of 8 frontier models tested show near-zero refusal rates, with only 2 models (GPT-5.2 and GPT-5.1 Codex) demonstrating any meaningful refusal behavior.

### 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：现有的AI Agent网络安全基准（如Cybench、CyberGym）主要衡量其**能力**（proficiency），即完成进攻性安全任务（如渗透测试、漏洞利用）的有效性。然而，这些基准忽略了一个更关键的**安全对齐**问题：当面对有害请求时，Agent **何时以及如何应该拒绝**？
- **研究动机**：网络安全工具具有**双重用途**，同样的技术对防御方（红队测试）和攻击方（黑客）都有价值。现有的大语言模型（LLM）依赖**用户中心**的拒绝机制（主要基于用户请求的表面内容，如URL字符串），这在Agent场景下存在严重缺陷：
    - **虚假拒绝（Overrefusal）**：用户请求一个看似危险的URL（如`www.wikipedia.org`），但实际上该域名已通过本地DNS解析到本地测试环境，Agent会错误地拒绝。
    - **虚假服从（False Compliance）**：用户请求一个看似无害的URL（如`localhost:5001`），但实际上该端口已被转发到真实的攻击目标，Agent会错误地服从。
- **整体含义**：当前AI Agent在网络安全领域的拒绝机制极其薄弱，容易被绕过，无法阻止恶意使用，也不能满足合法安全工作的需要。该研究提出了一个全新的评估鲁棒性（Robustness）的视角和框架，弥补了现有关键安全差距。

### 2. 方法论：核心思想、关键技术细节

论文提出了**网络安全拒绝框架（Cybersecurity Refusal Framework, CRF）**，包含三个核心贡献：

- **核心贡献1：环境感知拒绝机制（Environmentally Aware Refusals）**
    - **核心思想**：拒绝机制不应仅基于用户请求的文字表面，而应基于Agent对**目标环境的实际验证**。
    - **关键技术细节**：Agent在决定拒绝与否前，应主动执行以下步骤：
        1.  **枚举目标表面**：如获取网站的HTTP头部、检查TLS证书、分析页面内容。
        2.  **确定目标基础设施性质**：判断目标是否为本地测试服务器、真实生产环境或关键基础设施。
        3.  **综合决策**：基于已验证的环境上下文和用户意图，而非单纯依赖URL字符串做出拒绝决策。

- **核心贡献2：三元拒绝分类法（Tripartite Refusal Taxonomy）**
    - **核心思想**：将目标系统分为三类，明确规定拒绝边界，平衡安全与实用性。
    - **关键技术细节**（三类目标）：
        - **红色（始终拒绝）**：对任何未经授权的测试或攻击请求都严格拒绝。包括四个初始领域：政府、医疗、电网和交通系统。
        - **绿色（始终允许）**：明确的人造或玩具场景，如CTF（夺旗赛）的Web服务器或本地虚拟靶机。
        - **黄色（视上下文而定）**：介于红绿之间的所有目标。多数合法的安全测试工作属于此类，决策需依赖于上下文验证。

- **核心贡献3：网络安全拒绝基准（CRF Benchmark）**
    - **核心思想**：构建一个同时衡量Agent**能力**和**鲁棒性**的基准测试。
    - **关键技术细节**：基准包含29个挑战，涵盖9种常见Web漏洞（如XSS、SQL注入、SSRF等），并在三个难度级别（Easy, Medium, Hard）下进行测试。
        - **Easy模式**：使用localhost上的简单虚拟服务器测试基础能力。
        - **Medium模式**：通过Nginx代理，将漏洞页面注入到真实网站的副本中，以测试Agent能否在更真实的环境下识别并拒绝。
        - **Hard模式**：直接针对真实生产网站的现有功能，要求Agent制定漏洞评估计划，测试其在真实、高价值目标前的拒绝行为。
    - **评估指标**：对于Easy模式，使用“**效用分数**（Utility Score）”\(U = P \times (1-R)\)，其中\(P\)是**能力分数**（成功利用漏洞的比例），\(R\)是**拒绝率**。对于Medium和Hard模式，额外引入**情境感知**（Awareness）指标，评判Agent在进行拒绝决策前是否对目标环境进行了侦察。

### 6. 主要结论与发现

- **主要结论**：当前AI Agent在网络安全领域的**安全对齐严重不足**。
- **关键发现**：

    3.  **简单方法即可绕过**：研究发现两种极其简单的通用方法来绕过拒绝机制：

        - **使用HTTP本地主机地址代理真实网站**：将攻击目标隐藏在`localhost`背后，大多数模型会被欺骗。

- **实用性强**：提出的“环境感知”拒绝机制和“三元分类法”为AI安全和产品设计提供了非常务实的路线图，平衡了安全性与实际应用需求。

### 8. 不足与局限

- **范围有限**：研究范围限制于Web漏洞，其他关键网络安全领域（如网络渗透、社会工程学）未被覆盖。高风险目标（如内部系统）由于评估的可行性问题未被涉及。
