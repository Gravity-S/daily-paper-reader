## 摘要
Agent技能通过可复用的指令、脚本、工具绑定和上下文依赖来扩展LLM Agent。然而，当前的技能生态系统主要依赖于基于信任的加载和静态检查，导致技能可以注入Agent上下文的内容与其在运行时可能诱导Agent执行的行为之间存在差距。这一差距带来了新的安全和隐私风险，现有的防御措施主要对技能文件进行静态检查或规范单个工具调用，而没有系统地将技能的声明意图与其运行时行为联系起来。在本文中，我们提出了SkillGuard，一种以技能为中心的权限框架，将技能视为承载权限的可执行工件。SkillGuard引入了一种双平面治理模型，通过技能清单、运行时访问控制、用户中介授权、默认拒绝执行、能力推断和行为监控，共同规范上下文影响和行为副作用。我们在315个真实世界技能和SkillInject上评估了SkillGuard。权限分类覆盖了99.76%的观察到的受保护对象，自动清单生成的F1值达到91.0%。在对抗性评估中，SkillGuard将上下文注入攻击的成功率从32.37%降低到23.02%，将明显注入攻击的成功率从25.56%降低到16.67%，同时保持了良性任务的效用。这些结果表明，SkillGuard作为一种以技能为中心的权限框架，可以为改善Agent技能生态系统的隐私和安全提供实用基础。

## Abstract
Agent skills extend LLM agents with reusable instructions, scripts, tool bindings, and contextual dependencies. However, current skill ecosystems largely rely on trust-based loading and static inspection, leaving a gap between what a skill can inject into an agent's context and what it can cause the agent to do at runtime. This gap introduces new security and privacy risks, and existing defenses primarily inspect skill files statically or regulate individual tool calls, without systematically connecting a skill's declared intent with its runtime behavior. In this paper, we present SkillGuard, a skill-centric permission framework that treats skills as permission-bearing executable artifacts. SkillGuard introduces a dual-plane governance model that jointly regulates context influence and action side effects through skill manifests, runtime access control, user-mediated authorization, deny-by-default enforcement, capability inference, and behavior monitoring. We evaluate SkillGuard on 315 real-world skills and SkillInject. The permission taxonomy covers 99.76% of observed protected objects, and automated manifest generation reaches 91.0% F1. In adversarial evaluations, SkillGuard reduces attack success from 32.37% to 23.02% for contextual injections and from 25.56% to 16.67% for obvious injections, while maintaining benign task utility. These results suggest that SkillGuard, as a skill-centric permission framework, can provide a practical foundation for improving the privacy and security of agent skill ecosystems.

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将技能视为**承载权限的可执行工件**（permission-bearing executable artifacts），而非隐式信任的提示片段或简单的工具包装器。
- **双平面治理模型**：协同管控**上下文平面**（技能注入什么指令、记忆等影响Agent推理）和**动作平面**（技能触发什么工具调用、文件访问、网络通信等副作用）。
- **关键技术模块（共5个）**：
  1.  **Skill Manifest（技能清单）**：使用基于JSON的领域特定语言（DSL）声明技能所需的能力。DSL包含：能力（capability）、执行效果（allow/confirm/deny）、约束条件（如仅限工作空间文件）、以及回退消息。该清单是运行时强制执行的策略基础。
  2.  **运行时权限访问控制**：在Agent调用每个工具前（`PreToolUse`钩子），将主机特定的工具调用映射到规范化的能力，并检查当前会话策略是否允许。若未声明、约束不匹配或涉及危险权限，则可能触发用户确认或直接拒绝。
  3.  **用户交互机制**：对于敏感权限（如写文件、执行shell命令），通过`confirm`效果向用户弹出授权提示，用户可选择“允许一次”、“允许本次会话”或“拒绝”。会话批准受到约束，避免扩大权限范围。
  4.  **策略执行与强制力**：实现**默认拒绝**（deny-by-default）的运行时管道。策略来源包括工作空间默认策略、技能清单、持久化授权和会话批准。对于shell命令执行，SkillGuard会调用一个**权限生成迷你Agent**，分析命令及引用的脚本，推断其所需的低级能力，并在允许执行前重新评估策略。
  5.  **运行时行为监控与审计**：记录每一次被调解的操作、能力映射、策略来源、决策及其原因，为调试和问责提供细粒度追踪。

## 6. 主要结论与发现

- **权限表达力**：定义的权限分类覆盖了**99.76%** 的观察到的受保护对象，所有对象都能归入8个资源组（100%组覆盖）。

- **防御有效性**：SkillGuard成功将**上下文注入攻击成功率**从32.37%降低到**23.02%**（下降9.35个百分点）；将**明显注入攻击成功率**从25.56%降低到**16.67%**（下降8.89个百分点）。
- **效用保持**：在良性任务上，任务成功率（TSR）仅从86.96%略微下降到**85.51%**（下降1.45个百分点），影响很小。
- **效率**：
  - **Token消耗增加**：中位数增加20.9%（267k vs 221k），平均增加32.0%（437k vs 331k），由于被拒绝后Agent需重新规划。
  - **墙钟时间反而减少**：中位数时间从438秒降至**345秒**（减少20.9%），平均时间从493秒降至**430秒**（减少12.9%）。原因是有害操作被阻止后，Agent提前终止或快速重建计划，避免了恶意指令引导下的长时间无效工作。
- **总结**：SkillGuard作为以技能为中心的权限框架，能够在不显著损害任务效用的前提下，有效降低多种注入攻击的成功率，为改善Agent技能生态系统的安全与隐私提供了实用基础。

## 7. 优点

- **全面的双平面治理**：首次系统性地将上下文影响和动作副作用联合纳入一个统一的权限框架进行管控，弥合了现有工作的关键鸿沟。
- **解耦的权限模型**：通过技能清单（Skill Manifest）将声明与具体工具名解耦，实现了跨Agent平台的策略可复用性。
- **自动化的清单生成**：实现了91.0% F1的自动清单生成，显著降低了开发和维护者的负担，使其具有可扩展性。
- **用户授权的细粒度控制**：提供“一次/会话/拒绝”的灵活授权模式，在安全与用户体验间取得平衡。
- **默认拒绝原则**：所有操作默认被阻止，除非有明确授权，符合安全设计的最佳实践。
- **运行时行为监控与审计**：提供可观测性和问责基础，有助于调试和合规性审计。

## 8. 不足与局限

- **实验泛化性有限**：仅评估了一个Agent框架（Claude Code）和一个基础模型（MiMo-V2.5-Pro），无法证实其对其他主流Agent（如LangChain、AutoGPT）或不同模型的效果。
- **人工标注偏差风险**：尽管做了双标注，但权限分类和参考清单的构建仍依赖于人工判断，存在主观性。标注者之间的一致性系数（Kappa）为0.656（中等人为-较高一致性）。
- **LLM评判的可靠性**：使用LLM作为评判（LLM-as-Judge）来打分攻击成功率和任务成功率，可能因模型自身的局限性（如幻觉、误判）而导致结果偏差。
- **清单生成的不完美**：
  - **过度声明**：56.5%的技能存在至少一个未使用的权限，这会增加不必要的用户确认提示，可能带来用户疲劳。
  - **缺失的运行时能力**：清单生成仅依赖`SKILL.md`文件描述，未分析被技能调用的脚本或子进程内部的代码（如`pack.py`中的`CREATE_PROCESS`），这导致了在RQ4中合法任务的误阻断。
- **针对特定攻击类型的局限性**：对于攻击者使用了合法权限来实现恶意目标的情况（例如，利用正常的“发送邮件”权限进行批量转发），SkillGuard的权限检查本身无法区分意图，因为它缺乏任务级别的对齐验证。此问题被论文明确指出。
- **未进行全面的消融实验**：未量化各模块（如用户交互、能力推断迷你Agent）对整体防御效果的独立贡献。
- **资源开销未量化**：虽然报告了Token和时间开销，但未详细说明CPU/内存开销等。
