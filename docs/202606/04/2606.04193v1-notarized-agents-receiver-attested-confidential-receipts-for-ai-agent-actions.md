## 摘要
当前AI代理的可观测性在结构上存在缺陷：生成活动日志的实体与被记录活动的实体是同一个。被攻破或有缺陷的代理可能会省略、修改或伪造其自身的追踪记录，而运行该代理的操作者没有独立的方法来检测篡改。我们提出了一类协议，通过反转信任边界来解决这一问题：接收代理调用的服务使用自己的密钥对观察到的情况签名，将收据加密给代理的所有者，并将其发布到公共透明度日志中。所有者无需信任代理或其操作者即可重建防篡改的追踪记录。我们将该类协议实例化为Sello，它结合了当前任何系统中都不存在的四个属性：(P1) 接收方签名，(P2) 使用HPKE加密到所有者公钥，并通过JWS绑定到授权令牌，(P3) 发布到见证人联署的Merkle日志，以及(P4) 所有者通过令牌引用进行发现。我们描述了该协议，分析了在攻击者控制代理及其操作者情况下的安全性，提供了密码操作的微基准测试，并将Sello与相邻的收据协议工作（Signet、AgentROA、Agent Passport System、draft-farley-acta、SCITT）进行了比较。我们讨论了已知的限制，包括压制攻击、服务共谋以及采用激励问题。

## Abstract
Current AI agent observability is structurally compromised: the entity producing the activity log is the same entity whose activity is being logged. A compromised or buggy agent can omit, alter, or fabricate its own traces, and the operator running the agent has no independent way to detect tampering. We propose a class of protocols that resolves this by inverting the trust boundary: the service that receives an agent's call signs a receipt of what it observed using its own key, encrypts the receipt to the agent's owner, and publishes it to a public transparency log. The owner reconstructs a tamper-evident trail without trusting the agent or its operator. We instantiate the class as Sello, a protocol combining four properties absent in any current system: (P1) receiver-side signing, (P2) HPKE encryption to an owner public key bound to the authorization token via JWS, (P3) publication to a witness-cosigned Merkle log, and (P4) owner-side discovery by token reference. We describe the protocol, analyze its security under an adversary that controls the agent and its operator, present microbenchmarks of the cryptographic operations, and situate Sello among adjacent receipt-protocol work (Signet, AgentROA, Agent Passport System, draft-farley-acta, SCITT). We discuss known limitations including the suppression attack, service collusion, and the adoption-incentive problem.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：当前 AI 代理的可观测性存在结构性缺陷——**日志生成者与行为执行者是同一实体**。被攻破或存在 bug 的代理可以随意篡改、伪造或删除自身行为记录，而运行代理的操作者无法独立检测此类篡改。
- **研究背景**：现有主流工具（Langfuse、LangSmith 等）都依赖代理自身或代理运行环境采集追踪信息，本质上信任代理会诚实记录。然而，在金融交易、医疗数据访问、代码修改等高风险场景中，这种信任是不合理的。
- **整体含义**：论文借鉴了人类社会的古老实践（如美索不达米亚泥板合约的第三方见证）以及 Apple Find My（独立 iPhone 充当见证者）的架构思路，提出**将信任边界反转**：由接收代理调用的服务（而非代理自身）使用自己的密钥对交互行为进行签名、加密并发布到公开透明日志，使得所有者可以独立验证代理的真实行为。

## 2. 论文提出的方法论：核心思想、关键技术细节、算法流程

### 核心思想
- **信任边界反转**：将“记录行为”的职责从代理（及其操作者）转移到**接收服务**，因为这些服务天然是独立于代理的存在，且直接观察到了代理的每一次调用。

### 四个设计属性（P1–P4）
1. **P1 接收方签名**：签名密钥由被调用的服务持有，而非代理或操作者。
2. **P2 所有者加密**：收据内容使用 HPKE 加密到代理所有者的公钥，密文可安全地存放在公开基础设施上。
3. **P3 公开透明日志**：加密收据发布到带见证人联署的仅追加 Merkle 树日志，提供防篡改性和全局可验证性。
4. **P4 所有者侧发现**：所有者通过授权令牌的哈希引用查询日志，本地解密并验证签名，无需信任任何中间方。

### 协议具体流程（Sello 实例）
1. **令牌绑定**：代理的授权令牌（JWS）中携带“owner_hpke_pk”声明的公钥，该公钥通过 JWS 签名与令牌绑定，防止替换攻击。
2. **服务端操作**：
   - 代理调用服务 → 服务验证 JWS → 执行动作。
   - 服务构造收据体（CBOR 格式，包含 action-type、input-hash、output-hash、timestamp 等）。
   - 使用 HPKE（X25519 + HKDF-SHA256 + ChaCha20-Poly1305）将收据体加密到所有者公钥。
   - 使用服务自己的 Ed25519 私钥对加密信封进行 COSE_Sign1 签名。
   - 将签名信封提交到透明日志（如 Sigstore Rekor），获取包含证明。
3. **所有者端验证**：
   - 计算令牌哈希 sello_token_ref。
   - 查询日志获得匹配条目。
   - 验证 Merkle 包含证明 → 查询服务身份注册表获得公钥 → 验证 COSE_Sign1 签名 → 使用所有者私钥解密 HPKE 密文 → 检查收据体。

## 6. 论文的主要结论与发现

- **Sello 可实现接收方认证的机密收据，且性能开销很小**：服务端每收据增加约 0.45 ms CPU 时间，所有者端验证约 0.28 ms。批处理验证可进一步提高吞吐量。
- **唯一同时满足 P1–P4 四个属性的协议**：与现有七个系统对比，Sello 在接收方签名、所有者加密、公开透明日志和所有者发现方面实现了完整组合。
- **安全性分析表明**：即使攻击者完全控制代理和操作者，也无法伪造合法服务签名的收据；收据内容对被动攻击者保密；日志的 Merkle 树和见证人联署保障已收录条目的防篡改。
- **局限性明确**：无法防范抑制攻击（代理不调用服务）、服务合谋、令牌引用枚举等问题。采用激励问题是目前最根本的障碍。

## 7. 优点

1. **架构新颖**：将信任边界反转这一简洁思想落实到实用的密码学协议中，填补了现有系统在接收方签名 + 所有者加密 + 公开日志 + 所有者发现组合上的空白。
2. **使用标准密码学原语**：COSE_Sign1、HPKE、Ed25519、Merkle 透明日志，易于与现有基础设施（如 Sigstore Rekor）集成。
3. **性能优越**：微基准测试显示单次收据操作在亚毫秒级别，适合高吞吐场景（如 1000 调用/秒仅占 0.45 秒 CPU）。
4. **安全设计细致**：
   - 使用 JWS 令牌绑定所有者加密公钥，防止替换攻击。
   - 用日志集成时间而非服务自述时间作为密钥撤销的时间基准，防止回溯伪造。
   - 收据内只包含输入输出的哈希值，保护隐私。
5. **诚实讨论局限**：作者明确列出了抑制攻击、服务合谋、采用激励等未解决问题，并提出了未来扩展方向（监管者作为第二接收方、心跳检测等）。
