## 摘要
智能体越来越多地通过API路由器访问大型语言模型（LLM）。路由器终止客户端的传输层安全会话，并打开一个独立的上游会话，因此它以明文形式持有完整的交互。这使得路由器成为应用层的中间人：它可以重写智能体工具调用，将依赖项替换为拼写错误包，仅在逃避审计的条件下触发攻击，并被动地窃取秘密。现有的客户端防御措施是可规避的。我们提出AEGIS，一个提供者透明的可认证API路由器，其数据路径是客户端验证的忠实直通。AEGIS将明文处理限制在一个小型硬件飞地组件内，同时将认证、调度、计费和管理留在不受信任的主机上。客户端在释放明文前验证飞地。主机既不能读取也不能篡改交互，明文仅离开由测量映像确定的目的地。我们展示了所有四种恶意路由器攻击类别都能针对明文访问基线成功实施，并被AEGIS阻止，包括针对同一边界的适应性测试。可信路径为851行代码，承载三种提供者原生API而无转换，并在真实提供者工作负载和并发下完成每个请求。在一个种子审计试点中，两个商品编码智能体分别发现十个植入的不变量违规中的八个和十个。本地中继开销约为每个请求六毫秒。

## Abstract
Agents increasingly access large language models (LLMs) through API routers. A router terminates the client's transport-layer security session and opens a separate upstream session, so it holds the full interaction in plaintext. This makes the router an application-layer man-in-the-middle: it can rewrite agent tool calls, swap dependencies for typosquatted packages, trigger attacks only under audit-evading conditions, and passively exfiltrate secrets. Existing client-side defenses are evadable.   We propose AEGIS, a provider-transparent attested API router whose data path is a client-verified faithful passthrough. AEGISconfines plaintext handling to a small hardware-enclave component while leaving authentication, scheduling, accounting, and management on the untrusted host. The client verifies the enclave before releasing plaintext. The host can neither read nor alter the interaction, and plaintext leaves only toward destinations fixed by the measured image. We show that all four malicious-router attack classes succeed against a plaintext-access baseline and are blocked by AEGIS, including adaptive tests against the same boundary. The trusted path is $851$ lines, carries three provider-native APIs without conversion, and completes every request under real-provider workload and concurrency. In a seeded audit pilot, two commodity coding agents find eight and ten of ten planted invariant violations. The local relay overhead is about six milliseconds per request.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **问题**：LLM API 路由器（如 LiteLLM、OpenRouter）作为客户端与模型提供方之间的中间代理，会终止客户端的 TLS 会话并建立新的上游连接，从而获得交互的完整明文。这使得路由器成为**应用层中间人**，能够：
  - 重写工具的调用（如 shell 命令），导致远程代码执行；
  - 将依赖命令中的包名替换为拼写错误包，实现供应链攻击；
  - 仅在某些触发条件下实施攻击（回避审计）；
  - 被动嗅探并窃取凭据等敏感信息。
- **背景**：现有客户端侧防御（如策略门控、异常检测、响应日志）均被证明可被绕过，无法从根本上解决路由器掌握明文的问题。而要求提供方新增签名 API 或基础设施又难以部署。
- **意义**：提出一种**提供方透明**的解决方案，无需提供方配合，即可消除路由器对交互的明文访问能力，从而阻断上述四类攻击。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **最小可信基**：仅将处理明文的**数据平面**（客户端 TLS 终止、请求/响应的忠实中继）放入硬件飞地（enclave），而认证、调度、计费、管理等非敏感逻辑保留在不受信任的主机上。
- **客户端先验证后释放明文**：客户端侧的小型 sidecar 在释放任何明文之前，先检查飞地的远程证明（attestation），确认其运行的是可审计、可复现构建的镜像，并绑定会话密钥。
- **忠实中继**：飞地内的中继代码仅做字节级的转发，不进行任何语义转换，因此测量值直接代表代码的忠实性。

### 关键技术细节
1. **信任拆分**：主机通过一个窄控制通道与飞地通信，该通道只携带认证、策略名称等元数据，**绝不携带交互体或网络地址**。表 I 列出了边界所携带和永不携带的字段。
2. **测量锚定**：通过发布、审计、重建、固定四步（ARPA 协议）将飞地测量值锚定到公开审计的源代码上。运行者将构建清单签名后发布到透明日志，客户端的 sidecar 验证日志包含性、一致性并重建源代码，只固定匹配的测量值。
3. **认证协议**（Algorithm 1–3）：
   - 客户端生成随机数 nonce，飞地返回绑定该 nonce、测量值和会话证书的证明（quote）。
   - sidecar 验证签名、测量值是否为固定值 m*、nonce 是否匹配、证书绑定正确。只有全部通过才发送明文。
   - 飞地收到明文后解密，仅向主机选择的策略名称对应的固定目标地址转发请求，并插入提供方凭据；响应字节逐块加密返回客户端，不进行任何改写。

## 6. 论文的主要结论与发现

- AEGIS 成功阻止了所有四类已知恶意路由器攻击（AC-1、AC-1.a、AC-1.b、AC-2），包括各种适应性变异。
- 可信路径仅有 **851 行代码**（整体 1,236 行，对比同类网关的 10,434 行），大幅缩小审计范围。
- 本地中继开销稳定：小请求中位数 **5.7 ms**，95 百分位 6.3 ms；1 MB 载荷时升至 52 ms，4 MB 多模态载荷时 183 ms。
- 真实提供者下，AEGIS 不改变提供者语义，流式分块间隔与直达访问一致（中位数 3.5 ms vs 3.0 ms），端到端延迟由提供者主导。
- 在编程智能体审计试点中，Claude Code 发现了全部 10 个注入违规，Codex 发现了 8 个，表明小型忠实中继代码可被商品智能体有效审查。
- 形式化安全证明（ProVerif 符号化模型 + 游戏归约）确认了秘密性、目的地权威性、响应完整性等属性。

## 7. 优点：方法或实验设计上的亮点

- **最小可信基设计**：仅将数据平面放入飞地，使审计目标小且稳定；主机变更（如计费、管理）不改变测量值，减少重新认证频率。
- **先验证后释放**：客户端 sidecar 在发送任何明文之前完成测量值绑定检查，避免“先发送后验证”的安全漏洞，确保主机从未接触明文。
- **提供者透明**：不要求提供者修改 API 或响应签名，兼容现有提供者。
- **测量锚定到公开透明日志**：通过可复现构建和公共日志确保测量值的公开性和不可否认性，任何用户均可审计。
- **兼容流式响应**：字节级保护使每个流式块都被完整性保护，编码智能体可即时执行工具调用而无需等待响应结束。
- **实验设计全面**：安全测试、性能测试、审计测试三者结合，且同时提供计算归约和符号化验证两种形式化分析。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

- **残余侧信道**：AEGIS 仅隐藏明文内容，但主机仍可观察到请求/响应的大小、时序、流节奏等。已有的攻击可以仅通过加密流大小和时序恢复文本（如远程击键攻击），AEGIS 未应用流量填充或时序扰动来防御。
- **单信任锚**：当前原型使用第一方提供者作为普通 HTTPS 上游，因此飞地仍然需要终止客户端会话并持有明文。作者指出当下一跳也是可认证路由器时，可以变成盲中继，但原型未实现。
- **依赖平台安全性**：假设硬件飞地的安全（内存隔离、证明不可伪造）和证书基础架构（CA）可信。实际的物理/微架构侧信道（如 Foreshadow）或 CA 误发证书属于模型外。
- **审计依赖人工检查**：忠实中继的 `Faithful` 属性目前仅通过拼写测试和模糊测试确保，未进行完整的非干涉性证明（作者承认未来工作）。
- **性能上限有限**：1 MB 以上大载荷开销增长明显（4 MB 时 183 ms），且 AES 加密和虚拟套接字传输导致开销随载荷线性增长，不适用于超大文件或实时要求极高的场景。
- **实验范围限制**：
  - 审计测试仅使用两个编码智能体，且违规为预设模式，未覆盖现实中的复杂隐蔽攻击；
  - 主机控制通道的模式枚举虽覆盖源码，但可能存在未预期的隐式通道；
  - 未测试大量提供者（仅三个）或异构飞地平台（如 Intel SGX、AMD SEV-SNP）的兼容性。
- **部署限制**：需要 AWS Nitro Enclaves 等特定平台支持；运营者需发布可复现构建，增加运维复杂度。
