## 摘要
技能使得LLM智能体能够复用指令、资源、工具和工作流，但也为恶意行为提供了新的藏身之处。一项技能在其文档或代码中可能看似良性，但只有在与特定用户请求、本地资产、持久化状态或多步工具交互一同调用时才会变得有害。这使得纯静态审查变得脆弱。我们提出了运行时技能审计（RSA），一种动态分析方法，通过询问技能中介的智能体在目标运行时条件下实际执行的操作来审计技能。RSA不是用相同的通用任务测试每个技能，而是分析风险相关的接口，准备执行这些接口所需的上下文，并根据由此产生的迹证据分配安全标签。我们在OpenClaw上实例化RSA，并针对100个技能与代表性的静态基线进行评估。RSA达到了90.0%的准确率，88.0%的真正例率和8.0%的假正例率，相比最佳静态基线准确率提高了13.0个百分点。在自进化攻击下，静态检测器在一两轮后崩溃，而RSA在每轮中仍能检测出20个恶意技能中的19到20个。

## Abstract
Agent skills let LLM agents reuse instructions, resources, tools, and workflows, but they also create a new place for malicious behavior to hide. A skill may look benign in its documentation or code while becoming harmful only when it is invoked with particular user requests, local assets, persistent state, or multi-step tool interactions. This makes purely static vetting brittle. We present Runtime Skill Audit (RSA), a dynamic analysis method that audits skills by asking what the skill-mediated agent actually does under targeted runtime conditions. Instead of testing every skill with the same generic tasks, RSA profiles risk-relevant interfaces, prepares the execution context needed to exercise them, and assigns security labels from the resulting trace evidence. We instantiate RSA on OpenClaw and evaluate it on 100 skills against representative static baselines. RSA achieves 90.0\% accuracy with an 88.0\% true positive rate and an 8.0\% false positive rate, improving accuracy by 13.0 percentage points over the best static baseline. Under self-evolving attacks, static detectors collapse after one or two rounds, while RSA continues to detect 19--20 out of 20 malicious skills across rounds.

# Runtime Skill Audit 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

- **问题背景**：随着 LLM Agent（如 OpenClaw、Claude Code 等）的普及，Agent 技能（skill）成为复用指令、工具和工作流的关键机制。然而，开放技能生态系统引入了新的安全威胁：恶意技能可以在看似良性的文档或代码中隐藏恶意行为，仅在特定运行时条件（用户请求、本地资产、持久状态、多步工具交互）下才显现。这种“执行依赖性”使得纯静态审查（如模式匹配、LLM 对指令的语义判断）极易被绕过。
- **核心挑战**：  
  1. 技能执行高度依赖**使用场景**：同样的技能在不同请求下行为不同。  
  2. 评估需要**环境依赖**：恶意行为只有在与本地文件、工具、外部服务交互时才暴露。  
  3. 恶意行为往往是**执行依赖**的：可能由间接触发或多步操作展开。
- **研究意义**：传统的静态分析方法（如规则匹配、LLM 静态评分）无法可靠地判断技能在真实运行时是否安全。因此，需要一种面向运行时的动态审计方法，为 Agent 技能安全提供行为层面的证据。

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：RSA 是一种动态分析方法，通过“以风险为导向的任务生成 → 上下文感知执行 → 痕迹接地判断”三个步骤，将技能审查转变为对运行时行为的审计。
- **关键技术步骤**：
  1. **风险引导的技能分析（Risk-Guided Skill Profiling and Tasking）**  
     - 输入原始技能工件（指令、脚本、资源等）。  
     - 利用 LLM 和预定义的工具分类法（Tool Taxonomy）推断技能可能调用的平台工具及其风险面（文件访问、Shell 执行、网络交互、内存访问等）。  
     - 生成结构化的安全档案（profile），记录技能的可能良性用途、工具接口、敏感资源和猜测的攻击模式。  
     - 基于档案而非通用任务列表，生成**针对性的运行时探测任务**（包含用户请求、沙箱准备和可复现约束）。
  2. **上下文感知的运行时执行（Context-Aware Runtime Execution）**  
     - 在隔离的沙箱环境中执行每个探测任务，仅启用目标技能。  
     - **上下文准备**：根据任务生成相关文件、目录、模拟凭据、持久状态等，以触发潜在恶意分支。  
     - 使用**自修复循环**（bounded repair loop）处理执行失败的情况（如任务准备不完整），尝试自动修复并重试。  
     - 对依赖外部应用或平台的技能，使用**确定性伪造应用垫片**（mock shims）记录技能如何使用不可用的接口。
  3. **痕迹接地判断（Trace-Grounded Judgment）**  
     - 收集执行轨迹（结构化 trace），包括工具调用、文件读写、网络请求、内存交互、拒绝行为等。  
     - 评估者（evaluator）检查：良性目标是否完成、风险面是否被激活、观察到的动作是否表明安全相关的滥用。  
     - 输出四种判定：  
       - *benign executed*：完成任务且无安全副作用。  
       - *harmful blocked*：存在恶意分支但 agent 阻止了执行。  
       - *harmful executed*：存在恶意动作。  
       - *uncertain*：轨迹不完整。  
     - 判定基于具体轨迹事件证据，而非静态关键词语义。
- **附增强组件**：  
  - **知识库（Knowledge base）**：固定资源，包含工具分类、风险映射、攻击模式模板、判定定义等，指导探测和证据解释。  
  - **记忆模块（Memory）**：跨技能执行累积的压缩摘要（有效触发器、误报模式、遗漏行为等），避免重复探测，提升一致性。  
  - 最终判定仍依赖于当前轨迹；知识库和记忆仅起引导作用。
- **流程文字总结**：  
  输入技能工件 → [Profiling] 生成安全档案 → [Task Generation] 生成针对性任务 → [Execution] 在沙箱中执行并记录轨迹 → [Judgment] 根据轨迹输出安全标签。

## 6. 主要结论与发现

1. **动态审计显著优于静态审计**：  
   - RSA 在 100 技能上达到 90.0% 准确率，88.0% TPR，8.0% FPR，比最佳静态基线（SkillScanner 77.0% 准确率）提升 13 个百分点，同时 FPR 从 12% 降至 8%。
   - 静态基线难以平衡检测率和误报率：SkillScanner（LLM 变体）虽 TPR 极高但 FPR 高达 90%，实际上无法使用。
2. **对自适应攻击的鲁棒性**：  
   - 静态检测器（Aguara, SkillScanner）在 1-2 轮突变后检测率骤降至 0，因为它们仅依赖工件表面特征，攻击者可以反向工程规避。  
   - RSA 在全部突变轮次中保持 19-20/20 的检测率，因为恶意行为最终会通过运行轨迹暴露，无法通过隐藏工件表面形式而彻底消除。
3. **知识引导组件的作用**：  
   - 移除知识库后 RSA 的 TPR 下降 6 个百分点（82%→88%），FPR 上升 8 个百分点（16%→8%），说明知识库帮助生成更有针对性的任务并更一致地解释轨迹证据。
4. **案例验证**：  
   - 在 `report-brief-sync` 技能中，RSA 通过轨迹证据发现了数据外泄分支（脚本加载配置、重构路径、准备 HTTP POST），即使 agent 阻止了执行，RSA 仍将其标记为 `harmful blocked`，实现了行为层面的准确判定。

## 7. 优点

- **方法创新**：将安全审计从静态分析转向“行为接地”的动态审计，直接回答“技能在运行时到底做了什么”，更贴近真实威胁。
- **风险导向**：不对所有技能采用通用探测任务，而是先分析风险接口再生成针对性任务，提高探测效率和覆盖率。
- **上下文意识**：通过动态准备文件、状态等上下文，能触发仅在特定环境下才出现的恶意分支。
- **健全的判定体系**：区分“恶意分支存在但被阻止”与“恶意执行”，保留行为证据，避免将 agent 防护能力误判为技能安全。
- **自修复与垫片机制**：提高了动态执行的鲁棒性，降低因环境准备失败导致的无效轨迹。
- **对抗性鲁棒性**：在自适应攻击下稳定性突出，验证了动态审计在长期对抗中的优势。
- **可复现与开放式贡献**：代码已开源，实验细节（数据集构成、工具分类、任务 schema）提供完整附录。

## 8. 不足与局限

- **平台依赖与规模限制**：仅在 OpenClaw 上实例化，100 技能的评估集规模较小，恶意技能部分来自受控生成和有限攻击模式，**不能完全代表真实世界攻击多样性**。未来需要更多平台和更大数据集验证。
- **LLM 组件脆弱性**：RSA 的分析、任务生成、判断均依赖 LLM，可能存在模型偏差、提示敏感性、不一致性等问题，影响审计质量。
- **动态审计非穷尽**：生成的探测任务并非完全覆盖所有可能的运行时触发条件；一个技能可能在未测试的输入、外部状态或长程交互下仍存在隐藏恶意行为。RSA 应被视为“行为测试”而非“完全安全证明”。
- **计算成本**：动态执行需要真实运行 agent，相比静态扫描开销更大；论文未量化成本（如 LLM 调用次数、执行时间），实际部署可能受限。
- **缺乏对其他攻击类型的评估**：未考虑更复杂的供应链攻击（如依赖包投毒、模型权重操纵）或基于 agent 记忆的长期攻击。
- **修复循环的局限性**：自修复仅针对简单失败场景（缺少文件、权限问题等），对复杂环境依赖或平台限制可能无法修复，导致部分技能无法获得可用轨迹。
- **消融实验不够深入**：仅对“知识有无”进行消融，未单独分析记忆模块、修复循环、垫片机制等组件的贡献。
