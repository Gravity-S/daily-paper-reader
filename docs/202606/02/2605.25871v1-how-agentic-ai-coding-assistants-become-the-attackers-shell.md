## 摘要
智能体AI编程助手可以编辑文件、运行命令以及代表开发者访问互联网。然而，它们对外部未审查工件的依赖引入了一种新的攻击向量。外部工件中的隐藏指令可以劫持这些助手，将其变成攻击者的Shell，以执行未经授权的命令。在本文中，我们研究了这些提示注入攻击的工作原理，衡量了它们的普遍性，讨论了当前防御的局限性和挑战，并提出了未来的研究方向。

## Abstract
Agentic AI coding assistants can edit files, run commands, and access the internet on behalf of developers. However, their reliance on unvetted external artifacts introduces a new attack vector. Hidden instructions in external artifacts can hijack these assistants, turning them into an attacker's shell to run unauthorized commands. In this article, we examine how these prompt injection attacks work, measure their prevalence, discuss the limitations and challenges of current defenses, and suggest future research directions.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：智能体AI编程助手（如Cursor、GitHub Copilot）正变得越来越自主，能够代表开发者编辑文件、运行命令、访问互联网。这种自动化显著提高了开发效率，但也引入了严重的安全风险。
- **核心问题**：开发者经常从在线来源导入外部工件（如编码规则文件、代理技能文件、项目模板）到工作区。这些外部工件可能包含攻击者隐藏的恶意指令（提示注入）。由于当前AI编程助手无法可靠地区分受信任的开发者指令和不受信任的攻击者载荷，隐藏指令可以劫持助手，使其成为攻击者的“Shell”，在开发者不知情或未批准的情况下执行未授权命令（如窃取凭证、修改认证文件、安装后门等）。

## 6. 论文的主要结论与发现

- **威胁是系统性的，而非偶然**：攻击成功率在41%~84%之间，跨不同编程语言、工具和底层模型一致，表明这是架构性缺陷。
- **损害覆盖整个攻击生命周期**：被劫持的助手能够完成从侦察、凭证窃取、持久化到最终渗出数据的完整攻击链，甚至可以创建新用户、安装cron任务等。
- **助手的智能性使攻击更有效**：与传统固定脚本的恶意软件不同，AI助手能自适应调整策略（例如扫描目录后自动细化搜索范围），即使攻击指令是高层次的，也能成功达成目标。
- **现有防御存在结构性不足**：
  - 权限限制（如命令白名单、信任对话框）可被绕过（例如在对话框显示前触发攻击，或通过提示注入启用自动批准）。
  - 安全过滤器（后端系统提示）无法从根本上解决问题，因为所有输入（开发者提示、文件、外部数据）合并为单一token流，无法区分可信与不可信内容。
- **攻击面广泛**：不仅限于规则文件，还包括仓库/工作区文件、社区共享的技能文件、连接的服务（MCP）、实时上下文（聊天、邮件、网页）等。
