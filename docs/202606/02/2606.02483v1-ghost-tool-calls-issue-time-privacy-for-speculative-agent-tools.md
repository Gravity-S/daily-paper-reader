## 摘要
增强工具的语言代理会推测性地发出可能的未来工具调用以隐藏延迟，但这些调用会在代理承诺分支之前将推断出的用户意图泄露给外部服务。每个接收到该调用的外部观察者在代理放弃该分支后仍然保留该披露信息。问题在于时机而非授权：没有提交时清理、只读限制或访问控制允许列表能够撤回观察者已经持有的信息。我们将这些调用称为幽灵工具调用，并提出了推测性工具隐私合约，这是一种运行时抽象，将承诺前的观察视为一等效应，与状态变更区分开来。我们在原型运行中实现了这些合约，并评估了三个语料库上的十二种策略。推测性分发增加了观察者可以推断的用户意图；事后过滤器、只读限制和访问控制允许列表保持该推断不变；只有那些在分发前改变或抑制推测性调用的参数或目标投影的发布时策略才能减少推断。

## Abstract
Tool-augmented language agents speculatively issue likely future tool calls to hide latency, but those calls leak inferred user intent to external services before the agent commits to the branch. Every external observer that received the call retains the disclosure after the agent abandons the branch. Timing is the issue, not authorization: no commit-time cleanup, read-only restriction, or access-control allow-list unsends what an observer already holds. We call these invocations ghost tool calls and propose Speculative Tool Privacy Contracts, a runtime abstraction that treats observation before commitment as a first-class effect, distinct from state mutation. We implement the contracts in a prototype runtime and evaluate twelve policies across three corpora. Speculative dispatch increases what an observer can infer about user intent; post-hoc filters, read-only restrictions, and access-control allow-lists leave that inference intact; only issue-time policies that change or suppress the speculative call's argument or destination projection before dispatch reduce it.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：工具增强的语言代理（tool-augmented language agents）为了隐藏延迟，会推测性地（speculatively）发出未来的工具调用（如搜索、API调用）。然而，这些调用在代理最终承诺（commit）某个分支之前，就已经将推断出的用户意图泄露给了外部服务（如搜索提供商、日志系统）。即使用户最终没有用到这些推测性调用的结果（称为“幽灵工具调用”），外部观察者仍然保留了这些信息，且无法撤回。
- **时机问题，而非授权问题**：现有的安全机制（如访问控制列表、只读限制、事后清理）都无法“撤销”已经发送给观察者的信息。关键在于，泄露发生在调度时刻（issue time），而不是提交时刻（commit time）。
- **研究动机**：当前主流推测执行系统（如SIA、PASTE等）只关注状态变更的安全性（读/写、可逆/不可逆），而忽略了观察者状态的安全性。作者提出，需要一种发布时（issue-time）的隐私策略，在推测性调用离开受信任边界之前决定是否、如何发送。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将“提交前的观察”（observation before commitment）作为一等效应（first-class effect），与状态变更区分。提出**推测性工具隐私合约（Speculative Tool Privacy Contracts）**，它是一个运行时抽象。
- **关键技术细节**：
  - **合约标签**：每个工具事件在被调度之前，合约会绑定多个字段：执行模式（committed/speculative等）、效果类（pure-local/external-read等）、观察者类（provider/network/runtime log等）、参数标签（public/personal/sensitive/intent-revealing）、目标标签（public/tenant-internal/sensitive等）、分支置信度/预算、审计策略。
  - **决策函数**：对于提议的事件e和合约κ，监控器计算D(e,κ) → {ALLOW, REWRITE, SHADOW, DEFER, BLOCK}，按失败关闭管线执行：若工具或目标不在允许列表则BLOCK；若事件是推测性的且分支置信度低于阈值则DEFER；若任何参数被标记为高敏感性则REWRITE；若需要非提供商可见的替代品且目标是外部则SHADOW；若超出每任务披露预算则DEFER；否则审计并ALLOW。
  - **动作语义**：
    - ALLOW：不变调度。
    - REWRITE：对敏感参数进行变换（如实体编辑、时间范围粗化、意图短语替换）。
    - SHADOW：路由到受信任的本地替代品，结果对提供商不可见。
    - DEFER：排队等待，仅当分支被提升到承诺计划时才重新评估调度。
    - BLOCK：丢弃事件并记录审计。
  - **安全性（定理1）**：在完全中介、标签正确、动作保持的条件下，监控器在发布时阻止了合约禁止的字段进入任意观察者的边际暴露Δi。

## 6. 论文的主要结论与发现

- **推测性跟踪会泄露推断出的意图**：静态计划实验中，Naive策略下对抗恢复率从No-Spec的0.44上升到0.97；真实规划器下从0.34上升到0.49。事后过滤器（Late-Scrub, SIA, ACL）和只读限制无效，接近Naive。
- **只有发布时变换才能减少泄露**：Rewrite（改写参数）和Shadow（路由到本地替代）显著降低恢复率（Shadow降至0.00，Rewrite至0.04~0.09）。Pre-Scrub（PII正则）无效，因为结构化字段和目的地未受影响。
- **残余信号主要来自参数和目的地URL**：工具名掩盖对指标无影响；目的地URL包含主题名称；参数通道单独携带大部分泄露。
- **结果对多对手、真实提供商、标签噪声和异步延迟都稳健**：跨模型和跨语料的排序一致。Gate提供可调隐私/效用权衡。AgentDojo子集也确认了排序。
- **运行时日志泄露几乎与提供商相同**；网络和共享缓存通道在当前语料中未展示显著泄露。
- **合约实施不会成为延迟瓶颈**：异步延迟节省5-12%，合约策略（Rewrite等）与Naive异步的延迟差异在1.5pp以内。

## 7. 优点

- **概念创新**：首次形式化“幽灵工具调用”和“发布时隐私 vs 提交时隐私”的区别，提出观察者状态作为一等效应。
- **通用框架**：合约标签体系覆盖多维度（执行模式、效果类、观察者类、标签、置信度等），决策函数包含五种动作，可适用于任何推测性运行时。
- **实验严密**：采用配对重放设计（固定推测前缘、仅变调度器行为）以消除规划器方差；使用多种对手（LLM、关键词、TF-IDF、朴素贝叶斯）和多种指标（top-k, NMI, 语义开放集, 诱饵标签）；进行了标签噪声鲁棒性测试。
- **实际验证**：不仅在模拟环境中评估，还使用真实搜索引擎（Brave, DuckDuckGo）进行HTTP边界验证，并与现有系统（SIA）直接比较。
- **可重复性**：作者承诺发布原型运行、冻结前缘文件、所有策略实现和提示，便于复现。

## 8. 不足与局限

- **完全中介依赖**：合约依赖所有IO路径（缓存、重试、度量、审计等）都经过监控器，违反此假设会破坏安全性。实际部署中易出漏洞。
- **标签正确性依赖**：规则标签器在参数敏感性上Macro-F1仅0.10，目的地0.59，效果0.45，远低于定理要求的水平。虽然实验显示排名在20%噪声下仍稳健，但绝对保证需要更强标签器。
- **模型提供商信任**：若规划器是远程LLM（如Claude），则其本身可观察用户输入和候选分支，合约无法控制模型提供商对内部状态的访问。作者将此排除在威胁模型外。
- **对手覆盖有限**：仅测试了3种商用LLM和3种非LLM基线，未测试子10B开放权重模型、人类分析师或拥有私有标记跟踪的对手。
- **通道覆盖不全**：仅测量了网络、运行时日志和一个共享缓存通道；多租户竞争、排队延迟、速率限制残留、代理间泄露等未测量。
- **影子替代保真度**：Shadow的本地替代可能降低下游任务成功，作者声明隐私索赔独立于替代质量，但效用测量显示Shadow任务成功与Naive重叠，在某些任务上可能更低。
- **有限的外部有效性**：AgentDojo体验了完成率偏差（仅报告了66/86个任务），且仅一个种子。语料均为生成的敏感主题，未涵盖更多样化的真实用户场景。
- **延迟数字为序数比较**：异步延迟节省依赖具体实现（DuckDuckGo），作者明确不声称是生产级加速。
