## 摘要
AI代理越来越多地通过框架在生产环境中运行，即围绕大语言模型的软件，包括执行安全和有效性策略（例如“提交前运行测试”）的引擎。执行这些策略需要弥合语义鸿沟：策略意图以未详细说明的自然语言表达，而执行必须作用于具体的系统动作（例如运行哪个测试）。许多策略还定义了事件顺序或数据流动作。然而，现有方法存在不足。工具调用护栏会绕过工具层的系统动作，而操作系统沙箱控制资源访问而非动作，返回不透明的错误使代理困惑。我们的关键洞察是：策略上下文位于最接近任务的代理中，而执行必须在操作系统层面进行以覆盖所有执行路径。我们提出ActPlane，一个策略引擎，允许代理声明策略，并在操作系统内核中通过语义反馈和隔离执行策略。ActPlane使用一个简单的信息流控制（IFC）DSL来支持跨事件策略。我们使用eBPF实现ActPlane，并通过实证研究、编码任务基准和安全基准中的策略进行评估。ActPlane提高了策略合规性，包括工具调用拦截无法观察到的间接执行路径，开销为1.9%-8.4%。ActPlane位于https://github.com/eunomia-bpf/ActPlane

## Abstract
AI agents increasingly run in production through harnesses, the software around the LLM, including an engine that enforces safety and effectiveness policies, e.g., 'run tests before committing.' Enforcing these policies requires bridging a semantic gap: policy intent is expressed in underspecified natural language, while enforcement must act on concrete system actions, e.g., which test to run. Many policies also define event ordering or data flow actions. Yet existing approaches fall short. Tool-call guardrails miss system actions that bypass the tool layer, while OS sandboxes control resource access instead of actions, returning opaque errors that confuse the agent. Our key insight is that policy context lives within the agent closest to the task, while enforcement must happen at the OS to cover all execution paths. We introduce ActPlane, a policy engine that lets agents declare policies and enforces them in the OS kernel with semantic feedback and isolation. ActPlane uses a simple information-flow control (IFC) DSL to support cross-event policies. We implement ActPlane with eBPF and evaluate it on policies from the empirical study, coding-task benchmarks, and safety benchmarks. ActPlane improves policy compliance, including on indirect execution paths that tool-call interception cannot observe, with 1.9%-8.4% overhead. ActPlane is at https://github.com/eunomia-bpf/ActPlane

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：AI代理（如Claude Code、Codex）在生产环境中通过“框架”（harness）运行，框架包含策略引擎，用于强制执行安全与有效性策略（例如“提交前先运行测试”）。这些策略通常以自然语言写在 `CLAUDE.md` 或 `AGENTS.md` 文件中，但执行时需转换为具体的系统动作（如执行 `go test ./...`）。
- **核心问题**：**语义鸿沟**（semantic gap）——自然语言策略意图过于抽象，而执行需绑定到具体系统动作；大量策略依赖项目或任务上下文（73.6%的策略非自包含），且涉及跨事件约束（如事件顺序、数据流），现有方法无法有效覆盖。
- **现有方法不足**：
  - 工具调用护栏（prompt-filter, tool-regex, FIDES）：只能拦截工具层调用，无法观察间接执行路径（如子进程、脚本内部命令）；跨事件状态缺失。
  - OS沙箱（seccomp, AppArmor, Landlock）：控制资源访问而非动作；返回不透明错误（如EPERM），让代理困惑；期望静态预定义策略。
- **本文意义**：提出ActPlane，在OS内核层面通过eBPF实现可编程策略引擎，让代理声明策略并强制执行，同时提供语义反馈，弥合语义鸿沟。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- 策略上下文位于**最接近任务的代理**中（已读取仓库、理解当前任务），而执行必须在**OS内核层**进行，以覆盖所有执行路径（包括子进程、脚本、编译二进制）。
- 代理编写具体策略DSL，编译为eBPF程序在内核执行；支持跨事件策略（IFC）和分层安全域。

### 关键技术细节

#### a) 策略DSL
- 简单DSL，接近自然语言，代理能可靠生成。每条规则包含：
  - **source**：系统对象（exec、file、endpoint）
  - **target**：操作（exec、open、write、connect等）
  - **effect**：`block`（预先阻断）、`kill`（事后终止）、`notify`（仅警告）
  - **temporal gate**：`after ... since ...`（时间顺序）、`lineage-includes`（进程血统）、`exits N`（退出码条件）
  - **reason**：语义反馈文本
- 示例：`kill exec "git" "commit" unless after exec "go" "test" exits 0 since write "**/*.go"`

#### b) 信息流控制（IFC）
- 每个进程、文件、网络端点携带标签集（bitmask），根据OS操作（fork, exec, read, write, connect）传播标签。
- 规则根据源标签检查是否匹配；标签单调（只加不减），保证跨事件检查能力。
- 类似于Flume、HiStar的IFC模型，但去掉了显式子类规则（declassify/endorse用于降级）。

#### c) 分层策略域（Hierarchical Policy Domains）
- 每个进程树属于一个域，子域继承父域策略，可添加本地规则但不能削弱或删除父域规则。
- 运行时更新delta必须通过内核权限检查器，确保不修改继承位。
- 防止代理绕过高级别安全约束（如用户或平台管理员设定的规则）。

#### d) 实现
- 用户空间：Rust编译器/运行时（~3.2K LoC），解析DSL生成eBPF配置。
- 内核空间：eBPF程序（~1.8K LoC BPF C），挂载到exec、open、read、write、unlink、connect、fork、exit等hooks；使用BPF maps存储标签位掩码；支持128条并发规则。
- 语义反馈：内核仅输出规则ID，用户空间格式化后返回给代理。

## 3. 实验设计

### 数据集/场景
1. **实证研究**：64个流行GitHub仓库的CLAUDE.md/AGENTS.md，提取2116条语句，分析策略类型、上下文依赖性。
2. **决策合规基准（RQ2）**：从607条OS可执行策略中采样38条规则（20 per-event + 18 cross-event），每条规则生成5条轨迹（2条合规 + 3条违规），共计190条轨迹。违规轨迹分为三类：直接工具调用、脚本间接执行、隐藏间接执行。
3. **编码任务基准（RQ4）**：OctoBench的21个任务子集（含61条OS可执行策略项），评价官方reward。
4. **安全基准（RQ5）**：OpenAgentSafety的361个任务，覆盖凭据泄露、系统篡改、金融记录篡改等场景。

## 6. 主要结论与发现

- **策略分布**：64%指令文件语句为策略；83%策略可系统观测；16%需跨事件状态；73.6%需项目或任务上下文。
- **ActPlane有效性**：DCR 75.8%，比prompt-filter（48.4%）、tool-regex（45.3%）、FIDES（48.9%）、无反馈Kernel IFC（53.7%）高22-31个百分点；对违规轨迹检测率77.2%（baseline仅34-40%），经一次修正达94.7%。
- **间接路径覆盖**：在脚本和隐藏违规家族上，ActPlane检测率远高于工具层方法（接近零）。
- **语义反馈关键**：完整ActPlane比无反馈版本多3倍正确违规处理（86 vs 27），恢复率97.7%。
- **安全性**：在361安全任务中，防止74%的不安全行为（78/106）。
- **开销**：宏基准1.9-8.4%端到端开销；微基准中打开文件等新增13.4μs，fork/exec新增几十微秒；相对LLM推理（2-10秒）可忽略。
- **DSL可生成性**：所有607策略均一次或两次翻译成功，L1成本仅$0.028/策略。

## 7. 优点

- **创新性**：首次在OS内核层面实现面向AI代理的可编程策略执行，结合IFC与eBPF，弥合语义鸿沟。

- **安全性**：分层域设计确保高级别策略不可被代理削弱，满足实际部署需求。
- **实验全面**：覆盖实证研究、决策合规、编码任务、安全基准；对比多个主流方法；消融实验揭示反馈价值；跨模型验证稳定性。

## 8. 不足与局限

- **覆盖范围有限**：仅覆盖45%的策略（per-event + cross-event）；内容策略（如代码风格检查）需要linter等静态分析；语义纯策略（如沟通风格）无法处理。
- **内容安全盲区**：无法检测不安全文件内容，只能监控写入操作本身（如写`.ssh/config`可被检测，但写恶意内容无法判断）。
- **服务端协议盲区**：无法拦截通过WebDAV、数据库API等协议进行的操作（hook覆盖不全）。
- **标签单调膨胀**：长时间运行可能过度限制（over-tainting），虽可派生新进程来缓解，但非根本解决（declassify/endorse未引入）。
- **实验偏差**：轨迹生成与测试代理均使用Codex，可能低估跨框架泛化；人工验证样本量（100条）较小。
- **现实部署成本**：需要内核eBPF支持（Linux 6.15+），且需特权进程加载策略，增加TGB。
- **动态更新限制**：运行时delta目前只支持部分钩子，更复杂门控的更新尚未实现。
