## 摘要
AI智能体越来越多地通过外部工具行动：它们读取私有数据、构造结构化载荷、提交写入请求、导出记录，并跨应用边界协调工作流。现有的授权机制通常询问集成凭据、应用或令牌是否可以调用某个工具。这个问题是必要的，但不够充分。一个工具调用可能被静态凭据授权，但仍然无法由用户当前请求证明其合理性。例如，一个可以读取和导出记录的凭据，当用户仅要求一个有限度的摘要时，不应暴露导出权限；而模型生成的删除调用也不应仅仅因为集成拥有删除范围而执行。本文提出意图驱动的访问控制（IGAC），这是一种服务器端授权层，将用户表达的意图视为AI智能体工具使用的单调、可审计的策略属性。IGAC引入了意图证书、会话范围的策略收窄、意图感知的清单过滤以及意图-工具-载荷一致性检查。核心不变性是：用户意图只能减少由静态集成策略授予的权限；它永远不会扩展范围、数据策略、租户边界或审查要求。我们将IGAC映射到OpenPort上，OpenPort是一个现有的治理基础，已经实现了授权依赖发现、范围和ABAC风格的策略检查、草稿优先写入、预飞影响绑定、状态见证检查、幂等性、稳定原因码和审计。

## Abstract
AI agents increasingly act through external tools: they read private data, construct structured payloads, submit write requests, export records, and coordinate workflows across application boundaries. Existing authorization mechanisms usually ask whether an integration credential, app, or token can call a tool. That question is necessary but incomplete. A tool call can be authorized by static credentials and still be unjustified by the user's current request. For example, a credential that can read and export records should not expose export authority when the user only asked for a bounded summary, and a model-generated delete call should not execute merely because the integration has a delete scope. This paper proposes Intent-Governed Access Control (IGAC), a server-side authorization layer that treats the user's expressed intent as a monotone, auditable policy attribute for AI-agent tool use. IGAC introduces intent certificates, session-scoped policy narrowing, intent-aware manifest filtering, and intent-tool-payload consistency checks. The central invariant is that user intent may only reduce the authority granted by static integration policy; it never expands scopes, data policy, tenant boundaries, or review requirements. We map IGAC onto OpenPort, an existing governance substrate that already implements authorization-dependent discovery, scope and ABAC-style policy checks, draft-first writes, preflight impact binding, state-witness checks, idempotency, stable reason codes, and audit.

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：在现有的效果控制层（如OpenPort）之前插入一个意图控制层，通过意图证书、会话策略收窄、意图感知清单过滤、意图-工具-载荷一致性检查，确保每个工具调用与用户当前表达意图一致。
- **关键技术细节**：
  - **意图证书**：包含请求哈希、意图类（read/summarize/create/update/delete/export/admin/unknown）、资源边界、效果边界、置信度、审查模式、过期时间、分类器来源等。证书由服务器基于用户请求生成，仅作为缩减权限的输入。
  - **会话策略收窄**：从意图证书导出会话策略 \(P_C\)，满足 \(P_C \preceq P_A\)（\(P_A\)为静态策略），即只能隐藏工具，不能新增可见工具。
  - **清单过滤**：可见工具集 = 静态可见工具集 ∩ 会话策略允许的工具。确保 \(Visible_{IGAC} \subseteq Visible_{OpenPort}\)。
  - **意图-工具-载荷一致性检查**：检查 \(Consistent(C, t, x, e)\)，包括工具类是否在允许类中、资源标识符是否在边界内、日期/行数/风险匹配等。若不通过，则拒绝、转为草稿或预飞。
  - **置信度感知路由**：低置信度触发澄清或强制审查；中等置信度使用草稿/预飞；高置信度且低风险才允许直接执行。
  - **形式化模型**：定义了实体（用户请求、应用、凭据、资源、工具等）、静态授权谓词、证书有效性、单调会话策略、一致性谓词、最小权限目标、审计可重构性，并证明了三个安全命题（清单单调性、分类器错误不扩大权限、高风险失败关闭）。
- **算法流程**（文字说明）：
  1. 用户请求到达意图网关，生成意图证书。
  2. 服务器验证证书有效性，导出会话策略。
  3. 过滤工具清单，仅保留会话策略允许的工具。
  4. 模型/规划器在已过滤清单上选择工具并构造载荷。
  5. 一致性门检查工具、载荷、预期效果与证书是否一致。
  6. 通过后进入OpenPort效果控制层（草稿/预飞/幂等/审计等）。

### 6. 论文的主要结论与发现

1. **IGAC能有效减少高风险工具暴露**：在确定性微基准中，不安全执行率从OpenPort-only的0.1429降至0；真实LLM运行时基准中不安全执行率为0。
2. **分类器存在“意图宽泛但边界不精确”问题**：意图类准确率约0.73-0.85，但资源边界提取准确率仅0.18-0.58，表明模型擅长识别用户要什么，但难以提取精确数量/ID。这支持了IGAC将分类器输出作为建议而非权威的设计。
3. **剩余不安全接受权限集中在受限创建/负载扩展场景**：在运行时基准中，仍有9%-27%的攻击任务通过草稿路径被接受（无直接执行），主要原因是证书中缺少足够精确的资源/效果边界，或规划器被诱导向创建类草稿路径。
4. **效果控制的完整性：仅靠清单过滤不够**：清单过滤降低了不安全提议率，但只有加上一致性门才能完全消除不安全执行。
5. **正常任务完成率受审查负担影响**：BCR严格（直接完成）为0（所有写操作转为草稿），BCR安全（含草稿/预飞）约0.25-0.50，说明当前IGAC对良性高风险的写操作仍要求审查，牺牲了自动化便利性。

### 8. 不足与局限

- **任务规模与代表性**：所有任务均为手工编写或仿写，缺乏真实用户请求分布；外部基准适配仅25/12任务，且内容仍为合成，不能代表原基准的难度和多样性。

- **分类器质量限制**：实际LLM分类器在资源边界提取上表现差，导致IGAC无法完全消除不安全接受权限；正常化器消融显示严格正常化会损害良性完成率。
- **未覆盖的威胁**：论文明确排除恶意服务器操作者、网关绕过、直接数据库入侵等；对元数据投毒仅在清单层面处理，未涉及工具实现内的语义欺骗。

- **延迟与实用性**：虽然报告了P95延迟（如Llama-3.3-70B达50秒），但未进行优化讨论或与生产延迟要求对比；草稿/预飞带来的用户交互开销也未评估。
- **可能的选择偏差**：外部基线对比中未包含商业闭源模型（如GPT-4o），且仅使用3个开源模型，可能限制结论的普适性。
