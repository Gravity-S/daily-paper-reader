## 摘要
大型语言模型（LLM）智能体通过将语言模型与外部工具和环境集成，日益自动化地完成复杂任务。然而，它们的自主性带来了显著的安全风险：智能体可能执行破坏性命令、泄露敏感数据或违反领域约束。现有的安全方法面临一个基本的权衡：手工制定的规则可解释但脆弱，过于保守的规则会阻止安全操作（高误报率），而宽松的规则则会遗漏不安全行为（高漏报率）。神经分类器缺乏安全关键部署所需的可解释性。我们提出AutoSpec，一个通过反例引导的归纳综合（CEGIS）并借助归纳逻辑编程（ILP）自动从用户安全/不安全标注中演化部署的专家设计安全规则的框架。从专家规则和带标注的轨迹流开始，AutoSpec迭代评估规则，挖掘误报和漏报反例，使用ILP学习哪些谓词区分它们，生成候选规则编辑，并验证候选规则以选择最佳修订。关键洞察在于ILP高效识别那些在漏报中频繁出现但在误报中很少出现（反之亦然）的谓词，从而大幅剪枝规则编辑的指数级搜索空间。此过程持续至收敛，产生平衡精确率和召回率的可解释规则。我们在涵盖代码执行和具身智能体领域的291条执行轨迹上评估了AutoSpec。AutoSpec在两个领域将规则F1提升至0.98和0.93，在保持高召回率的同时实现了高达94%的误报减少，并在4-5次迭代内收敛。ILP引导的方法比启发式CEGIS实现了高达4.8倍的F1提升。学习到的规则可读、可审计，并能泛化到未见场景。

## Abstract
Large language model (LLM) agents increasingly automate complex tasks by integrating language models with external tools and environments. However, their autonomy poses significant safety risks: agents may execute destructive commands, leak sensitive data, or violate domain constraints. Existing safety approaches face a fundamental tradeoff: hand-crafted rules are interpretable but brittle, with overly conservative rules blocking safe operations (high false positives) while permissive rules miss unsafe behaviors (high false negatives). Neural classifiers lack the interpretability required for safety-critical deployments.   We present AutoSpec, a framework that automatically evolves deployed expert-designed safety rules from user safe/unsafe annotations through counterexample-guided inductive synthesis (CEGIS) guided by inductive logic programming (ILP). Starting from the expert rules and a stream of annotated traces, AutoSpec iteratively evaluates rules, mines false-positive and false-negative counterexamples, uses ILP to learn which predicates discriminate them, generates candidate rule edits, and verifies candidates to select the best revision. The key insight is that ILP efficiently identifies predicates that appear frequently in false negatives but rarely in false positives (or vice versa), dramatically pruning the exponential search space of rule edits. This continues until convergence, producing interpretable rules that balance precision and recall.   We evaluate AutoSpec on 291 execution traces spanning code execution and embodied agent domains. AutoSpec raises rule F1 to 0.98 and 0.93 across the two domains, achieving up to 94% false positive reduction while maintaining high recall, and converges within 4-5 iterations. The ILP-guided approach achieves up to 4.8x higher F1 than heuristic CEGIS. The learned rules are human-readable, auditable, and generalize to unseen scenarios.

- **核心问题**：LLM 智能体（LLM agent）在自主执行任务时（如代码执行、机器人控制）存在严重安全风险。现有安全方法面临根本性权衡：手工制定的安全规则（如 AgentSpec）可解释但脆弱，过于保守会阻塞正常操作（高误报），过于宽松则遗漏不安全行为（高漏报）；神经分类器（如 Llama Guard）适应性强但不可解释，无法满足安全关键部署的审计需求。
- **研究动机**：实际部署中，智能体环境（模型、工具、提示）不断演化，专家规则的“静态性”导致性能逐渐漂移。同时，用户运行时产生的“安全/不安全”标注（例如批准或拒绝边界工具调用）恰好可作为更新规则的信号。需要一种方法自动从这些标注中演化出可解释、高精确率且高召回率的规则。

## 2. 方法论：核心思想、关键技术细节
### 核心思想
- 将安全规则演化建模为 **CEGIS（反例引导的归纳合成）** 问题：用户标注的安全/不安全轨迹充当反例，规则集的误报（FP）和漏报（FN）指明修订方向。
- 利用 **ILP（归纳逻辑编程）** 高效识别能最好地区分反例的谓词（predicate），从而大幅剪枝规则编辑的指数级搜索空间。

### 关键技术细节
1. **输入**：初始规则集 \(R_0\)（专家规则）、标注轨迹集 \(D_L\)、谓词库 \(P\)（代码域 16-24 个谓词，具身域 16-24 个）。
2. **迭代循环**（算法 1）：
   - **阶段 1：反例挖掘** – 评估当前规则在 \(D_L\) 上的性能，收集 FP（误报）和 FN（漏报）轨迹。
   - **阶段 2：ILP 谓词学习** – 将 FN 作为正例、FP 作为负例，用 ILASP 求解器学习一个区分布尔假设 \(H\)（形如若干个合取分支），输出排名靠前的区分性谓词及其建议操作方向（收紧或放宽）。
   - **阶段 3：候选生成与选择** – 基于建议生成候选规则编辑（单步编辑、混合修复链、故障驱动析取分支）；对每个候选评估得分 \(S = (\text{F1}, \text{Prec}, -U, -|R'|)\)（字典序优先 F1，再精确率，再未解决反例数，最后规则复杂度），保留最佳候选。
   - **阶段 4：验证与收敛** – 若最佳候选得分优于当前规则则更新，否则终止；同时使用软多样性压力避免重复提议已存在的谓词。
3. **编辑算子**（表 2）：
   - AddConjunct（添加合取谓词）、AddException（添加否定例外）、Relax（删除一个文字）、AddDisjunct（在规则集层面添加析取分支）。
4. **谓词库**：每个谓词为布尔函数，判断事件属性（如 `destructive_os_inst`、`near_danger_zone`）。规则在事件级触发，分类在轨迹级（任何事件触发则判为不安全）。

## 6. 主要结论与发现
- **有效性**：AutoSpec 在两个域上达到最高 F1（代码 0.980，具身 0.933），比专家规则基线提升 40.4% 和 89.2%，误报降低高达 94%。
- **搜索效率**：ILP 引导比启发式 CEGIS 提升 F1 达 4.8 倍（代码域 0.980 vs 0.203），且收敛仅需 4-5 次迭代；总耗时 <21 秒。
- **泛化能力**：学习到的规则在未见新任务上仅产生微小性能下降（代码域 F1 从 0.973 降至 0.957，差距 1.6%；具身域从 0.889 降至 0.842，差距 4.7%）。
- **可解释性与信任**：用户研究中 AutoSpec 规则在可解释性（4.97 vs 3.45）、可操作性（4.83 vs 2.73）、信任度（4.80 vs 2.77）上均显著优于 LLM 分类器（p < 0.01），从业者更喜欢 AutoSpec（84.2% 的对比）。
- **消融发现**：ILP 引导是必不可少的（去除后 F1 从 0.98 跌至 0.20）；多步编辑和故障驱动分支显著改进效果或加速收敛。

## 7. 优点
1. **概念创新**：首次将安全规则维护形式化为 ILP 指导的 CEGIS 问题，将用户标注天然视为反例，利用 ILP 的符号学习能力高效剪枝搜索空间。
2. **技术实用**：生成可解释、可审计的符号规则，直接部署且零推理开销；能与现有防御深度堆叠（图 2）并存，不替代其他层。
3. **实验全面**：两个不同领域（代码执行和具身智能体）、多种对比基线、消融、泛化、用户研究，验证方法有效性和可解释性。
4. **低资源需求**：仅需普通笔记本 CPU/内存即可运行（无 GPU 依赖），适合持续异步维护。
5. **泛化与收敛**：在 4-5 次迭代内快速收敛，规则泛化到新场景仅微小损失。

## 8. 不足与局限
1. **领域覆盖有限**：仅测试了代码执行和具身代理两个领域，未涵盖 Web 代理、数据库代理等更多应用场景。泛化性需进一步验证。
2. **数据集规模较小**：两个域共 291 条轨迹，尤其具身域仅 100 条。虽然来自标准基准，但真实环境数据可能更复杂、噪声更大。
3. **依赖手工谓词库**：谓词库由专家设计构建，其质量直接影响 AutoSpec 所能表达的规则范围。未提供谓词库自动扩展机制。
4. **无全局最优性保证**：作者承认该问题为 NP-hard，算法仅保证局部最优和单调提升。在复杂场景下可能陷入次优解。
5. **标签假设**：假设用户/标注者的安全/不安全标签完全准确，未考虑标注噪声。实际中标注可能不一致或存在错误。
6. **时序与状态处理有限**：谓词仅评估单事件及局部上下文，未原生支持时序逻辑（如 LTL）或跨事件状态依赖。作者提及可扩展但未实现。
7. **ILP 求解器局限性**：ILASP 求解带超时限制（120s），若谓词库极大或反例复杂可能超时影响质量。但当前实验中未出现此问题。
8. **编辑操作简洁性**：仅支持四种基本编辑算子，可能无法表达复杂规则结构调整（如规则拆分、合并等）。但附录未提供未能覆盖的编辑需求。
