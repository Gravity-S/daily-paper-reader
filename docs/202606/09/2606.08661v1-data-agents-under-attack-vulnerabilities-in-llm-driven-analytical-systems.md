## 摘要
数据代理将LLM驱动的推理与关系数据访问、可执行分析工具及多步工作流编排相结合，使其在企业分析中日益核心。这种集成在数据资源、数据库执行和代理推理方面引入了新的安全漏洞，将数据库安全与通用LLM代理安全中的问题重新组合成两类工作各自未能单独捕获的故障模式。为填补这一空白，我们对数据代理进行了系统性安全研究。我们的贡献有三方面。首先，我们开发了一个分层漏洞框架，识别了跨解释层、执行层和策略层的八种数据代理特有风险。其次，我们引入了一种按攻击者目标、策略和技术组织的攻击分类法，涵盖三种目标、七种策略和十四种技术，并将其与基于真实数据库模式的LLM驱动载荷生成流水线配对。第三，我们在六个系统上评估了这些攻击，包括四个开源数据代理和两个生产级云分析服务。我们的实验揭示了当前系统中存在的重大安全漏洞，并得出了四项关键发现。

## Abstract
Data agents integrate LLM-driven reasoning with relational data access, executable analytical tools, and multi-step workflow orchestration, making them increasingly central to enterprise analytics. This integration introduces new security vulnerabilities across data resources, database execution, and agent reasoning, recombining concerns from database security and general-purpose LLM-agent security into failure modes that neither line of work captures on its own. To address this gap, we present a systematic security study of data agents. Our contributions are threefold. First, we develop a layered vulnerability framework that identifies eight data agent-specific risks across interpretation, execution, and policy layers. Second, we introduce an attack taxonomy organized by adversary goal, tactic, and technique, covering three goals, seven tactics, and fourteen techniques, and pair it with an LLM-driven payload generation pipeline grounded in real database schemas. Third, we evaluate these attacks on six systems, including four open-source data agents and two production cloud analytics services. Our experiments reveal substantial security vulnerabilities across current systems and yield four key takeaways.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：构建一个分层漏洞框架，识别数据代理特有的安全风险；基于该框架设计攻击分类法；并利用LLM自动生成与真实数据库模式对齐的攻击载荷，在多个系统上实证评估。
- **关键技术细节**：
  - **分层漏洞框架**：将数据代理抽象为元组 Σ = ⟨(M, S), T, D, Π, E⟩，从三个层面分析漏洞：
    - *解释层（Interpretation Layer）*：输入（用户提示、数据资产、系统状态）转化为分析意图时的风险（V1:隐式信任偏差；V2:缺乏数据源验证）。
    - *执行层（Execution Layer）*：意图转化为工具调用（SQL、Python等）时的风险（V3:不受控制的查询成本；V4:跨引擎语义不一致；V5:无界多步查询链）。
    - *策略层（Policy Layer）*：安全与操作策略执行时的风险（V6:上下文压力下策略遗忘；V7:过度权限的数据库连接；V8:缺乏组合泄露控制）。
  - **攻击分类法**：按攻击者目标（Hijack劫持、Mislead误导、Drain消耗）组织，细分为7种策略、14种具体技术（如T1.1查询注入、T5.1数据类型偏差利用、T7.1低价值分支扩展等）。
  - **LLM驱动载荷生成管线**：三步流程——①将攻击分类法存入知识库；②以目标技术为查询，用LLM生成技术特定的攻击模板；③从目标数据库目录中抽取表、列、数据类型，将模板实例化为引用真实字段的载荷。每种技术生成5个机制不同的模板，并在5个企业数据库上实例化，共25个载荷，总计350个载荷（14个技术×25）。
- **公式/算法流程（文字说明）**：无复杂公式，但定义了三个量化的严重性指标：
  - **Blast Radius (BR)**：劫持攻击中未授权效果的传播层级（0=阻塞，1=单次响应，2=会话状态，3=容器范围，4=宿主机/网络）。
  - **Relative Error (RE)**：误导攻击输出与攻击者目标值的相对误差，RE = |v̂ - v*| / v* × 100%。
  - **Resource Amplification Ratio (RAR)**：消耗攻击的资源放大倍数，RAR_m = ĉ_m / c_ref_m，分别在token和时间两个维度测量。
  - **Attack Success Rate (ASR)**：基于阈值的二值成功率：劫持：BR ≥ 1；误导：RE ≤ 10%；消耗：整体RAR ≥ 5。

## 6. 论文的主要结论与发现

- **每个系统都广泛暴露**：6个系统中每个至少对5个漏洞易被攻击，且各漏洞均出现在至少3个系统上，说明弱点非单个系统实现特有。
- **隐式信任偏差（V1）最为突出**：数据代理在证据冲突时多依靠隐式偏好而非来源验证，是最高成功率的漏洞。
- **资源消耗是主要失败模式**：消耗攻击（Drain）无需触发显式策略违反，即可通过合法工作负载消耗大量token或时间。
- **数据泄露可通过多步分析实现**：多个单独合法的查询组合后可暴露受保护记录，泄露应被视为会话级属性。
- **闭源系统可抵抗部分劫持**（Databricks尤其优秀），但对误导证据和资源消耗仍脆弱；更换基座LLM会重新分布风险而非消除风险。
