## 摘要
智能体技能通过特定任务的流程、脚本和参考资料扩展了语言模型智能体的能力，但其所针对的任务和环境在不断变化。现有方法在有限运行中改进技能，并仅保留最终产物，丢弃了后续智能体理解先前修订、评估和拒绝备选方案所需的决策历史。我们提出SkillHone，一个基于持久决策历史实现智能体技能持续进化的框架。SkillHone将技能修订与提供实践反馈的评估侧证据配对，记录诊断、修订、证据和结果的结构化历史。角色分离的子智能体在实践探针上运行候选技能，并生成经过编辑的报告，根据先前的决策提出修订，从而无需重新发现过去的原理即可实现跨会话的优化。我们在原始开放网页环境中对深度研究基准评估了SkillHone，该环境中智能体无需集成搜索栈，必须通过便携式技能组织检索。我们将结果与基于商业检索服务的深度研究智能体进行比较。使用Qwen3.6-35B-A3B作为评估时骨干网络，生成的技能在GAIA上优于深度研究智能体15.8分，在WebWalkerQA-EN上优于3.2分，同时也超越了先前的技能进化方法。

## Abstract
Agent skills extend language-model agents with task-specific procedures, scripts, and references, but the tasks and environments they target continually change. Existing methods improve skills in bounded runs and retain only the final artifact, discarding the decision history that later agents need to interpret prior revisions, evaluations, and rejected alternatives. We introduce SkillHone, a harness for continual agent skill evolution grounded in persistent decision history. SkillHone pairs skill revisions with evaluation-side evidence that supplies practice feedback, recording structured histories of diagnoses, revisions, evidence, and outcomes. Role-separated subagents run candidate skills on practice probes with redacted reporting and propose revisions informed by prior decisions, enabling cross-session refinement without rediscovering past rationale. We evaluate SkillHone on deep-research benchmarks in a raw open-web setting, where agents are not given an integrated search stack and must organize retrieval through portable skills. We compare against a deep-research agent backed by commercial retrieval services. With Qwen3.6-35B-A3B as the evaluation-time backbone, the resulting skills outperform the deep-research agent by 15.8 points on GAIA and 3.2 points on WebWalkerQA-EN, while also exceeding prior skill-evolution methods.

## 1. 核心问题与整体含义（研究动机与背景）

- **研究动机**：现有智能体技能进化方法（如 Skill-Creator、Hermes-SE）仅在有限运行内改进技能，并只保留最终产物，**丢弃了决策历史**（诊断、修订理由、评估证据、被拒绝的备选方案等）。后续智能体继承最新技能，却无法理解为何这样修改、为何拒绝其他方案，导致跨会话优化时需要重新发现过往逻辑，甚至重复旧修复或引入回归。
- **整体含义**：技能（如搜索、提取、验证流程）是 LLM 智能体专长化的核心单元，但 API、任务分布、故障模式持续变化，技能必须能长期维护。论文提出 **SkillHone**，通过持久决策历史（persistent decision history）让技能进化成为持续的维护过程，而不是一次性优化的产物。

## 2. 方法论：核心思想、关键技术细节、算法流程

- **核心思想**：将技能演进视为持续过程，记录每个开发步骤的决策记录 `h_t = (q_t, r_t, e_t, o_t)`，其中 `q_t` 为诊断，`r_t` 为候选修订，`e_t` 为经过编辑的评估证据，`o_t` 为结果（接受/拒绝/修订/延迟）。累积历史 `H_t = {h_1,...,h_t}` 供后续优化子智能体检索，避免重复发现。
- **关键技术细节**：
  - **角色分离的调度架构**：SkillHone 由运行时调度器 `D`、优化子智能体团队 `T_opt` 和评估子智能体团队 `T_eval` 组成。优化子智能体可读写技能仓库（`SKILL.md`、脚本、引用等），但不能访问未编辑的探测目标、验证器或执行轨迹；评估子智能体可读取探测和轨迹，但不能写入技能仓库。两者通过编辑后的报告通信。
  - **持久决策历史**：每个步骤的决策记录包含诊断、候选修订、编辑后的评估证据和结果。该历史作为跨会话的“过程记忆”，使后续子智能体能判断故障是否是新的、某个修复是否已尝试过、某个备选方案为何被拒绝。
  - **实践反馈**：评估仓库存储探测任务实例、oracle 目标、验证器、轨迹等。评估子智能体运行当前技能并输出编辑后的问题报告（仅汇总故障模式、聚合结果、诊断假设），优化侧无法看到未编辑的目标或轨迹，防止过拟合。
  - **算法流程（Algorithm 1）**：
    1. 初始化技能 `S_0`、技能仓库 `R_s`、评估仓库 `R_e`、决策历史 `H_0 = ∅`。
    2. 循环 `t = 0` 到 `T-1`：
       - 评估调度：运行 `A(·; S_t)` 在 `R_e` 上，输出编辑后报告 `\tilde{E}_t`。
       - 优化调度：检索 `H_t`，提出诊断 `q_t` 和候选修订 `r_t`。
       - 评估调度：返回针对 `r_t` 的编辑后证据 `e_t`。
       - 决定结果 `o_t`（ACCEPT / REVISE / REJECT / DEFER）。
       - 若接受则更新技能 `S_{t+1} = APPLY(S_t, r_t)`，否则保持不变。
       - 更新历史 `H_{t+1} = H_t ∪ {(q_t, r_t, e_t, o_t)}`。
    3. 返回最终技能 `S_T` 和历史 `H_T`。
- **与现有方法的区别**：Skill-Creator 仅生成技能产物；Hermes-SE 使用 GEPA 对整个提示进行变异并基于标量信号接受/拒绝，缺乏结构化历史。SkillHone 的决策记录链接了问题、修订、证据和结果，使后续子智能体能做**有针对性回退**（保留有用编辑，只修复回归部分）。
