## 摘要
安全配置错误仍然是导致操作系统级别被入侵的主要原因，而手动使系统符合国防信息系统局（DISA）安全技术实施指南（STIG）等标准是一个繁琐且昂贵的过程。现有的合规自动化工具可以减轻部分负担，但它们依赖于静态的、预编写的纠正措施。在本文中，我们介绍了SHIELDS，一个多智能体系统，它利用大型语言模型（LLMs）将操作系统加固视为一个迭代的、基于反馈的过程。SHIELDS不是应用固定的修复措施，而是持续提出修复方案，并根据目标系统执行和验证扫描的反馈进行优化。我们使用六种当代LLMs（参数规模从20B到400B）在多个虚拟机配置上评估了该系统，发现SHIELDS成功修复了多达73%的扫描发现项。我们的结果还表明，在这种设置中，成功与否更多取决于有效的工具使用和信息收集，而不是模型大小（参数数量），这为在计算资源有限或安全隐私需求驱动使用本地模型的环境中减轻安全合规负担开辟了一条实用路径。

## Abstract
Security misconfigurations remain a leading cause of OS-level compromise, and manually keeping systems compliant with standards like Defense Information Systems Agency (DISA) Security Technical Implementation Guides (STIGs) is a tedious and expensive process. Existing compliance automation tools can reduce some of this burden, but they depend on static, pre-written corrective actions. In this paper, we introduce SHIELDS, a multi-agent system that uses large language models (LLMs) to approach OS hardening as an iterative, feedback-driven process. Instead of applying fixed remediations, SHIELDS continuously proposes fixes and refines them based on feedback from target system execution and validation scans. We evaluate the system across multiple virtual machine configurations using six contemporary LLMs ranging from 20B to 400B parameters, and find that SHIELDS successfully remediates up to 73% of scan findings. Our results also suggest that success in this setting depends less on model size (parameter count) than on effective tool use and information gathering, paving a practical path toward reducing the burden of security compliance in environments where compute is limited or security and privacy needs drive local model use.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：操作系统（OS）安全配置错误是导致系统被攻破的主要原因之一，而手动遵循DISA STIG等安全合规标准耗时且昂贵（每台服务器每年约16小时，一个组织可能需要20名全职工程师）。
- **现有自动化工具的局限**：现有工具（如OpenSCAP、ComplianceAsCode、Ansible Lockdown、PowerSTIG）依赖预定义的静态修复脚本，缺乏自适应能力。当修复失败或遇到新问题时，无法基于反馈迭代调整策略。
- **研究目标**：探索利用大语言模型（LLMs）实现自主、迭代的OS加固合规自动化，以减轻安全运维负担。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：将OS加固视为一个**反馈驱动的闭环过程**（扫描→分析→修复→验证→再扫描），而非一次性的静态修复。系统持续提出修复方案，并根据目标系统执行结果和扫描反馈进行优化。
- **多智能体架构**（SHIELDS）分为四个专用智能体，分工协作：
  - **Triage Agent（分诊智能体）**：预处理每个扫描发现项，决定是“安全可修复”、“需人工审查”还是“过于危险不可修”（例如文件系统/分区规则默认标记为危险）。
  - **Remedy Agent（修复智能体）**：核心组件，拥有三个工具（`run_cmd`、`read_file`、`write_file`），可迭代提出修复方案。默认每个发现项最多3次尝试。
  - **Review Agent（审查智能体）**：评估单个修复方案的质量，产生安全影响评分（1-10）、最优性评估和潜在问题列表。若拒绝则返回Remedy Agent重新提案。
  - **QA Agent（质量保证智能体）**：评估系统级安全性（如关键服务是否仍在运行、SSH是否可用等），输出“批准/回滚/调查”建议。若拒绝则返回Remedy Agent。
- **工作流程**（Phase 1→Phase 2→Phase 3→Phase 4）：
  1. **Phase 1 扫描**：远程连接目标主机，执行全面的OpenSCAP扫描，获取所有发现项。
  2. **Phase 2 分诊**：Triage Agent为每个发现项做出决策；安全可修的进入修复循环，否则被丢弃/标记。
  3. **Phase 3 修复循环**（最多3次）：Remedy Agent提出修复方案 → Review Agent和QA Agent双重审批（两者都批准才执行） → 执行修复 → 对该发现项进行独立OpenSCAP重扫描验证。若任何环节失败，反馈追加到上下文，进行下一次尝试。
  4. **Phase 4 聚合**：所有发现项处理完毕后，统计成功率，生成Ansible playbook（来自成功修复的记录）。
- **关键技术细节**：
  - 修复失败时，将失败反馈（包括Review/QA Agent的拒绝理由、命令执行错误等）追加到下一轮提示的上下文中，实现迭代优化。
  - 限制每次执行一个命令，避免管道/分号链接；修改前先读取文件；使用`dnf`（RHEL系）、`systemctl`等。
  - 默认3轮尝试，前期实验表明第4、5轮收益递减。

## 6. 主要结论与发现
- **修复效果**：SHIELDS成功修复了多达73%的扫描发现项（openai/gpt-oss-120b + 大型VM配置）。
- **模型规模非关键因素**：最大模型（trinity-large 400B）表现并非最好，甚至逊于小20倍的gpt-oss-20b。表现最佳的为120B规模的mercury-2和gpt-oss-120b，而同为120B的nemotron-3表现较差。表明**工具使用能力和信息收集能力比模型参数规模更重要**。
- **迭代模式**：大部分修复发生在第一轮，部分模型在第二、三轮仍有显著修复（如mercury-2在小型VM上三轮修复量大致相等）。第四、五轮收益递减，3轮是合理的折衷。
- **VM规模与修复率正相关**（对mercury-2和gpt-oss系列）：推测大型VM包含更多简单机械性修复（如文件权限、包升级），更易被模型解决。

## 7. 优点

- **多智能体设计**：职责分离（分诊、修复、审查、QA）降低了系统复杂度，提高了安全性和可解释性；通过双重审批（Review + QA）降低误操作风险。
- **反馈驱动自适应**：修复失败时，失败原因（包括拒绝理由、执行错误）被反馈到下一轮提示，实现反思和优化。
