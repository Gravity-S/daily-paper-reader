## 摘要
人工智能编码智能体（如 Claude Code 和 Gemini CLI）越来越多地通过第三方技能扩展自身：这些技能是打包了自然语言指令、可执行脚本和工具权限的 Markdown 包。由于技能同时包含代码和面向智能体的指令，它引入了一种供应链依赖，其风险既非纯代码也非纯提示。检测工具从未针对跨越这一混合空间的经验证真实数据进行评估，导致其有效性未知且仅依赖野外环境的评估存在偏差。我们提出了 MalSkillBench，这是首个运行时验证的恶意智能体技能基准：包含 3,944 个恶意技能，按照 108 个单元格的三维分类法进行标注。其中，3,214 个来自闭环的“生成-验证-反馈”流程，仅接受在 Docker 沙箱中通过系统调用监控和大语言模型裁判触发恶意行为的样本；我们额外添加了 703 个野外样本和 4,000 个匹配的良性技能。我们的测量结果一致：代码注入的验证成功率达到 94.5%，但提示注入仅为 75.8%，这种脆弱性也使其后续难以被检测；野外样本范围狭窄，主要来自一次加密货币窃取活动（86.6% 为单一行为，81% 来自两个账户），但存在少量架构新颖的攻击，针对智能体控制平面；最强的技能专用检测器在代码注入上召回率达到 98.4%，但在提示注入和智能体控制攻击上性能崩溃，仅使用野外样本进行评分会使排名波动高达 66 个召回率点；供应链扫描器和提示注入防御各自仅能看到技能的一半，任何组合都无法恢复代码-指令关系。因此，检测恶意技能需要联合推理任务意图、代码和指令。我们发布了数据集、流程、基线和结果。

## Abstract
AI coding agents such as Claude Code and Gemini CLI increasingly extend themselves with third-party skills: markdown packages bundling natural-language instructions, executable scripts, and tool permissions. Because a skill is at once code and agent-facing instruction, it introduces a supply chain dependency whose risk is neither pure code nor pure prompt. Detection tools have never been measured against verified ground truth spanning this hybrid space, leaving their effectiveness unknown and wild-only evaluations biased.   We present MalSkillBench, the first runtime-verified benchmark of malicious agent skills: 3,944 malicious skills labeled along a three-dimensional taxonomy of 108 cells. Of these, 3,214 come from a closed-loop Generate-Verify-Feedback pipeline admitting only samples whose malicious behavior fires inside a Docker sandbox under system-call monitoring and an LLM judge; we add 703 in-the-wild and 4,000 matched benign skills. Our measurements are consistent: code injection reaches 94.5% verification yield but prompt injection only 75.8%, the same fragility that later makes it hard to detect; the wild sample is narrow, dominated by one cryptocurrency-theft campaign (86.6% one behavior, 81% from two accounts) with a small but architecturally new tail attacking the agent control plane; the strongest skill-specific detector reaches 98.4% recall on code injection yet collapses on prompt-injection and agent-control attacks, and wild-only scoring swings the ranking by up to 66 recall points; supply-chain scanners and prompt-injection defenses each see only half of a skill, and no combination recovers the code-instruction relationship. Detecting malicious skills therefore requires reasoning jointly over task intent, code, and instructions. We release the dataset, pipeline, baselines, and results.

#### 2. 方法论

*   **核心思想**：通过一个**闭环的“生成-验证-反馈（Generate-Verify-Feedback）”** 自动化管道，系统地生产经过*运行时验证*的恶意技能样本，以覆盖整个攻击空间。检测恶意技能需要**联合推理**技能的任务意图、代码和指令，而非单独检查任何一个部分。

*   **关键技术细节**：
    1.  **攻击分类法（Taxonomy）**：定义了三维攻击空间，共 108 个单元格。
        *   **攻击向量 (v)**：代码注入 (CI)、提示注入 (PI)、混合攻击 (MIXED)。
        *   **恶意行为 (b)**：涵盖 15 种行为，包括代码级行为（B1-B9，如数据泄露）和智能体控制级行为（B10-B15，如角色劫持）。
        *   **插入策略 (s)**：指定恶意负载的放置方式（如新建脚本文件、完全伪装等）。
    2.  **生成代理（Generation Agent）**：
        *   **模式迁移**：从外部知识库（如 PyPI 恶意包数据集 `IntelliGraph` 和提示注入语料库）迁移真实攻击模式到技能格式。
        *   **模板选择**：从 3,458 个良性技能池中随机选取一个作为合法外表（伪装）。
        *   **样本合成**：使用**去抑制化的 LLM**（Qwen3.5-35B-abliterated）根据分类法坐标、检索到的攻击示例和模板来生成候选恶意技能。
    3.  **验证代理（Verification Agent）**：
        *   **沙箱执行**：在 Docker 容器中，由真实的编码智能体（OpenCode）加载候选技能。
        *   **双层验证**：
            *   **第一层（证据匹配）**：基于规则的确定性匹配，扫描系统调用痕迹（`strace`, `inotifywait`）以查找声明的攻击指标（IOCs），如目标域名、文件路径。
            *   **第二层（行为验证）**：LLM 判断器（GPT-5.4-mini）对痕迹进行语义分析，确认行为成功与否。
        *   **闭环反馈**：验证失败的样本会附带反馈信息返回给生成代理进行修改并重新验证，最多尝试 3 次。只有通过验证的样本才被纳入基准。

    *   **良性技能**：4,000 个，来自公开平台，用于评估假阳性。

*   **对比方法**：共评估了 12 个检测工具，分为三类：
    *   **技能专用检测器 (RQ3)**：9 个工具（如 Cisco Skill Scanner, Sentry Skill Scanner, AI-Infra-Guard, VirusTotal 等），涵盖静态规则、静态+LLM、纯LLM等范式。

    *   **提示注入防御 (RQ4-PI)**：6 个工具（如 Llama Guard 3, NeMo Guardrails），针对提示注入设计的防御。

#### 6. 论文的主要结论与发现

1.  **攻击可实现性不均**：代码注入 (CI) 的可实现性很高（94.5%），而提示注入 (PI) 较低（75.8%），这反映了 PI 固有的脆弱性。
2.  **真实世界攻击面狭窄**：703 个野生样本中，**86.3%** 是简单的依赖冒充攻击；**86.6%** 属于恶意软件投递（B4）；**81%** 来自仅两个攻击者账户。虽然存在少量针对智能体控制平面的创新攻击，但野生数据无法代表全部威胁。

4.  **工具迁移能力差**：现有的供应链扫描器和提示注入防御均只能检测技能的一半，无法覆盖混合攻击面。简单的组合（并集、交集）无法弥补这一根本性缺陷，因为这需要理解**代码与指令之间的关系**。

#### 7. 优点

1.  **方法论的创新性**：**运行时验证**是最大的亮点。这保证了生成的样本是真正有效、可触发的攻击，而不是理论上存在但实际无用的代码或提示，为后续评估提供了坚实可信的“金标准”。
2.  **覆盖面的系统性**：通过定义三维分类法和从外部知识库迁移模式，基准系统性地覆盖了现有数据集忽略的广大攻击空间（如 PI 和 IOA），解决了野生数据固有的偏见问题。
3.  **研究的综合性**：不仅提出了基准，还利用它完成了从攻击构建、真实世界分析到检测器评估和工具迁移的**全链条研究**，提供了非常全面和深刻的见解。
4.  **开源与可复现性**：承诺开源所有数据集、代码、基线和实验结果，有助于社区在此基础上进行更深入的研究和公平比较。

#### 8. 不足与局限

1.  **生成模型的“现实性”**：生成的恶意技能虽然覆盖了攻击空间，但其结构和具体实现是否与真实世界中的高级攻击者完全一致？可能存在**风格或模式上的趋同性**（LLM 输出偏差），导致某些检测器对这种特定模式过拟合。
2.  **运行时环境的局限**：只有在特定智能体（OpenCode）和沙箱（Docker）下验证通过的样本才被纳入。不同智能体或执行环境可能导致相同技能的行为不同，其**泛化性有待验证**。
3.  **评估深度的局限**：对检测器的评估主要集中在整体召回/精确率上。虽然分析了不同子集下的性能，但未深入剖析**检测器的具体决策逻辑**（如哪些规则或特征触发了警报，哪些没有）。
4.  **Wild样本的潜在偏差**：Wild样本主要来源于 ClawHub，可能**不代表其他平台或未来更复杂的攻击**。论文也指出其狭窄性，因此 Wild 部分可能更像一个Campaign样本集。
5.  **工具自评与互评未做**：论文未对基准本身的“难度”做分析（如不同分类法单元格的样本对检测器难度的差异）。也未评估各检测器之间的**互补性**，除了简单的逻辑组合外。
