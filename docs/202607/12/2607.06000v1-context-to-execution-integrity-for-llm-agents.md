## 摘要
语言模型代理读取攻击者可写的上下文来解决问题。工具执行需要针对受保护的接收器字段、接收器解释的有效载荷以及调用事件进行单独的授权检查。上下文到执行完整性（CXI）是针对此场景的执行边界系统。策略标记受保护的接收器字段，类型化发布将狭窄的验证值从可写上下文传递到特定目标，不透明数据槽将证据作为数据存储，确定性门仅在字段授权、确切效果授权和调用授权都绑定到同一操作清单后才允许调用。我们在开放权重的字段投影运行、AgentDojo实时片段、代码代理确切效果基准、清单绑定账本故障、提案压力控制以及托管/API兼容性轨迹上评估了CXI。AgentDojo涵盖720个实时片段和1,739次LLM调用；代码代理基准涵盖400个仓库片段，具有确切效果授权和租约绑定执行，产生了231个安全任务完成，且未观察到字段、效果或调用逃逸。会计报告输出解析结果、授权结果和任务质量结果，以及准入完整性结果。在评估的接收器中，CXI仅在字段、效果和调用授权绑定到同一操作清单时才允许执行。

## Abstract
Language-model agents read attacker-writable context to solve tasks. Tool execution needs a separate authority check for protected sink fields, sink-interpreted payloads, and the invocation event. Context-to-Execution Integrity (CXI) is an execution-boundary system for this setting. Policies mark protected sink fields, typed releases carry narrow validated values from writable context to specific destinations, opaque data slots keep evidence as data, and a deterministic gate admits a call only after field authority, exact-effect authorization, and invocation authority all bind to the same action manifest.   We evaluate CXI on open-weight field-projection runs, AgentDojo live episodes, a code-agent exact-effect benchmark, manifest-bound ledger faults, proposal-pressure controls, and hosted/API compatibility traces. AgentDojo covers 720 live episodes and 1,739 LLM calls; the code-agent benchmark covers 400 repository episodes with exact-effect authorization and lease-bound execution, yielding 231 safe task completions and zero observed field, effect, or invocation escapes. The accounting reports parser outcomes, authorization outcomes, and task-quality outcomes together with the admission-integrity result. Across the evaluated sinks, CXI admits execution only when field, effect, and invocation authority bind to the same action manifest.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM代理在执行工具调用时，会读取攻击者可写的上下文（如邮件、网页、CI日志、Issue评论、检索文档、工具输出等），这些上下文可能被用于**权威洗白（authority laundering）**——原本仅应作为证据的值，却被用于选择、授权或参数化特权副作用。例如，CI日志中的失败文件名不应被用来调用 `prod_admin.run_sql`；Issue正文中的“approved”字符串不应直接标记审批状态。
- **整体含义**：现有防御（如结构输出约束、参数级溯源PACT、运行时策略ARGUS等）粒度不够或边界不全，缺少一个统一的**执行边界完整性**机制，在副作用发生前确保只有通过恰当授权渠道的值才能影响受保护字段、效果和调用事件。
- **研究动机**：用户目标、策略等可信状态与攻击者可写上下文共存于模型上下文中，代理可能将恶意文本当作指令执行。需要一种系统化的方法，在“上下文”到“执行”的最后一步实施完整性检查。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
**上下文到执行完整性（CXI）**：在副作用发生前，受保护接收器字段、接收器解释的效果以及调用事件**必须全部针对同一个规范动作清单（canonical action manifest）**获得授权，否则拒绝执行。

### 关键技术细节

1. **状态分类**：将边界状态分为五类：
   - **可信状态（T）**：用户目标、策略、注册表、审批等。
   - **可写上下文（W）**：攻击者可写或可影响输入。
   - **类型化发布（D）**：从W到特定目标字段的验证后的窄值（如文件路径、审批引用）。
   - **不透明数据（O）**：无当前边界权限的W副本（可保留为证据，但不授权）。
   - **受保护接收器字段（X）**：选择、授权或参数化副作用的字段。

2. **准入流水线**（六步）：
   1. **规范化**候选动作为动作清单。
   2. **检查字段权限**：每个受保护字段必须仅有T或D来源授权，W不可直接影响X。
   3. **检查效果权限**：对于接收器解释的有效载荷（补丁、SQL、命令等），使用确切效果授权（Exact-Effect Authorization）基于可信快照、主体、适配器等计算实际预期效果，并绑定到清单。
   4. **检查调用权限**：消耗一个清单绑定的调用能力（Invocation Capability）以授权调用事件（包括重试、批处理、调度等）。
   5. **获得执行租约**：线性化账本消耗能力，返回租约。
   6. **通过中介接收器执行**：仅租约对应动作可执行。

3. **去分类器（Declassifiers）**：可信代码，将W中的候选值经过验证后铸成窄类型发布（D），作用域局限于单个目标字段。

4. **不透明数据槽**：保留W副本但不赋予当前边界权限，若后续被其他代理或工具读取，则重新作为W进入边界。

5. **策略模型**：失败-关闭；未知字段、未分类字段、过期版本、无效证据模式等均拒绝。受保护字段包括工具、操作、审批、接收者、路径等；效果性有效载荷（补丁、SQL等）需要适配器级授权。

6. **证据路径**：
   - **开放权重路径**：运行时约束受保护字段生成，通过后端证据记录（Backend Evidence Record）证明模型在生成该字段时未看到可写上下文的限定部分。
   - **托管/API路径**：由于无法观察提供者内部（注意力掩码、KV缓存），使用场域本地构造证据（field-local construction evidence），记录提取器、写入器的可见性。

## 7. 优点

- **设计精细**：区分字段、效果、调用三层授权，并用统一清单绑定，防止跨字段或跨调用滥用。
- **证据路径灵活**：支持开放权重（后端证据）和托管API（场域本地构造证据）两种模式，覆盖不同部署场景。
- **报告透明**：将解析失败、授权失败、任务质量失败分开记录，不模糊安全指标。
- **失败-关闭**：任何权限缺失、证据不匹配、策略过期等情况均拒绝执行，避免静默失败。
- **实用导向**：提供完整的中介性检查清单，明确部署时需要逐个接收器编写策略、验证器、适配器等。

## 8. 不足与局限

- **依赖完整中介**：如果存在未通过共享门控的副作用路径（如直接文件API、助手客户端），CXI无法保护。
- **依赖正确策略和验证器**：策略字段分类、作用域、验证器强度需人工保证，错误策略可能导致漏洞。
- **托管/API路径证据有限**：无法验证提供者内部的注意力掩码、KV缓存等细节，仅能依赖外部观察。
- **不保证任务质量**：即使授权通过，任务可能因模型输出错误、验证器不完备等失败。
- **不保证外部精确一次交付**：账本可以防止重复准入，但外部副作用去重需要适配器或基础设施支持。
- **残留风险清单**：包括受损主机（可伪造溯源）、弱验证器、未中介的助手、不透明数据槽重新进入等。
- **实验覆盖**：虽然丰富，但主要集中于代码代理和AgentDojo场景，其他领域（如生产力工具、记忆系统）的测试较少。
