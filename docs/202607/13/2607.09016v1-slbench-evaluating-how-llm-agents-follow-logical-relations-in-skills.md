## 摘要
技能扩展了LLM代理的可复用流程、工具和特定领域工作流，但其安全性取决于交互指令间依赖关系的解决。我们提出SkillLogic，一个分析技能文件中逻辑关系并从中构建可执行测试的框架。我们的分类法涵盖八种关系类型，包括约束有效动作的前提条件、限制允许动作执行方式的约束以及指定失败后恢复行为的备选方案。使用SkillLogic，我们扫描了超过5000个公共技能，发现70%包含至少一个逻辑关系。接着，我们从高置信度、高影响力和本地可测试的关系中构建了SLBench，一个包含86个案例的可执行基准。评估Codex和Claude Code在六个LLM骨干上的表现显示，不安全率高达70%，违规导致隐私泄露、不安全的配置更改和不完整的清理。人工审计将失败归因于代理能力差距和技能文本的低显著性。我们进一步表明，SLGuard（一种轻量级推理时辅助工具）在目标案例中减少了63%的违规行为。我们的结果将逻辑关系遵循确立为技能引导代理的一个独特可靠性挑战。

## Abstract
Agent skills extend LLM agents with reusable procedures, tools, and domain-specific workflows, but their safety depends on resolving dependencies among interacting instructions. We introduce SkillLogic, a framework for analyzing logical relations in skill files and constructing executable tests from them. Our taxonomy covers eight relation types, including preconditions that gate valid actions, constraints that limit how allowed actions may be performed, and fallbacks that specify recovery behavior after failure. Using SkillLogic, we scan over 5000 public skills and find that 70% contain at least one logical relation. We then construct SLBench, an 86-case executable benchmark from high-confidence, high-impact, and locally testable relations. Evaluating Codex and Claude Code across six LLM backbones shows unsafe rates up to 70%, with violations leading to privacy leaks, unsafe configuration changes, and incomplete cleanup. The human audit attributes failures to both agent capability gaps and low-salience skill text. We further show that SLGuard, a lightweight inference-time scaffold, reduces violations by 63% on targeted cases. Our results establish logical-relation following as a distinct reliability challenge for skill-guided agents.

## 1. 论文的核心问题与整体含义（研究动机和背景）

LLM 智能体（Agent）越来越依赖外部“技能”（skill）文件——这些是封装了领域知识、工作流和操作约束的可复用指令文件。技能文件通常由多个自然语言条款（clause）组成，条款之间形成逻辑关系（例如前提条件、后置条件、约束、回退等）。智能体如果只遵循了一个显眼的动作条款而忽略了其他制约性条款，可能会完成任务却导致不安全的后果（如隐私泄露、配置错误、未清理敏感文件等）。

现有基准测试（如工具使用、网页导航、安全政策遵从）没有系统性地评估智能体在技能文件内部解析和遵循多条款逻辑关系的能力。因此，本文的核心问题是：**LLM 智能体能否正确地遵循技能文件中条款间的逻辑关系？如何构建可执行的测试来暴露这类失败？**

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
提出 **SkillLogic** 框架，用于分析技能文件中的逻辑关系，并从中构建可执行的测试案例。核心是将技能文件建模为一组自然语言条款的集合，条款之间定义八种逻辑关系类型，然后基于这些关系生成可执行的基准测试。

### 关键技术细节
- **条款形式化**：每个条款表示为 tuple `ci = (ti, gi, ai, mi, oi)`，其中 ti 是源文本，gi 是适用条件，ai 是被支配的动作，mi 是模态（must/should/may/must not），oi 是对象或范围。
- **逻辑关系定义**：八种关系类型：前提条件（precondition）、后置条件（postcondition）、约束（constraint）、合取（conjunction）、回退（fallback）、异常（exception）、覆盖（override）、冲突（conflict）。每种关系有形式化定义和示例失败模式。
- **测试案例生成流程**：
  1. **逻辑关系分析**：对每个技能文件提取条款，选择主要工作流，识别其中所有逻辑关系，并标记可基准测试的关系。
  2. **测试案例构建**：对每个保留的关系，生成一个逼真的用户提示、一个受控的本地仓库（repository）、以及一个确定性评分器（grader）。评分器基于执行痕迹的产物（如文件状态、命令历史）而非自然语言响应来判断是否违规。
  3. **分级设计**：评分器采用“不安全优先”顺序：如果检测到任何违规信号则判为违规，只有没有违规时才检查安全信号。安全信号不足或混合证据则判为“无法确定”（inconclusive）。

## 6. 论文的主要结论与发现

- **违规率普遍很高**：所有评估的智能体在 SLBench 上都频繁失败，不安全率从 35% 到 70%。例如 Codex CLI + GPT-5.5 违规率达 70.2%。
- **关系类型影响失败模式**：前提条件、冲突和覆盖关系最难遵循；回退关系违规率较低。
- **失败原因不单一**：人类审计表明技能文本本身对细心人类读者是可理解的，但智能体仍然失败；清晰化技能修改可将违规减少一半，但仍有 5/12 案例持续失败，说明需要更强的机制。
- **SLGuard 有效减轻违规**：在 11 个原始违规案例上，SLGuard 将违规从 11 减至 4，安全结果从 0 增至 7。
- **核心论点**：逻辑关系遵循是技能引导智能体的一个**独特的可靠性挑战**，不同于单纯的指令遵循。
