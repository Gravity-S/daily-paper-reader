## 摘要
代理系统通过调用外部工具和维护持久化内存来增强能力。然而，这些外部依赖引入了新的攻击面。最近的工具与内存投毒攻击表明，恶意构造的工具描述符和被投毒的内存能够隐蔽地偏斜代理行为。这些威胁反映了一个更深层次的问题：代理用于规划和执行的上下文状态缺乏可验证的连续性。我们提出ElephantAgent，一种强制实施上下文状态连续性以防御上下文状态投毒的攻击的协议。受先前状态连续性机制（如Nimble）的启发，ElephantAgent将这种保护扩展到代理系统不断演化的上下文状态。我们将上下文状态定义为代理整个上下文中有限的、安全关键的子集（例如，工具状态和内存）。在处理每个查询之前，ElephantAgent重新计算本地上下文状态的摘要，并将其与最新授权的摘要进行验证。通过使用复制的可信硬件，ElephantAgent维护了一个可线性化的授权上下文状态转换账本，并检测带外状态篡改。为了处理带内语义滥用，ElephantAgent额外提供了历史可追溯性，支持有条件的事后审计并恢复到已知良好的先前状态。

## Abstract
Agentic systems enhance their capabilities by invoking external tools and maintaining persistent memory. However, these external dependencies introduce novel attack surfaces. Recent tool and memory poisoning attacks show that maliciously crafted tool descriptors and poisoned memory can covertly bias agent behavior. These threats reflect a deeper issue: the lack of verifiable continuity in the agent's contextual state for planning and execution. We present ElephantAgent, a protocol that enforces Contextual State Continuity to defend against contextual state poisoning. Inspired by prior state-continuity mechanisms (e.g., Nimble), ElephantAgent extends this protection to the evolving contextual state of agentic systems. We define the contextual state as the bounded, security-critical subset of the agent's entire context (e.g., tool state and memory). Before processing each query, ElephantAgent recomputes the digest of the local contextual state and verifies it against the latest authorized digest. Using replicated trusted hardware, ElephantAgent maintains a linearizable ledger of authorized contextual state transitions and detects out-of-band state tampering. To handle in-band semantic abuse, ElephantAgent additionally provides Historical Traceability, enabling conditional post-hoc audit and recovery to a known-good prior state.

# 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

代理系统通过调用外部工具和维护持久化内存来增强规划与执行能力。然而，这些外部依赖引入了新的攻击面：**工具投毒攻击**（如MCP “rug pull” 攻击）和**内存投毒攻击**（如操纵对话历史诱导恶意行为）可以隐蔽地偏斜代理的推理和行动。现有防御（如MindGuard、SecureMCP）主要依赖检测或阻断，但缺乏对代理上下文状态（工具描述符、长期指令、交互记录、产物等）连续性的可验证保护。论文指出，根本问题是代理的上下文状态（Contextual State）缺乏可验证的连续性，导致带外篡改（out-of-band tampering）和带内语义滥用（in-band semantic abuse）难以被可靠发现。为此，ElephantAgent 协议旨在通过绑定上下文状态到可验证且线性化的状态历史，强制**上下文状态连续性**，并额外提供**历史可追溯性**以支持事后审计和恢复。

## 2. 方法论

### 核心思想
- **上下文状态** $C_t = \langle T_t, M_t \rangle$，其中 $T_t$ 是工具状态（MCP工具描述符），$M_t$ 包含长期指令 $M_t^I$、交互记录 $M_t^R$、当前产物 $M_t^A$。上下文状态是代理上下文的有限、安全关键子集。
- 维护一个可线性化的、防篡改的授权状态转换账本，强制状态只能按顺序演化，任何带外修改（修改、回滚）都会被检测到。
- 对于带内语义滥用（如通过正常交互造成的恶意状态更新），提供历史可追溯性支持事后审计和恢复到已知良好状态。

### 关键技术细节
- **两个新增组件**：
  - **Context Guard (CG)**：运行在 TEE（Intel TDX）内，位于 MCP Host 中。每个查询前重新计算本地上下文状态摘要 $\hat{h} = H(\hat{C})$，并从 State Continuity Module (SCM) 获取最新授权摘要，验证两者一致。状态更新采用“先提交再最终化”（commit-before-finalize）流程。
  - **State Continuity Module (SCM)**：作为复制的 TEE 后端服务，维护每个账本 $\ell$ 的线性化、仅追加的授权摘要序列。核心接口是 `Commit`（追加新条目并返回签名收据）和 `GetLatestState`（返回最新条目签名证明）。
- **协议工作流**：
  1. **状态初始化**：通过 `ApprovePolicy`（原型采用 Trust On First Use）批准初始状态，计算摘要并锚定到 SCM，快照本地状态。
  2. **状态验证**（每个查询前）：CG 调用 SCM.GetLatestState 获取带随机数 $n_r$ 的签名证明，验证签名和非匹配，比对本地摘要。失败则中止。
  3. **状态更新**：查询处理后，CG 通过 `PreviewApply` 导出下一个摘要，调用 SCM.Commit 提交，验证收据，然后密封待定更新记录，最终化本地副作用，最后追加本地账本条目（包含摘要、收据、快照引用）。
- **历史可追溯性**：本地账本与快照共同支持审计（检查有序的转换历史和状态内容）和恢复（从已知良好步骤的快照恢复，并重新锚定到新账本）。

### 安全分析
- 基于五个假设：SCM账本正确性（线性化）、签名不可伪造、哈希抗碰撞、CG可信且不可绕过、已认证的SCM访问。证明了单步的完整性/新鲜性（引理1）和序列一致性（推论1），以及带内转换的授权与可审计性（引理2、推论2）。

## 3. 实验设计

### 数据集/场景
- 使用**三个代理工作流**，模拟真实任务：
  - **Email Triage**：加载邮件线程，生成结构化分类产物（摘要、行动项、回复草稿）。
  - **Code Bug Fix**：基于 QuixBugs 仓库，处理预定义 bug，执行读写、修复报告、补丁导出。
  - **Calendar Scheduling**：基于 CalDAVTester 日历数据，加载事件、计算空闲时段、生成调度产物。
- 每个任务的工具集是任务相关的（共实现19个MCP工具），使用特定提示词和结构化输出。

### Benchmark
- **基线方法**：**NoGuardAgent**——关闭 CG 和 SCM 的传统 MCP 工作流，运行在非TEE VM。
- **对比对象**：主要对比 ElephantAgent 与 NoGuardAgent 的整体时间开销。另外在消融研究中对比 **AttestedSrvAgent**——将 MCP Server 运行在 TDX VM 中并通过远程认证，但无状态连续性保护。

### 评价指标
- 初始化时间、数据加载时间、状态验证时间、任务处理时间（使用 API 服务器端计算时间）、更新时间、总时间、开销比。

## 4. 资源与算力

论文使用了阿里云 `ecs.g9i.xlarge` 实例（支持 Intel TDX）。具体配置：
- 三台 TDX 虚拟机（TDX1、TDX2、TDX3）：TDX1 运行 MCP Host（含 CG），TDX2 和 TDX3 运行复制的 SCM（共5个节点进程）。
- 两台非 TDX 虚拟机（NT1、NT2）：NT1 运行 MCP Server，NT2 运行 NoGuardAgent。
- 所有机器位于阿里云北京区域，通过内网连接。
- 代理规划使用外部 LLM API：**qwen-flash**（阿里云 Model Studio），作者特别选用延迟优化的 Qwen 模型以便观察状态维护开销。**未明确说明 GPU 数量或类型**，也未提及训练（本工作无需训练模型，仅对现有LLM API调用）。因此，**资源与算力未系统报告**。

## 5. 实验数量与充分性

- 每个任务执行 **50 次有效运行**，取平均时间。共三个任务，总计 **150 次一次实验**。
- 消融实验（AttestedMCP Server）同样三个任务各50次。
- 实验覆盖了初始化开销、单次查询分解开销、端到端开销，并给出了统计平均值。
- 作者指出由于是确定性验证防御，不报告检测率（因为带外篡改总是被拒绝）；实验重点在于性能成本。
- **充分性评价**：实验设计合理，任务有代表性，重复次数足够（50次），并测量了主要组成部分。但未进行大规模多变量实验（如不同模型、不同网络延迟、不同快照策略），也未测试极端负载或故障场景。实验相对简练但能支撑论文的核心论点——引入的开销很小。

## 6. 主要结论与发现

1. **ElephantAgent 的端到端开销很小**：三个任务的总时间开销比 NoGuardAgent 仅为 1.02× 到 1.04×。初始化开销较大（8.22×），但这是单次成本。
2. **单次状态验证和更新开销仅为毫秒级**：验证平均 4.23-4.33 ms，更新平均 5.25-5.39 ms，数据加载额外不足1 ms。
3. **消融实验显示认证 MCP Server 引入约 1.12-1.22× 的工具调用时间增加**，适用于高安全需求场景。
4. **结论**：ElephantAgent 以极小的性能代价提供了强上下文状态连续性保护，支持历史可追溯性，且是协议无关的可插拔方案。

## 7. 优点

- **首次将状态连续性保护系统性地扩展到代理系统**，形式化定义了上下文状态连续性和历史可追溯性两个安全属性。
- **验证而非检测**：通过密码学绑定线性化账本，带外篡改被确定性拒绝，不依赖统计分类器，避免了误报/漏报。
- **协议独立**：可应用于 MCP 或 CLI 等不同架构，增式部署。
- **实践性好**：利用现有 TDX TEE 和 Nimble 实现，只需要跑在 TEE VM 中，无需大型代码改造；实验证明开销极小。
- **架构清晰**：CG负责本地验证，SCM负责远端账本，职责分离，可伸缩。
- **安全分析严谨**：在明确假设下证明了两个安全性质。

## 8. 不足与局限

- **LLM通信通道安全未覆盖**：代理与外部LLM API之间的信道可能被篡改，不在上下文状态连续性保护范围内（论文提及可考虑本地模型+GPU机密计算）。
- **可用性受限于故障关闭行为**：带外篡改导致中止，恶意破坏可发起拒绝服务。历史追溯性要求本地账本/快照可用，但TEE封装不保证可用性（攻击者可删除）。
- **恢复范围有限**：恢复仅作用于保护的上下文状态子集，不包括完整运行时上下文；恢复不保证状态的新鲜性（可能恢复到较早状态）。
- **初始信任根设置依赖 ApprovePolicy（如TOFU）**，若被欺骗允许恶意初始状态，则安全性受损。作者建议生产环境使用更强策略（如签名验证）。
- **SCM容错限制**：多数节点失效时，SCM丧失活性，系统故障关闭直到重新配置。
- **实验覆盖有限**：未评估不同网络延迟、不同快照策略、多轮交互长序列下的累积开销，也未与启发式检测方法对比性能。实验规模较小。
- **无真实攻击实验**：由于防御是确定性的，论文未报告实际攻击下的检测成功率或恢复成功率（但依设计应是确定性拒绝或恢复）。
