## 摘要
自主代表用户执行工具调用的AI智能体引发了关于权限管理的紧迫问题：用户可能扮演什么角色，又应该扮演什么角色？尽管有许多提出的方法，用户在智能体权限管理中的角色仍未得到充分探索。我们介绍了Janus，一个用于实现和评估用户参与的智能体权限管理设计的实验平台系统。Janus包含两个组件：Janus-Core，一个支持多种权限管理设计的模块化智能体系统；以及Janus-Harness，一个自动化评估框架。基于一个识别用户参与关键设计轴的概念模型，我们实现了跨越设计空间的六个权限助手，并在三个场景和三个合成响应者上进行了评估。我们证明了用户输入至关重要，可以显著增强隐私和安全性；AI增强用户决策有助于减少认知负载；并且系统设计必须考虑现实用户行为，包括权限疲劳。没有任何单一设计在所有上下文中表现最佳，这促使在智能体系统中部署权限助手时采用更原则性和上下文敏感的方法。Janus公开可用，以支持未来对这一智能体系统设计维度的研究。

## Abstract
AI agents that autonomously execute tool calls on a user's behalf raise pressing questions about permission management: what role could users play, and what role should they play? Despite many proposed approaches, the user's role in agentic permission management remains under explored. We introduce Janus, a playground system for implementing and evaluating user-involved agentic permission management designs. Janus consists of two components: Janus-Core, a modular agentic system supporting a diverse spectrum of permission management designs, and Janus-Harness, an automated evaluation framework. Grounded in a conceptual model that identifies key design axes for user involvement, we implement six permission assistants spanning the design space and evaluate them across three scenarios and three synthetic responders. We demonstrate that user input is critical and can significantly strengthen privacy and security, that AI augmentation of user decisions can help reduce cognitive load, and that realistic user behavior including permission fatigue must be accounted for in system design. No single design performs optimally across all contexts, motivating a more principled and context-sensitive approach to deploying permission assistants in agentic systems. Janus is publicly available to support future investigation into this dimension of agentic system design.

### 论文核心问题与整体含义（研究动机和背景）

- **核心问题**：AI 智能体（agent）能够自主调用工具（如发送邮件、修改日历），但这种自主性带来了严峻的隐私与安全风险（如提示注入、幻觉导致的误操作）。如何管理系统对用户数据和功能的“权限”——即谁在何时能做什么——成为关键挑战。
- **背景矛盾**：传统安全原则（最小权限、完全仲裁）要求对所有访问进行检查，但智能体的动态性、上下文依赖性和用户交互的模糊性使得这一原则难以落地。现有方案要么完全由用户运行时确认（导致疲劳和过度授权），要么依赖预定义持久策略（缺乏灵活性，无法处理复杂上下文），要么完全自动化（忽略用户意图）。
- **研究缺口**：**用户参与**在智能体权限管理中处于“未被充分探索”的状态。虽然已有若干系统级方案（如 CaMeL、IronCurtain），但缺乏一个统一的实验平台来系统比较不同用户参与设计的效果及其权衡。

### 方法论：核心思想、关键技术细节与流程

- **核心思想**：构建一个可扩展的实验平台 **Janus**，支持多种用户参与式权限管理设计的实现与自动化评估。通过识别关键设计轴，将不同权限管理策略映射到统一概念模型，从而进行系统比较。
- **关键技术细节**：
  - **概念模型**：将代理系统拆分为用户、代理（LLM）、工具、权限管理模块。运行时控制流：用户发任务 → 代理决策 → 每个工具调用都需经过权限管理模块（先检查持久策略，再交由权限助手处理）。
  - **Janus-Core**（实验系统）：
    - **代理**：基于 Google ADK 和 LiteLLM 的 ReAct 风格的 observe-plan-act 循环，工具调用被钩子拦截。
    - **工具**：三类模拟工具（邮件、日历、文件），操作于合成数据，具有明确的元数据（类别、动作、参数模式）。
    - **策略管理器**：确定性规则引擎，快速匹配持久策略（allow/deny）。若未匹配，则调用权限助手。
    - **权限助手**：模块化单元，可自定义。提供多轮用户交互接口（通过 CLI 模拟），最终输出 allow/deny，并可修改持久策略。
  - **Janus-Harness**（评估框架）：
    - 自动化运行：通过合成响应者模拟用户对权限提示的回应。
    - 场景定义：JSON 文件指定工具数据、用户目标、**ground truth**（哪些工具调用是攻击/越权/期望的）。
    - 提供输出验证（正则或 LLM judge）和全面日志记录。
- **算法流程**（文字说明）：
  1. Janus-Harness 向代理发送用户任务。
  2. 代理进入 ReAct 循环，每次提议工具调用时，策略管理器先对持久策略做匹配。
  3. 若策略不覆盖，则调用权限助手；权限助手可能触发用户交互（通过合成响应者回应）。
  4. 若授权，则执行工具调用；否则代理跳过该调用。
  5. 重复直至代理回应完成或超过尝试次数（5次）。
  6. 记录所有调用、决策、用户交互次数、输出验证结果。

### 实验设计：场景、基准与对比方法

- **数据集/场景**：设计了三个典型使用场景（均基于 AgentDojo 启发）：
  - **场景1：个人邮件分类**（如查看收件箱、转发邮件）。
  - **场景2：日历管理冲刺**（如创建事件、检查空闲）。
  - **场景3：项目协作工作流**（如发送团队照片、共享反馈）。
- **每场景包含4个子场景**（代表不同用户意图）：
  - **attack**：包含恶意提示注入攻击，特定工具调用被标记为“攻击”。
  - **permissive**：高度宽松用户，无工具调用被标记为越权。
  - **balanced**：中等严格，部分工具调用越权。
  - **restrictive**：高度严格，大部分隐私模糊工具调用越权。
- **基准（ground truth）**：每个子场景明确指定哪些工具调用属于攻击、越权或期望路径。
- **对比方法（共8种权限助手配置）**：
  - **auto_approve**：总是批准，等价于风险阈值为1的 risk_assessment。
  - **user_confirmation**：总是询问用户，等价于风险阈值为0。
  - **risk_assessment**：LLM 评估风险（0~1），设阈值0.2或0.7，低于阈值自动批准，高于则询问用户。
  - **risk_assessment_autonomous**：同样评估风险，高于阈值直接拒绝，不询问用户（阈值为0.2或0.7）。
  - **constitution**：基于自然语言“宪法”编译为JSON规则，第一匹配规则后如果未命中则意图匹配自动批准，否则询问用户。
  - **policy_suggestion**：提供用户拒绝或创建持久策略的选项，通过LLM提案策略。
- **合成响应者**：模拟用户对权限提示的反应：
  - **always_yes**：总是批准。
  - **always_no**：总是拒绝。
  - **alignment_aware**：只拒绝被标记为攻击或越权的调用，其余批准（理想用户）。

### 主要结论与发现

1. **用户输入至关重要**：在用户输入正确的条件下（alignment_aware），可以显著减少攻击/越权工具调用。但在 always_yes 下，几乎所有助手都失效。
2. **AI 辅助可降低认知负载**：如 risk_assessment 和 risk_assessment_autonomous 通过自动过滤低风险调用，减少了用户交互次数，同时保持一定安全性。
3. **现实用户行为（如权限疲劳）必须纳入设计**：always_yes 导致了高失败率，说明权限助手不能仅依靠用户否决；需内置自动拒绝机制。
4. **局部正确决策可能导致全局错误**：例如 policy_suggestion 中，用户批准一个看似合理的持久策略，后续可能自动授权越权调用。
5. **无单一设计在所有场景最优**：不同权限助手在不同子场景下表现差异很大。constitution 在攻击子场景有漏洞，但在越权子场景表现良好。
6. **设计需考虑非确定性**：同一权限助手在不同重复中可能表现出不同的攻击/越权率，影响用户理解和信任。

### 优点：方法与实验设计亮点

- **模块化、可扩展平台**：Janus-Core 和 Janus-Harness 的设计使得新增工具、场景、权限助手非常方便（例如第二作者独立半天内实现了 constitution 助手）。
- **概念模型清晰**：通过设计轴（持久策略、自动行动、用户参与程度、依据）统一描述多种权限管理方案，便于系统比较。
- **包含多种用户行为模拟**：使用三个合成响应者（always_yes, always_no, alignment_aware）覆盖极端和理想情况，利于暴露设计的脆弱点。
- **自动化评估**：减少人工误差，并能快速收集大量统计指标（攻击/越权调用数、用户交互次数、输出验证等）。
- **开源**：Janus 公开在 GitHub，促进社区后续研究。
- **实验设计系统化**：全因子设计，重复 5 次，提供方差信息；结果可视化（包络图）直观展示权衡。

### 不足与局限

- **场景覆盖有限**：仅三个场景（邮件、日历、协作），每个场景四个子场景，远不足代表真实世界多样性。场景数据为合成，可能缺乏真实性。
- **权限助手实现不够成熟**：均为简单原型（最短42行，最长408行），未充分利用高级技术（如多轮对话、学习用户偏好、防御提示注入）。结论可能不适用于更复杂的生产系统。
- **用户行为模拟粗糙**：合成响应者 binary 式回应，无法模拟真实用户在疲劳、被攻击、或理解偏差下的复杂行为。缺乏真实用户研究。
- **未考虑多代理协作**：系统只针对单代理场景，未探索代理间权限传递与协调。
- **未涉及策略生命周期管理**：如策略的长期遗忘、更新、冲突解决等。
- **非确定性未充分分析**：尽管重复5次，但未深入讨论影响非确定性的因素（模型随机性、上下文长度等）。
