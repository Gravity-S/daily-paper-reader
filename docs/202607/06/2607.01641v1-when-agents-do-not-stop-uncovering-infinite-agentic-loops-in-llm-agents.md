## 摘要
LLM智能体越来越依赖迭代执行，通过规划、工具使用、状态更新和智能体协作来完成任务。虽然这种设计实现了灵活的自动化，但也产生了一类新的故障：当反馈路径未有效约束时，智能体可能会重复执行模型调用、工具、工作流转换或智能体交接。我们将此问题称为无限代理循环（IAL）。IAL不是普通的编程循环；它们源于智能体逻辑、框架语义、运行时观察和终止机制之间的交互。此类故障可能将单个请求放大为长时间运行的模型和工具执行，导致成本耗尽、模型拒绝服务、上下文增长以及重复的外部副作用。我们提出了IAL-Scan，一种用于检测真实世界LLM智能体项目中IAL故障的静态分析工具。IAL-Scan将异构的智能体代码抽象为框架无关的智能体IR，构建代理循环依赖图（ALDG）以恢复显式和框架诱导的反馈路径，并检查这些路径能否在无有效约束的情况下重复到达高成本或状态增长的操作。我们在6,549个LLM智能体仓库上评估了IAL-Scan。它报告了74个潜在发现，其中人工审查确认了47个项目中的68个IAL故障，实现了91.9%的精确率。

## Abstract
LLM agents increasingly rely on iterative execution to solve tasks through planning, tool use, state updates, and agent collaboration. While this design enables flexible automation, it also creates a new class of failures: an agent may repeatedly execute model calls, tools, workflow transitions, or agent handoffs when the feedback path is not effectively bounded. We call this problem Infinite Agentic Loops (IALs). IALs are not ordinary programming loops; they arise from the interaction between agent logic, framework semantics, runtime observations, and termination mechanisms. Such failures can amplify a single request into long running model and tool execution, causing cost exhaustion, model denial of service, context growth, and repeated external side effects. We propose IAL-Scan, a static analysis tool for detecting IAL failures in real-world LLM agent projects. IAL-Scan abstracts heterogeneous agent code into a framework independent Agent IR, builds an Agentic Loop Dependence Graph (ALDG) to recover explicit and framework induced feedback paths, and checks whether these paths can repeatedly reach costly or state growing operations without an effective bound. We evaluate IAL-Scan on 6,549 LLM agent repositories. It reports 74 potential findings, among which manual review confirms 68 IAL failures across 47 projects, achieving 91.9% precision.

LLM 智能体通过迭代的“感知–推理–行动”循环执行任务，这种设计在带来灵活自动化的同时，也引入了一类新的结构性故障——**无限代理循环（Infinite Agentic Loops, IALs）**。当智能体代码中存在反馈路径（如模型调用→工具执行→状态更新→再次模型调用），且该路径未受到有效的终止条件约束时，单个请求可能被放大为持续不断的模型调用与工具执行，导致 API 成本耗尽、模型拒绝服务、上下文窗口膨胀以及重复的外部副作用。尽管主流框架（LangChain、LangGraph、OpenAI Agents SDK 等）提供了 `max_iterations`、`max_turns` 等循环控制机制，但开发者可能遗漏、误配置或使其位于实际反馈路径之外，因此 IAL 仍然是部署中的真实风险。现有静态分析工具（如 CodeQL、Semgrep）不建模代理执行语义，而专门针对代理的研究（AgentProof、Agent Audit 等）不关注循环的无限性。本文正是为了填补这一检测空白。

### 2. 论文提出的方法论：核心思想、关键技术细节

论文提出 **IAL-Scan**，一个静态分析工具，用于检测下游 LLM 代理项目中的 IAL 故障。其核心流程分为三步：

- **Agent IR 构建**：将异构的代理代码（不同框架、自定义代码）抽象为框架无关的中间表示（Agent IR）。Agent IR 由事实（Fact）和关系（Relation）组成，包括执行单元（ExecutionUnit）、控制器（Controller，如循环、重试、路由）、调用（Invocation，如模型调用、工具调用）、状态更新（StateUpdate）、边界（Bound）和退出记录（ExitRecord）。  
   - 程序事实提取：解析 Python AST，识别循环、递归、调用表达式、状态更新等，转化为 Agent IR 事实。  
   - 框架行为建模：针对 8 个主流框架（LangChain、LangGraph、CrewAI、AutoGen、LlamaIndex、OpenAI Agents SDK、Google ADK、Semantic Kernel），建模框架特有的构造（如工具注册、图节点、条件边、委托、重试等），将其转换为 Agent IR 事实和关系。

- **ALDG 构建**：在 Agent IR 基础上构建**代理循环依赖图（Agentic Loop Dependence Graph, ALDG）**，一个带类型的有向属性图。节点包括控制器、高成本调用、增长状态更新、执行作用域等 12 种节点；边包括控制流、调用、工作流转换、工具分发、反馈（回边、递归、代理重入等）和退出（条件真/假）等 10 种边。对每个控制器节点，ALDG 还标注循环属性（如是否包含高成本操作、状态增长、退出来源等）。

- **IAL 故障检测**（算法 1）：  
   1. **SCC 候选发现**：从 ALDG 中提取含有反馈边的强连通分量（SCC），保留能从代理入口到达且包含代理动作、高成本调用或状态增长的 SCC。  
   2. **控制器与边界验证**：对每个候选 SCC，选择其延续控制器（根据控制器节点、守卫依赖等分类为确定性、模型控制、工具控制等），然后检查是否存在有效边界覆盖该反馈路径。边界需直接约束控制器或其运行作用域，而不是任意的外部限制。  
   3. **IAL 识别**：报告满足“存在代理反馈路径”、“路径到达高成本/状态增长操作”、“无有效边界覆盖”三个条件的候选。对于模糊案例，使用可选的 LLM 辅助修剪（仅作为负向过滤器）排除非目标循环（如流消费者、生命周期循环、测试代码等）。

**关键公式/算法流程**（文字说明）：  
- 将 Agent IR 关系隐式转换为 ALDG 边：`ri --κ,Δ--> rj` 给出 ALDG 边 `(vri, vrj, κ)`。  
- 边界覆盖检查：`Bound(c) = { b | b constrains c or its owner scope }`，若候选路径的所有控制器/执行单元均无有效边界则判定为 IAL。

### 6. 论文的主要结论与发现

1. **IAL 是真实且普遍存在的问题**：在 6,549 个仓库中，47 个项目（0.72%）存在经确认的 IAL 故障，共 68 例。  
2. **IAL-Scan 达到高精确率**：91.9%，且覆盖 8 个主流框架。LangGraph 和 AutoGen 贡献了 66.2% 的发现。  
3. **主要失败模式**：重试反馈无边界（25%）、工具调用迭代无边界（23.5%）、多智能体对话无轮次限制（20.6%）。  
4. **核心根因**：所有 68 例均缺失有效边界；模型/工具控制的延续和状态增长是主要放大器。  
5. **各分析组件均必要**：消融实验表明，移除任一组件都会降低精/或增加假阳性/Token 消耗。  
6. **通用 LLM 基线无法替代**：纯 LLM API 仅覆盖 23/68 真阳，编码助手覆盖 50/68 但代价高、超时多。  
7. **静态阶段稳定，LLM 修剪有波动**：静态候选完全重复，LLM 修剪的告警数变化约 ±4，TP 覆盖变化 60-68。

### 7. 优点

- **框架无关的抽象**：Agent IR 统一了不同框架的异构语义，使分析可扩展。  
- **专门针对 IAL 的设计**：ALDG 显式提取反馈路径和边界覆盖，避免了普通循环检测的盲目性。  
- **高精度与低误报**：91.9% 的精确率，通过多层过滤（良性循环过滤、LLM 修剪）控制假阳性。  
- **全面的消融与对比**：实验设计系统，充分验证了各组件贡献和与 LLM 基线的差距。  
- **可解释性**：ALDG 图可展示反馈路径、控制器、边界等，便于开发者理解问题。  
- **安全性泛化**：定义了 IAL 新故障模式，对框架设计者和应用开发者都有指导意义。

### 8. 不足与局限

- **静态分析过度近似**：可能产生假阳性；对间接边界（如外部计数器、异步超时）解析能力有限，导致 6 个 FP。  
- **语言和框架支持有限**：仅支持 Python，且仅覆盖 8 个框架；其他语言或自定义框架可能导致假阴性。  
- **高度定制化语义支持不足**：项目特定的调度器、基于自然语言输出的语义检查等无法准确建模。  
- **LLM 修剪不稳定**：不同模型修剪结果差异明显（TP 覆盖 41 到 68），且可能存在错误剔除，需要谨慎使用。  
- **未考虑动态执行路径**：无法分析运行时条件、外部状态变化引起的循环变化。  
- **实验范围限制**：对比基线中 LLM 方法因成本和超时未完全覆盖所有项目，可能低估了它们的潜力；另外缺乏与其他专门代理分析工具的直接对比（如 AgentProof、Agent Audit），因为它们不针对 IAL。  
- **性能开销**：虽然平均 31.2s/项目可接受，但最大项目可能更耗时，且 Token 消耗（4.2K）在大型代码库中可能累积。
