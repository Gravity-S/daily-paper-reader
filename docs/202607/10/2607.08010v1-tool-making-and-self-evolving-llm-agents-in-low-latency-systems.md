## 摘要
生产环境中的LLM智能体在处理重复程序步骤时，往往因每次请求都重新生成代码而导致延迟和可靠性问题。我们用一个智能体工具制造流水线替代这种推理时编码循环，该流水线在部署前将重复的标准操作流程（SOP）步骤编译为经过验证且版本化的工具。工具制造者在实时环境中进行合成，收集执行轨迹、观察后端模式与数值、生成候选工具，并根据标注样本进行修复。运行时，生产智能体直接调用这些工具，仅在必要时回退到代码生成。我们将该方法部署在一个履行中心的报警分类系统中，该系统的智能体需针对44节点SOP在异构指标后端上诊断报警。在生产环境中，工具调用将p50延迟降低了42%。在1500个历史报警中，通过抑制重复步骤的逐次方差，端到端错误率最高降低了53%。由于工具返回紧凑的结构化判定结果，它们还支持更简单的直接调用架构，在受控消融实验中进一步将p50延迟降低了62%。版本化工具还改善了可审计性，并暴露了规范缺口及上游数据漂移。我们的结果表明，自演化智能体可以使工业LLM系统更快、更可靠、更易于操作。

## Abstract
Production LLM agents often waste latency and reliability by regenerating code for the same procedural steps on every request. We replace this inference-time coding loop with an agentic tool-making pipeline that compiles repeated SOP steps into validated, versioned tools before deployment. The tool-maker grounds synthesis in the live environment as it collects execution traces, observes backend schemas and values, generates candidate tools, and repairs them against labeled cases. At runtime, the production agent calls these tools directly and falls back to code generation only when needed. We deploy the approach in a Fulfillment Center alarm-triage system, where an agent diagnoses alarms against a 44-node SOP over heterogeneous metric backends. In production, tool calls reduce p50 latency by 42%. On 1,500 historical alarms, they reduce end-to-end error rate by up to 53% by suppressing run-to-run variance in repeated steps. Because tools return compact structured verdicts, they also enable a simpler direct-call architecture, reducing p50 latency by a further 62% in a controlled ablation. Versioned tools also improve auditability and expose specification gaps and upstream data drift. Our results show that self-evolving agents can make industrial LLM systems faster, more reliable, and easier to operate.

- **核心问题**：当前生产环境中，LLM 智能体在处理重复的标准操作流程（SOP）步骤时，每次请求都重新生成代码，导致高延迟、高成本和高运行方差。这种“推理时代码生成”模式在稳定的后端环境中效率低下。
- **整体含义**：论文提出将重复的推理时编码工作离线预编译为版本化的工具（Tool），使智能体在运行时直接调用，从而大幅降低延迟、提高可靠性和可审计性。这一范式从“每次即兴编码”转向“自演化工具库”，为工业级低延迟 LLM 系统提供了实际可行的路径。

## 2. 方法论

### 核心思想
- **离线工具制造**：在部署前，将每个 SOP 节点（决策或动作）编译成一个确定的工具函数。工具返回结构化判定（true/false/no data）并附带支持细节。
- **运行时调用**：主智能体直接调用已编译的工具，仅在工具缺失或失败时回退到代码生成。

### 关键技术细节
1. **工具制造管道**（Figure 1）：
   - **数据收集子智能体**：对每个 SOP 节点，在 3 个采样的标注案例上运行基线 CodeAct 子智能体（访问生产 MCP），收集执行轨迹（包含查询代码、MCP 响应、观察到的模式、判定及标签）。轨迹为工具制造者提供环境中的具体细节（字段名、数据类型、空值处理等）。
   - **工具制造 LLM**：基于 SOP 文本、节点在决策树中的位置以及数据收集轨迹，生成候选工具代码。工具具有统一签名：输入（仓库，时间戳，上下文）→ 输出（判定+细节）。
   - **测试-修复循环**：将候选工具在完整训练集上测试，失败案例交由反射 LLM 生成简短诊断，然后工具制造 LLM 根据诊断重写工具。循环最多 3 轮，选择训练集通过率最高的候选部署。

2. **推断时使用**：两种配置：
   - **子代理调用工具**：主智能体将节点委托给子智能体，后者调用工具。
   - **主代理直接调用工具**：主智能体直接调用工具（去掉了子智能体层）。
   - 异常情况回退到 CodeAct 子智能体。

3. **智能体微调**：将大教师模型（GLM-4.7）的知识蒸馏到小模型（Qwen3 32B / GLM-4.5-Air）。教师生成轨迹时模拟工具可用性变化（75% 工具可用，25% 随机禁用部分工具，25% 全禁），使小模型在工具缺失时能可靠回退。

## 6. 主要结论与发现

1. **延迟降低**：工具调用使 p50 延迟降低 42%；直接调用架构进一步降低 62%（控制模型和推理框架）。
2. **错误率降低**：端到端错误率降低高达 53%（GLM-4.5-Air 从 1.7% 降到 0.8%）。主要源于消除子智能体的代码生成方差和 SOP 偏离。
3. **工具组件必要性**：数据收集轨迹（+D）和测试-修复循环（+R）两者均必要，缺一不可。单独移除任一方，pass@1 下降 5–10pp。
4. **瓶颈在规范质量**：剩余错误集中在 4 个节点（缺乏边缘案例标注或 SOP 模糊）。增强数据和明确 SOP 后 pass@1 从 94.5% 提升至 99.9%。
5. **修复策略**：带反射的修复优于无反射；数据轨迹对反射修复更重要（+4.0pp vs +1.2pp）。
6. **数据效率**：仅需 5 个标注案例即可接近全数据集的性能（91.8% vs 94.5%）。无标签方案也达 91.6%，接近有标签天花板。
7. **附加效益**：版本化工具日志暴露上游数据不一致（类型差异、指标语义错误、格式变更），提升可审计性和运维洞察。
8. **模型无关性**：工具库可跨不同小模型使用（GLM-4.7 生成 → Qwen3 32B、GLM-4.5-Air），无需重新生成。

## 7. 优点

- **实用的离线-在线分离**：在离线用强模型做一次工具制造，在线用轻量模型直接调用，实现性价比最优。
- **环境接地**：数据收集轨迹提供了 SOP 文本缺失的实操细节，从根本上减少了歧义。
- **修复循环的设计**：结合反射诊断，有效纠正了语义偏差和边界漏洞。
- **全面的实验验证**：从每个组件的消融到端到端生产测量，统计方法严谨，结论可靠。
- **发现隐藏问题**：工具日志揭示环境漂移和不一致，不仅优化了当前系统，还能预防后续故障。
- **参数化与可迁移**：工具可调阈值参数适应多站点，且无模型耦合，降低维护成本。

## 8. 不足与局限

1. **单一应用领域**：仅部署在“出站码头报警分类”一个场景，泛化到其他工业流程（如自由格式运行手册）未经验证。虽然论文指出典型代理任务有相似性，但尚存不确定性。
2. **依赖标注数据**：工具制造需要每个节点有标注训练案例（100–200 个）。虽然无标签方案接近但仍有差距（91.6% vs 94.5%）；完全无标签的通用解决方案仍需探索。
3. **人工监督环节**：修复循环尚不能保证完全自动化——监控代理仍需人工审查；SOP 规范澄清本质上是人工任务（依赖领域专家）。
4. **性能基线有限**：对比基线仅为无工具的 CodeAct 子智能体；未对比其他工具生成方法（如单次提示、其他 SOP-编译系统），但文中讨论的相关工作中已有提及。
5. **计算成本**：尽管离线工具制造是一次性成本，但每次节点需多轮 LLM 调用（数据收集+修复循环），总 token 消耗达数十万至百万级。对于快速迭代的环境，可能更新频繁而成本上升。
6. **未考虑极端动态环境**：如果后端 schema 频繁变更，工具可能频繁失败，回退代码生成的比例增加，削弱优势。
7. **微调依赖教师模型**：小模型的泛化能力依赖于大教师模型在多种工具可用性下的轨迹质量，存在知识蒸馏偏差风险。
8. **实验统计**：虽然置信区间和显著性检验已充分，但部分比较（如无标签方案）未报告与有标签的天花板的统计检验；且 seed 数（30 或 10）较高但部分消融种子数偏少（如混合轨迹微调部分未明确 seed）。？
