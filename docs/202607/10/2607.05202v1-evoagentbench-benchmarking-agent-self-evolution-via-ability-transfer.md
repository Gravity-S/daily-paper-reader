## 摘要
在长周期LLM系统中，智能体自我进化在很大程度上是程序性的：有用的经验不仅仅是存储的信息，而是用于搜索、调试和验证的可重用程序。然而，当前的评估并未隔离这种形式的迁移。智能体基准测试评估单轮任务解决；记忆基准测试针对信息保留而非程序重用。我们引入了EvoAgentBench，这是一个通过能力引导迁移在四个智能体领域（网络研究、算法推理、软件工程和知识工作）进行智能体自我进化的基准测试。EvoAgentBench从智能体执行中提取基于轨迹的能力，将其规范化成操作单元，并构建连接共享程序重叠任务的领域特定能力图。通过设计，每个测试任务都由经过验证的训练侧能力支持。在528/267的训练/测试划分、两种支架和三种基座模型上，精心策划的能力内容能够可靠地在不同模型家族间迁移，但目前没有任何自动化方法能在所有设置中持续获得正向收益。EvoAgentBench将自我进化评估从总体准确率比较转向对经验编码、路由和吸收的细粒度诊断。该基准测试公开可用：https://huggingface.co/datasets/EverMind-AI/EvoAgentBench

## Abstract
Agent self-evolution in long-horizon LLM systems is largely procedural: useful experience is not merely stored information, but reusable procedures for searching, debugging, and verification. Yet current evaluations do not isolate this form of transfer. Agent benchmarks test single-episode task solving; memory benchmarks target information retention rather than procedural reuse. We introduce EvoAgentBench, a benchmark for agent self-evolution via Ability-guided transfer across four agentic domains: web research, algorithmic reasoning, software engineering, and knowledge work. EvoAgentBench extracts trace-grounded Abilities from agent executions, canonicalizes them into operational units, and builds domain-specific Ability Graphs linking tasks that share procedural overlap. By design, every test task is backed by verified training-side Ability support. Across a 528/267 train/test split, two scaffolds, and three backbones, curated Ability content transfers reliably across model families, but no current automatic method sustains positive gain in all settings. EvoAgentBench shifts self-evolution evaluation from aggregate accuracy comparison to fine-grained diagnosis of experience encoding, routing, and uptake. The benchmark is publicly available at https://huggingface.co/datasets/EverMind-AI/EvoAgentBench.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：当前智能体（LLM agent）虽然在长周期任务上表现提升，但评估范式存在盲区——单任务求解基准（如 SWE-Bench、GAIA）只测一次性能，记忆基准（如 LongMemEval）只测信息回忆，都不测试“从过往经验中提取可复用的**程序性知识**并迁移到新任务”的能力。
- **整体含义**：智能体自我进化的本质是轨迹到程序的转移，即把搜索、调试、验证等过程提炼为可重用程序，并在相关任务间泛化。现有基准无法隔离并度量这一能力，阻碍了方法开发和诊断。
- **关键洞察**：评估迁移需要满足“能力支撑但实例不重叠”——测试任务必须与训练任务共享可转移的能力，但不能是重复题目。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：定义**能力（Ability）**——从执行轨迹中提取的可复用操作单元（如搜索策略、调试流程、验证规则），包含触发条件、过程、边界和角色（Method/Guard/Workflow）。构建**能力图（Ability Graph）**，节点为任务，边表示共享至少一个能力，由此实现程序级别的迁移测量。
- **关键步骤**：
  1. **多基座轨迹收集**：对每个任务，使用 Kimi-K2.5、GLM-5.1、DeepSeek-V3.2 三个基座模型，在两种脚手架（OpenClaw、Nanobot）下执行，获得无进化基线的轨迹池。
  2. **能力提取**：利用 Claude Sonnet 4.6 作为提取器 Eϕ，联合任务描述、答案和所有基座轨迹，生成原始能力卡片（触发条件、过程、边界、角色、证据）。每张卡片必须是可操作的具体程序，而非主题标签。
  3. **能力规范化**：通过嵌入阻塞 + 三模型 LLM 裁决 + 领域专家复审，将等价卡片合并为规范能力单元。合并必须满足：相同角色、兼容触发、等价过程、相同成功机制。组级一致性检查后形成最终能力词汇表。
  4. **能力图构建**：每个领域构建无向图，边连接至少共享一个规范能力的两个任务。仅支持至少两个任务且过程具体的能力才用于建边。
  5. **能力感知分割**：在社区（Louvain 检测）内进行约束抽样，确保每个测试任务至少与训练任务共享一个能力，且每个测试能力在训练侧有支撑。优先选择基线分数适中的任务以保留提升空间。

- **最终数据集**：528 训练任务 + 267 测试任务，零无支持测试任务。包含 170 个规范能力单元，覆盖四个领域。

## 3. 实验设计

- **数据集/场景**：四个长周期智能体领域
  - **网络研究**：BrowseComp-Plus（LLM 裁判评估）
  - **算法推理**：LiveCodeBench（隐藏测试用例 pass@1）
  - **软件工程**：SWE-Bench Verified（仓库测试套件解决率）
  - **知识工作**：GDPVal（参考答案 LLM 比较）

- **基准方法**：退化基准基线为无进化的 Vanilla（无经验状态）。对比三种自动自我进化方法：
  - **Memento**：检索最近似训练案例（query-plan-outcome 三元组）。
  - **ReasoningBank**：检索蒸馏推理记忆池中的自然语言策略。
  - **GEPA**：在训练集上进化一个提示（无检索，广播至所有测试任务）。
  - **Anchor Skill**（诊断参考）：使用 curator 侧能力标签检索训练侧能力技能，技能内容只来自训练任务原始卡片；构造基座（Kimi/GLM/DeepSeek）与评估基座不重叠。

- **评估设置**：两个脚手架（OpenClaw、Nanobot）× 三个基座（Qwen3.5-27B、Qwen3.5-397B、Gemma-4-31B），共 6 种配置。每个设置内任务、工具、超时、评分合同固定。每个实例重复 3 次取均值标准差。

## 6. 主要结论与发现

1. **能力内容跨模型家族可靠迁移**：Anchor 在所有 24 个条件中均正向改善（平均 ∆ +5.8~+10.5），证明训练侧能力信息足以支撑跨基座迁移。
2. **当前自动方法普遍不稳定**：Memento、ReasoningBank、GEPA 都在至少一个条件组合中出现负迁移（∆ < 0），成功率因领域、脚手架、基座而变化。
3. **瓶颈在于方法侧提取与路由**：由于任务、工具、配置完全相同，自动方法的失败可归因于自身机制（案例匹配表面相似性、策略蒸馏抽象损失、无检索的通用提示）。
4. **成本与准确率解耦**：进化可能增加回合数（即使任务解决），也可能减少；单报告准确率掩盖了方法行为异质性。例如 Memento 在已解决任务上仍消耗更多回合（+39.4%），而 Anchor 常节省回合。
5. **单一设置评估不可靠**：方法效果随基座规模、脚手架类型变化，跨条件评估对可靠比较至关重要。

## 7. 优点

- **创新评估框架**：首次将自我进化评估从整体准确率转向对“能力编码→路由→吸收”各阶段的细粒度诊断，通过能力图保证测试任务一定存在可转移知识，避免混淆。
- **能力定义的严谨性**：基于轨迹的多基座证据提取，通过三模型裁决 + 人类专家复审保证操作等价性，不是简单的语义聚类。
- **信息泄漏控制严格**：自动方法只访问训练提示和验证结果，Anchor 技能内容来自不同构造基座且无测试侧轨迹，测试侧能力标签仅用于路由（不可自动获得）。
- **多维度分析**：同时报告准确率、回合成本、已解决/未解决任务的变化，提供更全面的行为画像。
- **数据集公开**：提供 HuggingFace 数据集链接，促进可重复研究。

## 8. 不足与局限

- **领域覆盖有限**：仅四个文本型智能体领域，未涉及多模态、具身、对话交互等场景。
- **能力提取器偏差**：仅使用单一 LLM（Claude Sonnet 4.6）进行原始卡片提取，不同提取器可能导致不同能力词汇；后续三模型裁决和人工复审虽缓解但未完全消除。
- **评估规模**：每个领域 56~86 个测试任务，能力图保留 1108 个任务但仅用 267 个测试；更大规模评估可增强统计检测力。
- **构造基座与评估基座分离**：Anchor 技能来自不同构造基座，虽然显示跨族泛化，但无法保证能力图对评估基座自身是最优的。
- **头尾部任务选择偏差**：测试任务优先选择构造基座留有提升空间的任务，绝对 ∆ 大小不宜推广到随机任务抽样；但方法间比较公平（所有方法共享相同分割）。
- **预训练数据污染**：评估基座可能在预训练中见过源基准数据，但这不影响条件比较（所有条件共享相同模型和分割）。
- **计算资源未报告**：复现成本难以估算，可能影响研究可访问性。
