## 摘要
自主Web代理有望自动化日常浏览任务，但继承了Web上最古老的攻击面之一。跨站脚本已经证明，即使在良性页面上，混合可信与不可信内容也是危险的。代理通过将自然语言解释为指令而重新浮现这种风险，允许第三方和用户生成的内容通过提示注入劫持代理。核心挑战在于，推导任务特定的安全策略需要对页面结构进行推理，而页面结构与攻击者内容纠缠在一起。我们提出Prismata，一种为Web代理实施上下文最小权限的防御机制，同时约束代理所能看到的内容和所能执行的操作。Prismata的动态信任推导为页面内容生成权限标签，并具有结构限制保证，受经典完整性模型启发，这些保证限制了任何标签错误，使得标签只能降低权限，且错误标签是受限的。Prismata的机械限制通过删改内容和限制代理能力来执行这些标签。重要的是，这些机制不需要开发者注释，因此Prismata支持长尾网站。在最近公开的Web代理攻击（包括自适应变种）中，Prismata在保持良性任务效用的同时，大幅降低了攻击成功率。

## Abstract
Autonomous web agents promise to automate everyday browsing tasks, but inherit one of the web's oldest attack surfaces. Cross-Site Scripting proved that mixing trusted and untrusted content is dangerous, even on benign pages. Agents resurface this risk by interpreting natural language as instructions, allowing third-party and user-generated content to hijack the agent via prompt injection. The core challenge is that deriving a task-specific security policy requires reasoning over page structure that is entangled with the attacker's content.   We present Prismata, a defense enforcing contextual least privilege for web agents, constraining both what the agent sees and what it can do. Prismata's dynamic trust derivation produces permission labels for page content, with structural confinement guarantees, inspired by classical integrity models, that bound any labeling errors so that labels can only decrease in privilege and mislabelings are bounded. Prismata's mechanical confinement enforces these labels by redacting content and restricting agent capabilities. Importantly, these mechanisms require no developer annotations, so Prismata supports the long tail of websites. Across recent published web agent attacks, including adaptive variants, Prismata substantially reduces attack success while preserving benign task utility.

- **研究背景**：

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：**上下文最小权限（Contextual Least Privilege）**。Prismata同时约束代理“能看到什么”和“能做什么”，基于其当前任务动态决定页面上哪些内容是必需的，哪些操作是允许的。

- **关键技术细节**：

    1.  **架构定位**：Prismata作为透明防御层，部署于Web代理和浏览器之间。它拦截每次页面渲染后的DOM（文档对象模型），应用信任推导和限制策略，再将过滤后的观察结果传递给代理。

    2.  **两大核心机制**：
        - **动作门（Action Gate）**：
            - 枚举页面上所有可交互元素（按钮、链接等）。
            - 对每个元素，追踪其从DOM根节点到该元素的“关键路径”（Critical Path）。
            - 使用一个单独的LLM，**仅**基于用户任务和该关键路径的上下文，独立判断该元素是否在任务范围内。
            - **安全性**：由于攻击者注入内容不在该路径上，它就**无法影响**这个判断，保证了决策的纯净性（Case 1）。

        - **Biba解析（Biba Parsing）**：
            - 处理攻击者注入内容恰好位于关键路径上的情况（Case 2 & 3）。
            - **灵感**：受1977年Biba完整性模型启发，应用“不上读、不下写”策略。
            - **过程**：递归地逐层评估关键路径。在查看下一层节点的内容之前，先根据当前节点的结构线索（如标题“Reviews”，CSS类名`review-card`）推断其内容来源（开发者、用户、外部广告等）。
            - **效果**：一旦检测到不可信内容的边界（如评论区的标题），则**锁定标签**并级联应用到所有子孙节点。对于与任务无关的不可信内容直接删除；对于任务所需的（如商品评价），则限制为“只读”，代理可以看但不能与之交互。这保证了在关键标签决策时刻，攻击者内容尚未被暴露给标签器（Case 2）。

    3.  **机械限制（Mechanical Confinement）**：
        - 这是一个**确定性的**执行层，不依赖任何LLM。
        - 根据“动作门”和“Biba解析”评估结果的最小权限值（min）来决定每个元素的最终能力。
        - 通过从观察中移除不可信内容、禁用交互能力、并在执行代理动作前进行权限检查，确保所有决策被强制执行。

    4.  **安全性证明**：论文通过分类讨论（Case 1-3）证明，在大多数现实场景下，关键标签决策不受攻击者内容影响。攻击者需要同时绕过“动作门”和“Biba解析”才能提升某个元素的权限，这在实践中非常罕见。
