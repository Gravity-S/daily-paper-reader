## 摘要
在长周期LLM系统中，智能体自我进化在很大程度上是程序性的：有用经验不仅仅是存储的信息，而是可复用的搜索、调试和验证程序。然而，当前的评估并未孤立这种迁移形式。智能体基准测试评估单轮任务解决；记忆基准测试针对信息保留而非程序复用。我们引入了EvoAgentBench，这是一个通过能力引导迁移来评估智能体自我进化的基准测试，涵盖四个智能体领域：网络研究、算法推理、软件工程和知识工作。EvoAgentBench从智能体执行中提取基于轨迹的能力，将其规范化成为操作单元，并构建特定领域的能力图，将共享程序重叠的任务关联起来。通过设计，每个测试任务都有经过验证的训练侧能力支持。在528/267的训练/测试划分、两种脚手架和三种骨干模型上，精心策划的能力内容在模型系列间可靠迁移，但目前没有自动方法在所有设置中保持正向增益。EvoAgentBench将自我进化评估从聚合准确度比较转变为对经验编码、路由和吸收的细粒度诊断。该基准测试可在https://huggingface.co/datasets/EverMind-AI/EvoAgentBench公开获取。

## Abstract
Agent self-evolution in long-horizon LLM systems is largely procedural: useful experience is not merely stored information, but reusable procedures for searching, debugging, and verification. Yet current evaluations do not isolate this form of transfer. Agent benchmarks test single-episode task solving; memory benchmarks target information retention rather than procedural reuse. We introduce EvoAgentBench, a benchmark for agent self-evolution via Ability-guided transfer across four agentic domains: web research, algorithmic reasoning, software engineering, and knowledge work. EvoAgentBench extracts trace-grounded Abilities from agent executions, canonicalizes them into operational units, and builds domain-specific Ability Graphs linking tasks that share procedural overlap. By design, every test task is backed by verified training-side Ability support. Across a 528/267 train/test split, two scaffolds, and three backbones, curated Ability content transfers reliably across model families, but no current automatic method sustains positive gain in all settings. EvoAgentBench shifts self-evolution evaluation from aggregate accuracy comparison to fine-grained diagnosis of experience encoding, routing, and uptake. The benchmark is publicly available at https://huggingface.co/datasets/EverMind-AI/EvoAgentBench.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究现状**：当前 LLM 智能体在长程任务上已取得显著进展，但真正有用的智能体应能从过去经验中提炼可复用的“过程性知识”（如搜索、调试、验证策略），而非每次从零开始。现有基准存在两大盲点：
  - **任务型基准**（如 SWE-Bench、LiveCodeBench）只考察单轮任务解决，不测试经验如何转化为可复用程序。
  - **记忆型基准**（如 LongMemEval、AMA-Bench）只评估信息保留与回忆，未触及过程性复用。
  - **自进化基准**（如 LifelongAgentBench、SEA-Eval）将工件质量与任务顺序、检索策略、脚手架动态混杂，无法隔离归因。
- **核心问题**：如何构建一个能**严格控制训练侧能力支持**、同时**隔离经验编码与路由环节**的评估框架？现有方法失败时，无法判断瓶颈在于训练侧能力缺失、提取失败还是路由失效。
- **整体含义**：本文提出 **EvoAgentBench**，一个基于“能力迁移”评估智能体自我进化的基准，通过**能力图（Ability Graph）** 保证每个测试任务都有训练侧能力支持，从而将评估从聚合准确率比较转向对**经验编码、路由、吸收**的细粒度诊断。

## 2. 方法论：核心思想与关键技术细节

### 2.1 核心思想
- **定义**：智能体自我进化 = 从训练轨迹中提取**能力（Ability）**，并使其在测试时跨任务转移。能力被定义为**可复用的操作单元**（含触发条件、过程、边界），而非表面标签。
- **设计原则**：“能力支持但实例不共享”（ability-supported yet instance-disjoint），确保测试任务与训练任务共享过程性能力，但不包含相同实例。

### 2.2 三阶段构建流程（图1）

**阶段 1：多骨干轨迹收集**
- 使用 **3 个构造骨干**（Kimi-K2.5、GLM-5.1、DeepSeek-V3.2）和 **2 个脚手架**（OpenClaw、Nanobot）在 4 个领域的源任务上执行无技能基线，收集总计 2,605 个任务的轨迹。
- 每个任务收集跨骨干的成功/失败执行，以降低单模型偏见。

**阶段 2：基于轨迹的能力提取与规范化**
- **提取器 Eϕ**（以 Claude Sonnet 4.6 实例化）输入任务查询、参考答案和跨骨干轨迹，生成局部原始能力卡片，每张卡片包含：
  - **触发条件 γ**：何时应用
  - **可复用程序 π**：具体操作步骤
  - **边界 ∂**：适用性限制
  - **角色 ρ**：Method、Guard 或 Workflow
- **规范化**：通过嵌入相似性进行召回式分块，然后使用三个独立 LLM（Kimi、GLM、DeepSeek）对候选对进行**操作等价性判断**，非一致性对由领域专家复核。经组级一致性检查后，合并为规范能力单元。共得到 **170 个规范能力单元**（从 7,326 张原始卡片合并而来）。

**阶段 3：能力图构建与分割**
- 为每个领域构建**无向能力图 G_d**：节点为任务，边连接共享至少一个边合格能力的任务。
- 使用 Louvain 社区检测划分任务社区，然后进行**约束性采样**：每个测试任务必须与同一社区中的至少一个训练任务共享边合格能力。
- 最终得到 **528 个训练任务 + 267 个测试任务**，全部测试任务均有训练侧能力支持（零不支持的测试任务）。

### 2.3 评估协议
- **定义**：设方法 m 的迁移增益 Δ_m = 平均(rm(x) - r0(x))，其中 r0 是无进化基线，rm 是应用方法后的得分。
- **控制条件**：任务语句、工具集、评分合同、超时、基础代理配置在每种脚手架-骨干设置中完全固定，方法仅因进化状态不同而区分。

## 3. 实验设计

### 3.1 数据集/场景
- **四个长期智能体领域**：

  - **Algorithmic Reasoning**：源自 LiveCodeBench V6（1,055 个问题）

### 3.2 对比方法
- **Vanilla**：无进化状态的基线
- **自动方法**：
  - **Memento**：检索最相似保留训练案例（Zhou et al., 2025）
  - **ReasoningBank**：从蒸馏推理记忆池中检索（Ouyang et al., 2026b）
  - **GEPA**：在训练集上进化单个提示并广播所有测试任务，无检索（Agrawal et al., 2026）

### 3.3 骨干模型与脚手架

## 6. 论文主要结论与发现

1. **能力内容跨模型族可迁移**：Anchor Skill（使用不同构造骨干）在所有 6 个设置和所有领域均产生正迁移增益（平均 Δ 分别为 +7.5%、+10.5%、+5.8%），证明训练侧的能力信息足以带来普遍收益。
2. **当前自动方法普遍脆弱**：
   - Memento：在 Qwen3.5-27B 和 Gemma-4-31B 上平均 Δ 为负（-2.4、-0.7），且在 Nanobot/Qwen3.5-27B/SWE 领域暴跌 -36.3 个百分点。
   - ReasoningBank：平均 Δ 在 +0.4% 到 +3.6% 之间，但有 6 个单领域负值单元格。
   - GEPA：平均 Δ 在 +1.2% 到 +5.7% 之间，但有 4 个负单元格。
3. **瓶颈在于方法侧提取与路由**：Anchor 与自动方法的唯一差异在于自动化部分（提取、索引、检索），而任务、工具、超时完全相同，因此性能差距可归因于方法侧机制。
4. **规模不预测迁移性能**：较大的 Qwen3.5-397B 未在所有方法上持续优于小模型；Gemma-4-31B 上 GEPA 表现突出，但 Memento 却很弱。
5. **进化不统一降低成本**：Anchor 在 4/6 的设置中降低了 agent turns，而 GEPA 在 4/6 的设置中增加了 turns（即使解决了任务也会增加开销）。成本与准确率脱钩，需联合报告。

## 7. 优点

- **问题定义清晰**：首次将智能体自我进化定义为“轨迹到程序的过程性知识迁移”，并设计出可控的评估框架。
- **构建方法创新**：通过“能力图”保证测试任务有训练侧能力支持，同时确保实例不重合，严格隔离了经验编码与路由环节。
- **诊断能力**：通过 Anchor Skill 作为诊断参考，能够定位方法是因“无法提取”还是“错误路由”而失败，为方法论改进提供了方向。
- **多维度评估**：同时报告准确率、成本（turns 变化）和成本-准确率脱钩现象，避免单一指标误导。
- **跨领域覆盖**：四个长期程领域包含不同类型的过程性知识（信息搜索策略、算法模式、代码修复流程、知识工件验证），提升了基准的通用性。
- **可扩展性**：能力图构建流程是领域和骨架无关的，理论上可扩展至多模态或具身智能体。

## 8. 不足与局限

- **实验规模有限**：每个领域测试任务仅 56-86 个，统计显著性可能不足；论文自身在局限性中承认“细粒度分析需更大规模”。
- **骨干覆盖不全**：仅评估三个开源模型（Qwen3.5-27B/397B、Gemma-4-31B），未包含 GPT-4、Claude 3.5、Gemini 2.0 等闭源重量级模型。构造骨干与评估骨干不重叠的设计虽有利于测试内容迁移，但也限制了方法对比的完整性。
- **能力提取依赖单一提取器**：使用 Claude Sonnet 4.6 作为 Eϕ，虽然后续有三裁判规范化，但能力词汇仍受单一模型能力限制；不同提取器的影响未消融。
- **未包含多模态与具身场景**：目前仅关注文本智能体，在图像、视频、机器人等模态下的迁移能力未知。
- **成本分析粗糙**：仅报告 agent turns 变化，未报告 token 数、API 调用实际费用或延迟，且分母为全部任务而非仅成功任务时结果解释力有限。
- **潜在数据污染**：评估骨干可能在预训练中见过源基准数据，但论文认为所有条件共享同一骨干预受污染，因此比较性结论仍有效（绝对分数会受影响）。
- **人工评审成本高**：能力规范化中非一致性对需领域专家复核，限制了大规模自动生成。
