## 摘要
运行在用户本地机器上的个人AI代理（如OpenClaw）可自动化日常任务，包括网页搜索、电子邮件和文件管理。它们对计算机资源（包括网络、文件系统和shell）的访问使其面临间接提示注入（IPI）攻击。先前的双LLM防御通过用代理可引用但不可读的符号替换不可信数据来阻止IPI。然而，它们仅在代理的上下文中追踪不可信数据，因此当代理保存并随后重新读取不可信数据时，该数据（可能是攻击者的提示）可能作为可信数据而非符号返回，我们称之为存储型IPI。在用户真实环境（人类和程序共享）中运行正是使得像OpenClaw这样的代理变得实用的原因，也正是使得忽视这一点的防御不完整的原因。在这样的环境中保留符号很困难，因为人类和程序需要原始数据。我们提出DualView，它通过赋予每个通道两个视图，将不可信数据追踪从代理的上下文扩展到用户的环境，包括文件系统、shell、网络及其他代理。在AgentView中，即使代理将不可信数据写出并重新读取，代理仍将其视为符号，从而阻止存储型IPI，而HumanView则为人类和工具保留原始数据。DualView将每个工具调用路由到正确的视图，并在两个视图之间同步数据。DualView作为OpenClaw插件部署，仅使用工具钩子，无需更改代理的工具调用逻辑或工具实现。由于DualView通过设计隔离了不可信数据，其保护不限于已知的攻击模板。在IPI基准测试和PinchBench上的评估中，DualView阻止了所有IPI攻击（包括存储型IPI），同时保持了接近未受保护基线的实用性。

## Abstract
Personal AI agents that run on the user's local machine, such as OpenClaw, automate daily tasks including web search, email, and file management. Their access to computer resources, including the network, file system, and shell, exposes them to indirect prompt injection (IPI) attacks. Prior Dual LLM defenses block IPI by replacing untrusted data with symbols that the agent can reference but not read. However, they track untrusted data only inside the agent's context, so when the agent saves and later rereads untrusted data, that data, possibly an attacker's prompt, can return as trusted data rather than as a symbol, which we call stored IPI.   Operating on the user's real environment, which humans and programs share, is what makes agents like OpenClaw practical, and is exactly why a defense that ignores it is incomplete. Preserving symbols in such an environment is hard, because humans and programs need original data. We present DualView, which extends untrusted data tracking from the agent's context to the user's environment, including the file system, shell, network, and other agents, by giving each channel two views. In AgentView, the agent sees untrusted data as symbols even after writing it out and reading it back, blocking stored IPI, while HumanView preserves original data for humans and tools. DualView routes each tool call to the right view and synchronizes data across the two views. DualView deploys as an OpenClaw plugin using only tool hooks, without changing the agent's tool-call logic or tool implementations. Since DualView isolates untrusted data by design, its protection is not limited to known attack templates. In our evaluation on an IPI benchmark and PinchBench, DualView blocked every IPI attack, including stored IPI, while keeping utility close to the unprotected baseline.

- **核心问题**：个人AI代理（如OpenClaw）通过工具（文件系统、shell、网络）与用户环境深度交互，容易受到**间接提示注入（IPI）攻击**。现有双LLM防御仅在代理上下文内将不可信数据替换为符号，但一旦数据被保存到环境中（如文件），重新读取时可能作为可信文本返回，导致**存储型IPI（stored IPI）**。这种攻击无法被现有防御阻止。

### 2. 方法论：核心思想、关键技术细节

- **核心思想**：为每个通道（文件系统、shell、网络、代理间通信）提供两个视图：
  - **AgentView**：代理视角下，不可信数据始终以**符号**（opaque symbol）呈现，即使写入文件后再读取仍为符号，从而阻止存储型IPI。
  - **HumanView**：人类和程序视角下，数据保持**原始内容**，保证环境兼容性。
- **关键技术细节**：
  - **视图路由与同步**：DualView将工具调用自动路由到合适视图。例如，文件工具、本地shell（AgentShell）在AgentView运行；网络工具、需要原始数据的shell（HumanShell）在HumanView运行。在AgentView工具调用前后，通过**Git工作树**同步两个文件系统，确保人类修改可见且代理写入经去符号化后还原为原始数据。
  - **符号化与去符号化**：通过SQLite符号表映射符号与原始数据。当HumanView工具返回结果时，依据**数据信任策略**（基于工具模式和来源规则）将不可信字段符号化；当需要原始数据时（如向网络发送请求），先对符号去符号化再执行调用。
  - **策略机制**：
    - **数据信任策略**：定义哪些字段/来源是可信的（如工具元数据、本地文件）或不可信的（如网页正文、邮件内容）。信任策略可配置。
    - **数据使用策略**：在HumanShell中，若检测到符号被用作命令或代码（如`python -c <symbol>`），则要求用户审批，防止执行不可信数据。
  - **实现**：作为OpenClaw插件，利用事件钩子（pre-tool、post-tool、message delivering）实现，无需修改AI模型或工具内部逻辑。

### 6. 论文的主要结论与发现

- DualView将**立即IPI和存储IPI的攻击成功率均降至0%**，而基线分别为57.0%（Haiku）和27.8%（Sonnet）。
- **效用接近基线**：Haiku上81.6%（vs 83.4%），Sonnet上82.1%（vs 88.5%），是唯一同时实现0% ASR且保持高误用的防御（沙箱效用降至约64-66%）。
- **人类效用完整**：人类环境（文件、消息、输出）中无符号残留，用户可正常使用。
- **成本**：Token开销48.2%（Haiku）和93.4%（Sonnet），主要来自U-LLM的额外调用，特别是Sonnet多次尝试获取符号背后的原始值。
- **审查负载低**：PinchBench中仅触发1次用户审批（良性误报），远低于输入/输出护栏的误报率。

### 7. 优点

- **确定性安全保证**：通过符号隔离而非概率性判断，可抵御未知攻击变体。
- **同时满足三重目标**：安全、代理效用、人类效用，解决了现有防御的取舍矛盾。
- **轻量部署**：作为插件使用工具钩子，无需修改模型或工具代码，可迁移到其他支持钩子的代理框架（如Claude Code）。
- **策略可配置**：数据信任策略支持工具模式和来源规则，数据使用策略可扩展，用户可主动更新。
- **创新双视图机制**：有效处理存储型IPI这一关键被忽视的攻击面。

### 8. 不足与局限

- **数据使用策略覆盖不完整**：仅检测显式命令执行，难以覆盖所有敏感参数（如网络请求目标）或间接执行（如通过脚本文件）；需要更严格的shell监控或系统调用钩子。
- **无法追踪直接从HumanView注入的不可信数据**（如用户粘贴的未知内容），依赖用户自行配置来源规则。
- **自定义工具路由需手动分配**，未来可考虑自动化推断。
- **远程网络无法建立AgentView**，若信任策略过于宽松，仍可能遭受远程存储IPI（如云文档服务）。
- **成本较高**：尤其是Sonnet上多次U-LLM调用导致93.4%的token开销，在成本敏感场景下可能受限（可通过缓存、小模型U-LLM等方式优化）。
- **未测试适应性攻击**：虽声称不限于已知模板，但未设计对抗性变体（如符号枚举、数据拆分绕过）进行验证。
- **PinchBench实用实验仅运行一次**，虽有147个任务，但缺乏多次重复的方差分析。
