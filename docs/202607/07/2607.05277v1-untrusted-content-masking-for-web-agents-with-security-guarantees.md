## 摘要
针对提示注入攻击提供安全保障的防御措施依赖于可信指令与不可信数据之间的严格隔离。在工具调用API等基于文本的环境中，这种分离自然形成：代理可以通过接口定义进行推理，而无需处理不可信内容。将这些保障扩展到Web代理面临一个根本性挑战：为了感知并与环境交互，Web代理必须首先观察渲染后的页面，而页面中可信内容与不可信内容相互混杂。这种结构上的缠结消除了安全保障所依赖的信任边界，削弱了Web代理的可证明防御。在本文中，我们提出了不可信内容屏蔽（UCM），一种简单有效的方法，可在Web环境中恢复这一边界。我们利用了一个关键的结构性洞察：网页的文档对象模型（DOM）编码了足够的信息，可以在不读取内容的情况下区分可信区域和不可信区域。我们的框架通过在不可信区域到达代理之前对其进行编辑，并通过具有严格权限分离的沙盒接口路由交互，从而利用这一特性，使代理能够观察并与环境交互，同时与对抗性内容保持隔离。代码已公开。

## Abstract
Defenses that provide security guarantees against prompt injection attacks rely on strict isolation between trusted instructions and untrusted data. In text-based environments such as tool-use APIs, this separation arises naturally: agents can reason from interface definitions without ever processing untrusted content. Extending these guarantees to web agents faces a fundamental challenge: to perceive and interact with their environment, web agents must first observe the rendered page, which intermingles trusted content with untrusted content. This structural entanglement removes the trust boundary on which security guarantees depend, undermining provable defenses for web agents. In this paper, we present Untrusted Content Masking (UCM), a simple and effective approach that restores this boundary in web environments. We leverage a key structural insight: a webpage's Document Object Model (DOM) encodes sufficient information to distinguish trusted from untrusted regions without reading their content. Our framework exploits this by redacting untrusted regions before they reach the agent and routing interaction through a sandboxed interface with strict privilege separation, thereby enabling agents to observe and interact with their environment while remaining isolated from adversarial content. The code is publicly available.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：Web代理（如Claude Computer Use、OpenAI Operator）在自主浏览网页时会遭受**提示注入攻击**。网页中**可信内容**（导航、官方列表）与**不可信内容**（用户评论、广告、第三方数据）混杂在一起，代理必须观察整个渲染页面才能交互，这破坏了安全防御所依赖的**信任边界**。
- **现有防御的不足**：启发式防御缺乏形式化保证且可被自适应攻击绕过；提供了保证的防御（如单次规划、计划再执行）要么迫使代理对页面完全“失明”，要么需要为每个网站构建和维护类型化的API，限制了实用性和灵活性。
- **本文目标**：提出一种**同时提供安全保证、保持任务实用性、且只需最小页面改动**的方法——不可信内容屏蔽（UCM）。

### 2. 方法论：核心思想、关键技术细节

- **核心思想**：利用网页DOM结构**天然编码了信任边界信息**（例如，评论组件、小部件等具有专属的CSS类、ID或层级）。在代理观察页面之前，将这些不可信区域替换为**带标签的占位符**，使代理永远不接触对抗性文本。
- **关键技术细节**：
    1. **Unstrusted Content Masking（屏蔽）**：页面所有者或自动化工具在DOM中标出不可信元素（例如添加`data-untrusted="true"`属性），然后在渲染时用占位符（如`id: review:2`）替换其内容，保留结构信息和语义标签。
    2. **Quarantined Model（检疫模型，Q-Model）**：对于需要读取不可信内容的任务，代理不能直接看，而是通过一个隔离模型查询。代理提供元素ID、自然语言问题和**返回类型**（布尔、整数、浮点数、枚举、日期）。Q-Model读取真实内容，但只返回类型约束的结构化响应（如`{"result": true, "type": "bool"}`）。由于不能返回自由文本，注入指令无法传播。
    3. **可选扩展**：如果类型约束不足以完成任务，代理可声明任务不可解，然后允许Q-Model返回字符串，但需经用户审批后才能进入代理上下文。
    4. **自动边界推断**（用于被动诚实所有者）：使用LLM分析**内容清洗后的DOM**（文本全部替换为占位符），仅凭结构（标签、类、层次）输出CSS选择器来识别不可信区域。通过迭代提示提高覆盖率。

- **安全属性**：控制流被保护——阻断注入指令通过Q-Model返回；数据流攻击（操纵聚合或单个值）仍然可能，但受限于独立处理每个元素（MapReduce模式），且可通过并行查询、用户确认等缓解。

### 7. 优点

- **简单实用**：核心想法直观（屏蔽不可信内容），仅需网站添加少量标签（或自动推断），不需大规模重构。
- **可证明安全**：提供控制流隔离保证，不依赖启发式检测或模型鲁棒性。
- **保持ReAct循环**：代理仍能观察页面和交互（只是看不到被屏蔽的部分），比“计划再执行”范式更灵活。
- **兼容性**：可与任何下游安全策略（用户确认、限制敏感操作）组合使用。
- **自动边界识别**：允许被动所有者使用，且过程本身不受注入影响（使用内容清洗DOM）。
- **多样化评估**：涵盖多个模型、自定义+真实环境、攻击场景，结果可信。

### 8. 不足与局限

- **信任假设**：假设网站所有者诚实——对于恶意网站本身，UCM无法防御（因为页面加载即可触发有害行为）。实际部署需配合允许列表。
- **不涵盖主动内容漏洞**：不防御XSS或DOM重写（诚实网站应防止此类漏洞）。
- **数据流攻击局限**：类型约束无法防止欺骗性Q-Model返回错误布尔值（如选择劫持）。作者提出了缓解思路（并行查询、冲突检测），但未实现。

- **成本仍可能较高**：对于需要大量Q-Model调用的任务（如逐条分析评论），成本开销可能显著增加。
- **实现细节**：依赖浏览器侧JavaScript框架进行屏蔽，可能影响页面性能或与现有扩展兼容。
- **用户审批开销**：字符串输出回退依赖用户审批，可能降低自主性并增加用户负担。
