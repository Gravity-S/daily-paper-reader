## 摘要
自进化框架通常优化任务解决方案，而将周围的框架视为固定不变。我们引入了感知框架的自进化（HASE），这是一个主动强化学习框架，其中单个模型可以在多轮动作空间中生成任务解决方案或编辑选定的框架组件。HASE使得单个Qwen3-8B模型能够匹配使用Claude Code作为框架提议器的GPT-OSS-120B模型的文本分类性能。在alpha因子挖掘中，HASE超越了已报告的GPT-OSS-120B基线。HASE还能修复不完善的评估组件，并在圆填充算法发现中收敛到最先进的性能。这些结果表明，HASE通过统一的主动过程同时改进了框架和解决方案。

## Abstract
Self-evolving frameworks usually optimize task solutions while treating the surrounding harness as fixed. We introduce Harness-Aware Self-Evolving (HASE), an agentic reinforcement-learning framework in which a single model can generate task solutions or edit selected harness components in a multi-turn action space. HASE enables a single Qwen3-8B model to match the text-classification performance of a GPT-OSS-120B model that uses Claude Code as the harness proposer. In alpha factor mining, HASE outperforms the reported GPT-OSS-120B baseline. HASE also repairs imperfect evaluation components and converges to state-of-the-art performance in circle-packing algorithm discovery. These results show that HASE improves the harness and the solution through one unified agentic process.

## 1. 核心问题与研究动机

现有自进化（self-evolving）系统（如 FunSearch、AlphaEvolve、Meta-Harness 等）通常只优化对象级任务解决方案（如代码、答案），而将周围的基础设施——“harness”（包括提示词、记忆、检索策略、评估器等）视为固定不变。这种假设有两个关键局限：
- 评估器可能不完整或与外部环境错位，导致模型利用评估漏洞（reward hacking）而非真正解决问题。
- 即使评估器正确，提供搜索先验的引导性 harness（如提示词、记忆格式）可能设计不佳，限制了搜索效率。

因此，论文提出 **HASE (Harness-Aware Self-Evolving)**，让同一个模型在 RL 框架中既能生成任务解，也能编辑 harness 组件，实现模型权重、harness 与任务方案的协同进化。

## 2. 方法论与核心技术细节

### 2.1 核心思想
HASE 是一种基于智能体强化学习（agentic RL）的统一框架。它定义了**统一动作空间**，模型在同一多轮交互中可以选择：
- 提交任务解（如分类标签、alpha 因子代码）；
- 编辑 harness 组件（通过沙箱中的文件操作 `ls, cat, grep, replace, overwrite`）。

### 2.2 Harness 的两类组件
- **Guidance 组件**（可自由优化）：提示词、记忆格式化、检索脚本、上下文选择等，提升搜索质量但不改变任务正确性。
- **Evaluation 组件**（受约束优化）：本地可编辑的评估器（proxy scorer），但修复需依赖**真实世界评估器（oracle）** 的反馈，防止无限制的 reward hacking。

### 2.3 算法流程（Algorithm 1）
- 多阶段（phase）训练，每阶段包含多个 GRPO 轮次。
- 轨迹奖励仅基于最终提交解的质量；harness 编辑本身不直接获得奖励，而是通过编辑后获得的更好解来间接得到信用（proof-of-concept 评分）。
- 阶段边界处进行 **harness 选择**：
  - Guidance 组件：筛选有效编辑，用验证集评分，提交最好的。
  - Evaluation 组件：检查代理分与 oracle 分是否分歧，将分歧案例加入 mismatch set；修复必须同时通过回归测试并消除 mismatch。

### 2.4 关键设计
- **分段式安全机制**：评估器编辑必须有 oracle 分歧证据，且修复后需保持接口完整、不引入退化。
- **统一模型**：无需外部强大编码器（如 Claude Code）专门搜索 harness；HASE 自身的 8B 模型同时担任解生成器和 harness 提议器。

## 6. 主要结论与发现

1. HASE 使单个 8B 模型在**文本分类**上达到 GPT-OSS-120B + Claude Code 系统的性能（86.98% vs 86.8%），且无需外部提议器。
2. 在**Alpha 因子挖掘**中，HASE 以 8B 模型超越了 GPT-OSS-120B 的基线（AER 14.35% vs 7.89%，IR 1.70 vs 0.80）。
3. 在**圆填充**中，HASE 从破损评估器出发，自动修复两次后达到 SOTA（2.635983）。
4. 在**Heilbronn 三角形**中，协同进化策略明显优于冻结策略（0.03608 vs 0.02188），证明模型权重进化有助于内化修复后的约束。
5. 评估器修复并非无限制重写，而是受 oracle 分歧信号驱动并经过回归测试，有效防止 reward hacking。

## 7. 方法的优点

- **统一性**：同一模型同时进化 solution 和 harness，无需分离的外部代码搜索器，降低系统复杂度和成本。
- **安全性**：通过“阶段边界审查 + mismatch set + 回归测试”严格约束评估器编辑，避免 reward hacking。
- **灵活性**：区分两类 harness，对不同组件采用不同的进化规则（guidance 可自由优化，evaluation 需 oracle 锚定）。
- **可扩展性**：框架设计支持在不同任务（分类、优化、几何）中应用，且已被实验验证有效。
- **小模型大性能**：Qwen3-8B 在多个任务上匹配或超越数百亿参数模型系统，展示了 harness 进化对性能的巨大撬动效应。

## 8. 不足与局限

- **非全面扩展研究**：虽然覆盖了四种任务，但未在更广泛领域（如代码生成、机器人控制、多步推理）上验证，可推广性需更多证据。
- **保守的工程选择**：可编辑的 harness 组件被白名单化；评估器编辑严格依赖于 oracle 分歧且需要回归测试，可能拒绝了有益但更激进的修改。
- **算力资源未完全透明**：尽管给出了 GPU 数量和批量大小，但未报告训练总时长或 GPU 小时数，复现成本不易预估。
- **实际环境假设**：Real-world evaluator（oracle）的反馈频率在实验中是基于阶段边界进行查询，而在真实部署中 oracle 可能成本很高或延迟很大，论文未讨论该权衡。
- **消融实验数量有限**：仅在 Heilbronn 三角形上进行了权重冻结消融，其他任务缺乏类似分析（如不同 harness 组件重要性的消融）。
