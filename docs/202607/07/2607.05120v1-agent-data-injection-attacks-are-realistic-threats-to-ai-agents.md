## 摘要
AI代理代表用户提示执行操作，消耗外部数据并根据代理上下文采取行动。先前关于AI代理安全的研究主要集中在间接提示注入（IPI）上，其研究最充分的类别是指令注入，即攻击者控制的不可信数据被解释为指令。为此，已提出许多缓解措施来防止指令注入攻击。本文引入了一种新的IPI类别——代理数据注入攻击（ADI）。ADI将恶意数据伪装成可信数据，例如安全关键元数据（如资源标识符或数据来源）或代理上下文数据（如工具调用和响应格式）。结果，代理在不知情的情况下根据攻击者控制的数据执行非预期操作。ADI与指令注入攻击具有相似的攻击影响，因为它导致代理行为异常并执行非预期操作。尽管影响相似，ADI仍未被充分探索，且容易绕过现有的IPI防御。我们在真实世界的代理中发现了若干关键漏洞，使攻击者能够发起各种攻击：对Web代理（Claude in Chrome、Antigravity和Nanobrowser）的任意点击攻击，以及对编码代理（Claude Code、Codex和Gemini CLI）的远程代码执行和供应链攻击。我们评估了现成模型和AI代理中的ADI漏洞，发现ADI在独立LLM和AI代理设置中均有效。ADI暴露了代理安全中的一个关键缺口，表明当前AI代理并未遵循基本的安全原则：当前代理未能将可信数据与不可信数据隔离。

## Abstract
AI agents act on behalf of user prompts, consuming external data and taking actions based on the agent context. Prior research on AI agent security has primarily focused on indirect prompt injection (IPI). Its most well-studied category is instruction injection, where attacker-controlled untrusted data is interpreted as an instruction. In response, many mitigations have been proposed to prevent instruction injection attacks. In this paper, we introduce a new category of IPI, agent data injection attacks (ADI). ADI injects malicious data disguised as trusted data, such as security-critical metadata (e.g., resource identifiers or data origins) or agent context data (e.g., tool call and response formats). As a result, agents unknowingly execute unintended actions based on attacker-controlled data. ADI has similar attack impacts as instruction injection attacks, because it causes agents to misbehave and execute unintended actions. Despite the similar impact, ADI remains underexplored and easily bypasses existing IPI defenses. We found several critical vulnerabilities in real-world agents that allow an attacker to launch various attacks: arbitrary click attacks on web agents (Claude in Chrome, Antigravity, and Nanobrowser), and remote code execution and supply-chain attacks on coding agents (Claude Code, Codex, and Gemini CLI). We evaluate ADI vulnerabilities across off-the-shelf models and AI agents, and find that ADI is effective in both standalone LLMs and AI agent settings. ADI exposes a critical gap in agent security, signifying that current AI agents do not employ a fundamental security principle: current agents do not isolate trusted data from untrusted data.

## 1. 核心问题与整体含义（研究动机和背景）
- **背景**：AI Agent 通过工具与外部环境交互，处理来自不同来源的数据，其中混合了可信数据（如元数据）与不可信数据（如用户生成内容）。先前对 Agent 安全的研究主要集中于间接提示注入（IPI），尤其是指令注入，即攻击者将恶意数据伪装成指令，使 Agent 执行攻击者任务。已有多种防御（模型强化、输入/输出护栏、双LLM等）被提出以防范指令注入。
- **核心问题**：本文发现 Agent 内部另一个未被充分探索的攻击面——可信数据与不可信数据之间的隔离缺失。攻击者可将恶意数据伪装成可信数据（如资源标识符、数据来源、工具调用历史），导致 Agent 在执行用户原始任务时使用被篡改的可信数据，从而执行非预期操作。这种新攻击被称为**Agent数据注入攻击（ADI）**，它与指令注入危害类似，但能绕过现有专注于指令与数据分离的防御机制。
- **整体含义**：当前 AI Agent 缺乏一个基本安全原则——可信数据与不可信数据的隔离，ADI 暴露了这一重大缺口，亟需重新设计 Agent 数据防护方案。

## 2. 方法论
- **核心思想**：ADI 通过**概率性定界符注入（probabilistic delimiter injection）** 实现。攻击者在不可信字段（如邮件正文、评论内容）中注入看似普通文本但被 LLM 概率性误解为结构定界符的字符序列（例如 `\"` 被当作 JSON 双引号、`<button>` 被当作 DOM 元素标记），从而让 LLM 将攻击者伪造的数据误认为是系统生成的可信元数据（如发件人、元素 ID、工具调用结果）。
- **关键技术细节**：
  - 与确定性定界符注入（如 SQL 注入的 `'`、XSS 的 `<script>`）不同，LLM 概率性解释文本，即使注入的定界符并非精确匹配（如转义字符 `\"`、花引号 `”` 等），LLM 仍可能将其视为有效的结构边界。这称为**概率性定界符注入**。
  - 攻击者利用这种误解来**伪造可信数据**：例如在邮件正文中注入 `\"sender\":\"alice@gmail.com\"` 等片段，使 LLM 认为存在一个来自 alice 的邮件；或将网页评论中的文本伪装成按钮元素 `button "Read More" [ref_3]`，使 Agent 点击攻击者指定的真实元素。
- **攻击分类**：
  - **元素 ID 注入**（web agent）：伪造 DOM 元素描述，复用真实元素的 ID，导致 Agent 误点击攻击者指定的 UI 元素。
  - **来源注入**（coding agent）：伪造评论作者和角色，使 Agent 执行恶意命令。
  - **工具调用块注入**（coding agent）：伪造工具调用和响应的块结构，使 Agent 认为已经执行过代码审查，从而合并包含恶意代码的 PR。
- **形式化**：
  - 设指令为 I，可信数据为 D_T，不可信数据为 D_U，攻击者注入的恶意数据为 D_A。指令注入使 LLM 将 D_A 误认为指令 I 的一部分；ADI 使 LLM 将 D_A 误认为可信数据 D_T 的一部分。本质区别在于攻击所改变的数据类别不同。

## 3. 实验设计
- **独立 LLM 评估**：
  - **数据集/场景**：7 类真实工具响应数据（日历、云盘、GitHub 评论、邮件、GitHub issue、论文评审、Web DOM），共 157 个测试用例（JSON 112 个 + Web DOM 45 个）。
  - **Benchmark**：自建概率性定界符注入基准，测试 LLM 在结构提取和聚合任务上的表现。
  - **对比方法**：测试 6 种 LLM API（GPT-5.2、GPT-5-mini、Claude Opus 4.5、Claude Sonnet 4.5、Gemini 3 Pro、Gemini 3 Flash）。评估基线无防御、随机化防御（字段名附加随机 nonce）和清理防御（剥离定界符字符）下的成功率（ASR）和良性效用。
- **Agent 级评估**：
  - **场景**：使用 AgentDojo 基准库（4 个任务套件：工作区、Slack、银行、旅行），共 96 个用户任务，并添加 108 个 ADI 攻击案例。对比已有 935 个指令注入攻击。

- **评估指标**：攻击成功率（ASR）和良性任务完成效用。

## 6. 主要结论与发现
- **独立 LLM**：所有模型对概率性定界符注入均高度脆弱，JSON 格式 ASR 为 31.3%-43.3%，Web DOM 格式 ASR 为 33.3%-100%。结构一致性显著提高成功率，不同定界符变体同样有效（35.8%-43.3%）。随机化能有效防御（ASR 降至 0-3%），但清理防御虽然降低 ASR 但严重损害效用（utility 下降 10-20 个百分点）。
- **Agent 级**：现有指令注入防御（即使最先进的）在指令注入上已接近完美（ASR 0-0.7%），但对 ADI 基本无效。基线 ASR 为 49.1%，输入/输出护栏、计划-执行、沙箱、双LLM 均无法有效阻止（ASR 22.2%-50.0%）。只有 CaMeL Strict（严格数据流跟踪）完全阻止 ADI（0% ASR），但效用骤降至 36.5%。随机化提供较好的效用-安全权衡（utility 83.3%，ASR 28.7%）。
- **本质原因**：当前 Agent 架构未隔离可信与不可信数据，且现有防御过分关注指令-数据边界，忽视了数据内部的可信边界。

## 7. 优点

- **真实的攻击演示**：在 6 种现实世界 Agent（Claude in Chrome、Nanobrowser、Antigravity、Claude Code、Codex、Gemini CLI）上实现任意点击、远程代码执行、供应链攻击，具有较强说服力。
