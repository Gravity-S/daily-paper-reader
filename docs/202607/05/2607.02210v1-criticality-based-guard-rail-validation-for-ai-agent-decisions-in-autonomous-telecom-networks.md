## 摘要
向完全自主的电信网络（自主网络级别4-5）演进需要AI/ML代理在无需人工干预的情况下实时做出网络决策。然而，目前缺乏标准化的运行时机制来在单个推理输出触发实时网络状态变更之前对其进行拦截和验证，从而带来错误自主决策的风险。本文提出了防护栏验证（GRV）框架，这是一种可标准化的运行时架构，用于在执行前拦截和验证AI驱动的决策。该框架在多个加权维度上评估决策，包括行动范围、行动类型、服务关键性、代理自主级别、可逆性和时间行为模式，以确定关键性级别。基于该级别，应用分级验证机制：执行并记录、边界检查、独立代理验证或多代理共识。该框架还提供跨代理冲突检测，具有基于关键性加权的优先级解析，以及用于监管合规（例如欧盟AI法案第14条）的运行时一致性日志记录。我们介绍了该架构、算法流程、O-RAN部署模型，并评估了针对电信领域已知AI/ML攻击的威胁覆盖范围。

## Abstract
The evolution toward fully autonomous telecommunications networks (Autonomous Network Levels 4-5) requires AI/ML agents to make real-time network decisions without human intervention. However, no standardized runtime mechanism exists to intercept and validate individual inference outputs before they trigger live network state changes, creating risks of erroneous autonomous decisions. This paper proposes the Guard Rail Validation (GRV) framework, a standardizable runtime architecture for intercepting and validating AI-driven decisions before execution. The framework evaluates decisions across multiple weighted dimensions -- including action scope, action type, service criticality, agent autonomy level, reversibility, and temporal behavioural patterns -- to determine a criticality level. Based on this level, graduated validation mechanisms are applied: execute-with-logging, bounds checking, independent agent validation, or multi-agent consensus. The framework additionally provides cross-agent conflict detection with criticality-weighted priority resolution and runtime conformance logging for regulatory compliance (e.g., EU AI Act Article 14). We present the architecture, algorithmic procedures, O-RAN deployment model, and evaluate threat coverage against known AI/ML attacks in telecommunications.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：电信网络正迈向完全自主化（TM Forum定义的自主网络级别4-5）。AI/ML代理被用于实时监测网络状态（如流量、干扰、性能指标）并生成控制决策，通过推理管道（模型推理输出→网络管理功能→O1配置变化、A1策略或E2控制消息）改变实时网络状态。
- **核心问题**：当前标准化框架（如3GPP TS 28.105、O-RAN WG11）缺失**运行时步骤**来在推理输出执行前进行拦截和验证。这导致五个关键缺口：
  1. 无预行动验证（模型一旦部署，其输出直接执行，无安全检查）；
  2. 无关键性意识（所有决策被同等对待——读取PM计数器 vs 关闭紧急服务小区不应同样处理）；
  3. 无分级响应（仅有二进制激活/停用整个推理函数，而非针对单个不安全决策）；
  4. 无冲突解决（多代理场景下可能产生矛盾决策，缺乏检测与解析机制）；
  5. 无监管合规（欧盟AI法案第14条要求高风险AI系统具备人为监督能力，现有电信标准未满足）。
- **研究动机**：提出一种标准化、运行时、风险自适应的验证层，在不显著增加延迟的前提下保障自治网络决策的安全性，并满足监管要求。

### 核心思想
- **Guard Rail Validation (GRV) 框架**：在AI推理输出被消费转换为网络行动之前，作为强制性的验证层。验证结果有三种：**EXECUTE**（执行）、**BLOCK**（阻止）、**ESCALATE**（上报）。
- **多维度关键性评估**：对每个推理输出在六个加权维度上打分，确定关键性等级（LOW / MEDIUM / HIGH / CRITICAL）。
- **分级验证机制**：根据关键性等级采用定性不同的验证方式，而非统一处理。
- **跨代理冲突检测**：通过关键性加权优先级规则解决冲突。
- **运行时合规日志**：记录完整决策链，满足EU AI Act等监管要求。

**（2）分级验证**（图3）：
- **LOW (0-4)**：直接执行并记录日志，无额外验证。
- **MEDIUM (5-8)**：**边界检查**（Bounds Check）：参数值是否在安全范围内？变化率是否限定？是否违反硬约束（如最小小区覆盖）？
- **HIGH (9-12)**：**独立验证**（Independent Validation）：调用一个VALIDATOR代理（不同模型架构/训练数据/规则引擎）返回AGREE/DISAGREE/UNCERTAIN。AGREE则执行，DISAGREE则阻止并通知，UNCERTAIN则应用后备策略（延迟、缩减范围或上报）。若超时视为UNCERTAIN。
- **CRITICAL (13+)**：**多代理共识**（Multi-Agent Consensus）：路由到N个验证者（默认N=3，M-of-N同意，默认2-of-3；EMERGENCY切片需全票同意）。若验证者不足M个响应则退化为HIGH级，若无任何响应则阻止并上报。

## 7. 优点

1. **系统性架构设计**：将验证层从模型生命周期管理（预部署）扩展到运行时（推理后、执行前），弥补现有标准（3GPP TS 28.105、O-RAN）的缺失。
2. **风险比例化**：不是对所有决策施加相同验证负担，而是通过加权多维度评估实现风险感知的比例响应，最小化对常规操作的影响。
3. **可扩展维度模型**：六维度是电信特定实例，框架本身领域无关，允许插入其他维度。
4. **时间模式检测**：不仅检查单次行动，还通过滑动窗口检测频率、振荡和突发模式，识别协作或序列攻击。

7. **O-RAN适配**：针对延迟约束提供轻量变体，并将威胁防御与A1接口结合，可集成到现有Open RAN架构中。

## 8. 不足与局限

1. **无实证评估**：框架仅停留在架构描述层面，没有原型实现或在真实/模拟网络中测试。性能（延迟分布、吞吐量、误报率/漏报率）未知。正如作者在Discussions中指出的：“No empirical evaluation”.
2. **阈值敏感性问题**：作者提到“small weight changes can shift decisions between adjacent levels”，意味着阈值调优对效果影响大，但未提供校准指导或敏感性分析。
3. **仅检测直接冲突**：当前冲突检测限于同一实体同一参数，对于拓扑相关的间接冲突（如干扰耦合）未考虑。
4. **验证者可用性依赖**：HIGH和CRITICAL级别需要部署多样化的Validator代理。在没有足够验证者的情况下会退化为较低验证级别，削弱了安全保障。
5. **近实时变体安全折衷**：Near-RT RIC变体仅做边界检查，时间检测异步，实际上不能阻止不安全动作，只能事后报警，安全性显著降低。

8. **极端场景的保守性**：例如EMERGENCY切片要求全票一致同意才能执行，在正常操作中可能导致能量节约等有效决策被阻塞（误报）。缺乏对误报成本的定量分析。
