## 摘要
LLM智能体正迅速部署到生产环境中，包括编码助手、客户支持聊天机器人和科学研究助手，但它们在企业部署中仍基本保持静态。LLM权重、系统提示、工具库和上下文框架在部署时被冻结，任何改进都需要人工策划的数据收集、离线微调、智能体范式修改和重新部署的循环。最近关于自进化智能体的工作，例如面向个人用户的OpenClaw，表明智能体能力的下一次飞跃将来自那些能够从自身经验中持续学习的智能体。在本文中，我们认为，对于企业级大规模智能体服务，这种自进化智能体部署的愿景并非受限于强化学习算法，而是受限于智能体在线强化学习系统。具体而言，当前的智能体强化学习系统及周边的可观测性软件栈在三个关键方面存在不足：（i）缺乏标准化的智能体轨迹数据协议，无法在异构智能体范式中以步骤粒度携带强化学习学习信号；（ii）缺乏企业级综合数据代理，无法将实际工作负载转化为受治理的学习基质；（iii）缺乏统一的智能体演化控制平面，无法根据轨迹统计自动决定何时更新策略权重或演化上下文框架。下一代智能体强化学习系统必须围绕这三个支柱进行协同设计，我们勾勒了具体的架构、案例研究和反论。我们通过AReaL2.0实例化了一个分支，将现有的强化学习基础设施重组为面向智能体的在线强化学习循环，以根据部署的工作负载更新策略权重。

## Abstract
LLM agents are rapidly being deployed in production, including coding assistants, customer-support chatbots, and scientific research assistants, yet they remain fundamentally static in enterprise deployment. The LLM weights, system prompts, tool repertoires, and in-context harnesses are frozen at deployment time, and any improvement requires a manual loop of human-curated data collection, offline fine-tuning, modification of the agentic paradigm, and re-deployment. Recent work on self-evolving agents, such as OpenClaw for individual users, indicates that the next leap in agent capability will come from agents that continually learn from their own experience. In this paper, we argue that this vision for self-evolving agent deployment is being held back for enterprise-level large-scale agentic service not by reinforcement learning (RL) algorithms but by agentic online RL systems. Specifically, current agentic RL systems and the surrounding observability software stack are inadequate along three essential aspects: (i) there is no standardized agent trajectory data protocol capable of carrying RL learning signals at step granularity across heterogeneous agent paradigms; (ii) there is no enterprise-grade comprehensive data proxy that converts real workloads into governed learning substrates; and (iii) there is no unified agent evolution control plane that automatically decides, based on trajectory statistics, when to update policy weights or evolve the in-context harness. The next generation of agentic RL systems must be co-designed around these three pillars, and we sketch concrete architectures, case studies, and counter-arguments. We instantiate one branch through AReaL2.0, reorganizing existing RL infrastructure into an agent-oriented online RL loop for policy weight updates from deployed workloads.

## 2. 方法论

### 核心思想
提出下一代智能体在线强化学习系统需围绕**三个协同设计支柱**，使部署的智能体经验能够被捕获、转化为可学习的、受治理的、可回放的材料，并自动触发适当的演化动作。

### 关键技术细节

#### 支柱一：标准化智能体轨迹数据协议（ATDP）
- **定义**：轨迹 τ = (e1, e2, …, eT)，每个事件 et = ⟨ot, ht, at, yt, rt, mt⟩，其中 ot 是可观测状态，ht 是隐状态，at 是动作，yt 是动作结果，rt 是奖励信号，mt 是元数据。
- **设计原则**：
  - 决策相关但有界披露（不暴露所有内部token）
  - 跨框架任务统一（从prompt-response对转向类型化、可审计的事件记录）
  - 可信用分配（存储决策上下文，如工具版本、参数、错误类）
  - 支持延迟绑定学习信号（奖励可后期更新）
  - 版本化可回放性（归因到精确的执行环境快照）
  - 治理可观测性（隐私、安全、保留期、训练许可字段从开始就编码）

#### 支柱二：企业级智能体数据代理
- **作用**：在异构生产环境中拦截模型调用、工具调用、检索、记忆操作、文件/浏览器动作、人类反馈等，转换为ATDP轨迹。
- **设计要点**：
  - 框架无关拦截（LangChain、CrewAI、OpenAI SDK等）
  - 无损ATDP发射（保留所有学习所需字段）
  - 可回放能力（区分确定性回放、近似回放、不可回放事件）
  - 跨租户隔离与聚合（支持策略化聚合、联邦学习）
  - 奖励收割（从用户回复、测试失败、人工修正等提取弱/延迟奖励）
  - 数据完整性优先（在进入学习队列前完成编辑、访问控制、保留策略）

#### 支柱三：统一智能体演化控制平面
- **形式化**：智能体 A_t = ⟨πθ_t, Hψ_t, M_t, T_t, G_t⟩。控制平面观察轨迹窗口 D_t，选择演化动作 u* = argmax J_A(u | A_t, D_t)，其中 u ∈ U = {更新策略LLM, 更新上下文框架(harness), 更新记忆, 更新工具库/工具模式, 回滚, 无操作}。
- **实现考虑**：
  - 多表面适应（不同失败模式对应不同干预面：事实缺失→记忆插入，路由失败→框架编辑，广泛失败→策略更新）
  - 基于轨迹统计自动触发（评估分数、用户修正率、工具失败聚类、漂移指示等）
  - 算法多元性（统一触发下可调用不同RL算法：PPO、DPO、过程奖励学习、蒸馏等）
  - 安全分阶段部署（影子评估、回放检查、回归测试、金丝雀发布、回滚语义）
  - 回放优先评估（演进前必须在回放轨迹上测试）
  - 版本化溯源与回滚（所有演化工件必须版本化）

#### AReaL2.0 原型
- **目标**：聚焦于在线策略LLM权重更新这一个演化路径，展示现有RL基础设施可重组为面向智能体服务的在线学习循环。
- **核心设计**：将原始AReaL框架的rollout和训练worker封装为微服务组件，可插入任何现有在线智能体部署中，替换其标准推理后端。
- **组件**：
  - **Gateway**：公共入口，将外部LLM请求归一化并转发到RL运行时，最小化集成成本。
  - **Router**：提供轻量级会话亲和性管理，保持多轮对话的session一致性。
  - **Data Proxy**：记录轨迹数据并传递给训练管道。
  - **Agent-Compute Worker**：封装推理引擎（如SGLang、vLLM）和训练worker（Megatron/FSDP）为微服务接口。

## 6. 主要结论与发现

- 自进化智能体的核心瓶颈在于**系统基座**而非算法或模型大小，需要建设三大支柱：标准化轨迹协议、企业级数据代理、统一演化控制平面。
- 当前大多数自进化智能体工作（如Reflexion、Memento-Skills、OpenClaw-RL）关注记忆、技能、提示优化等单一机制，但缺乏企业级系统支撑来跨机制决策和受治理执行。
- AReaL2.0证明了现有RL基础设施可以较轻量改造接入在线智能体服务，使策略权重更新可以基于生产流量进行，无需重写智能体应用。
